신입 시절, 분명 동료와 같은 코드를 받았는데 "내 컴퓨터에선 잘 도는 게 동료 컴퓨터에선 죽는" 일이 있었어요. 며칠을 헤맨 끝에 범인은 의외로 단순했습니다. 두 컴퓨터에 깔린 라이브러리 버전이 미세하게 달랐던 것. 이 한 파일이 그런 일을 90% 막아줍니다.

## 한 줄 답

<div class="callout-accent">
package-lock.json 은 프로젝트의 <span class="key">실제로 설치된 모든 의존성</span> (직접 + transitive) 을 <span class="key">정확한 버전</span> 으로 고정해 기록한 파일이에요. package.json 의 범위 지정자 (^1.2.3 같은) 가 남기는 재현성 구멍을 메워 주는 역할. 파이썬의 uv.lock · poetry.lock 과 정확히 같은 자리입니다.
</div>

## 왜 필요한가: semver 가 남긴 구멍

이전 글에서 봤듯이 package.json 의 버전은 <span class="key">단일 버전이 아니라 범위</span> 예요.

```json
"dependencies": {
  "react": "^18.3.1"
}
```

이건 "18.3.1 이상 19.0.0 미만이면 뭐든 OK" 라는 선언. 문제는 <span class="key">언제 <span class="term">npm install</span> 하느냐에 따라 실제로 깔리는 버전이 다를 수 있다</span> 는 점이에요.

<table class="compare-table">
  <thead>
    <tr><th>환경</th><th>시점</th><th>명령</th><th>실제로 깔린 버전</th></tr>
  </thead>
  <tbody>
    <tr><td>내 컴퓨터</td><td>2026-04-01</td><td>npm install</td><td>react@18.3.1</td></tr>
    <tr><td>동료 컴퓨터</td><td>2026-05-15</td><td>npm install</td><td><span class="key">react@18.4.0</span> (그사이 18.4 가 npm registry 에 출시됨)</td></tr>
  </tbody>
</table>

<span class="key">같은 package.json</span> 을 받아서 <span class="key">같은 명령</span> 을 친 건데, 내 컴퓨터와 동료 컴퓨터에 다른 버전이 깔림. ^18.3.1 범위엔 둘 다 들어가니까 npm 이 각자 시점의 최신을 고른 결과예요. MINOR 업데이트 (기능 추가) 라 보통은 호환되지만, <span class="key">버그나 회귀</span> 가 몰래 섞여 들어올 수 있어요. 고전적인 <span class="key">"내 컴퓨터에선 되는데 왜 동료 컴퓨터에선 안 되지?"</span> 의 원인.

<h4><span class="key">시나리오 ②</span> &nbsp; Transitive 의존성의 버전 drift</h4>

더 교묘한 경우예요. 내 직접 의존성은 같은 버전이 깔렸는데, <span class="key">그 의존성의 의존성</span> 이 달라지는 경우.

    <tr><td>some-lib 의 package.json</td><td>"helper": "^2.0.0" (이건 범위)</td></tr>
  </tbody>
</table>

<table class="compare-table">
  <thead>
    <tr><th>환경</th><th>시점</th><th>실제로 깔린 버전</th></tr>
  </thead>
  <tbody>
    <tr><td>내 컴퓨터</td><td>2026-04-01</td><td>some-lib@1.0.0 + helper@2.0.0</td></tr>
    <tr><td>동료 컴퓨터</td><td>2026-05-15</td><td>some-lib@1.0.0 + <span class="key">helper@2.1.0</span> (그사이 helper 가 MINOR 업)</td></tr>
  </tbody>
</table>

내가 some-lib 를 정확 버전으로 잠갔는데도, <span class="key">some-lib 의 내부 의존성인 helper</span> 가 두 컴퓨터에서 다른 버전으로 깔렸어요. 내 package.json 엔 helper 가 적혀 있지도 않은데 (transitive 라서) 프로젝트 동작이 달라질 수 있음. 추적하기 가장 어려운 버그 원인.

## 해결책: package-lock.json

<span class="term">npm install</span> 이 끝나면 npm 이 <span class="key">실제로 깔린 모든 버전</span> 을 package-lock.json 에 기록합니다. 직접 의존성 · transitive 의존성 할 것 없이 <span class="key">전부</span>, 각각의 정확한 버전 + 내려받은 URL + 무결성 해시까지.

```json
"node_modules/react": {
  "version": "18.3.1",
  "resolved": "https://registry.npmjs.org/react/-/react-18.3.1.tgz",
  "integrity": "sha512-..."
}
```

이 파일이 있으면 나중에 누군가 <span class="term">npm install</span> 을 돌렸을 때, <span class="key">lock 파일에 적힌 정확히 그 버전들</span> 을 받아옵니다. 시간이 지나 react@18.4.0 이 나와 있어도 lock 파일에 18.3.1 이 적혀 있으면 그걸 써요.

<div class="note-box">

자동 생성 · 자동 갱신:

- <span class="key">자동 생성</span>: npm 5.0 (2017) 이후로 <span class="term">npm install</span> 시 자동 생성
- <span class="key">자동 갱신</span>: 새 패키지 설치 · 업데이트 때마다 자동으로 업데이트됨
- 🚫 개발자가 손으로 편집할 일은 거의 없습니다 (하지 않는 게 좋음)

</div>

## 핵심 규칙 1: **반드시 Git 에 커밋** 하세요

lock 파일의 존재 이유 자체가 <span class="key">"다른 컴퓨터에서 똑같은 버전이 깔리게"</span> 하는 건데, 이 파일이 레포에 없으면 lock 이 없는 것과 같아요. 그래서 무조건 커밋합니다.

<table class="compare-table">
  <thead>
    <tr><th>파일</th><th>커밋?</th><th>이유</th></tr>
  </thead>
  <tbody>
    <tr><td>package.json</td><td>O</td><td>의존성 선언 (범위)</td></tr>
    <tr><td>package-lock.json</td><td><strong>O 반드시</strong></td><td><strong>정확한 버전 고정 — 이게 재현성의 핵심</strong></td></tr>
    <tr><td>node_modules/</td><td>X</td><td>크고, lock 파일로부터 재생성 가능</td></tr>
  </tbody>
</table>

<div class="callout-warm">
.gitignore 에 package-lock.json 을 넣으면 안 됩니다. (반면 node_modules/ 는 반드시 넣어야 함.) 이 둘을 혼동해서 lock 을 ignore 처리한 레포를 실제로 종종 보게 돼요. 보이는 즉시 고쳐야 하는 실수.
</div>

## 핵심 규칙 2: **registry 에는 publish 되지 않습니다**

<span class="term">npm publish</span> 로 라이브러리를 올릴 때 package-lock.json 은 <span class="key">tarball 에 들어가지 않아요</span>. npm 이 기본적으로 제외합니다. 즉:

<div class="callout">

- 내가 라이브러리 작성자라 <span class="term">npm publish</span> 를 해도 → 내 lock 은 registry 에 안 올라감
- 내가 <span class="term">npm install some-lib</span> 를 해도 → some-lib 의 lock 은 내려오지 않음

</div>

왜? lock 은 <span class="key">"이 프로젝트를 이렇게 빌드했다"</span> 는 기록이지, "남이 내 패키지를 이 버전으로 써야 한다" 는 선언이 아니거든요. 내가 some-lib 를 설치하면 some-lib 의 package.json 에 적힌 <span class="key">범위</span> 를 내 프로젝트 기준에서 해석해서 <span class="key">내 lock</span> 에 새로 기록됩니다.

<div class="callout-accent">

의미:

- lock 파일은 <span class="key">"최종 애플리케이션" 쪽에서만</span> 의미를 가짐: Claude Code 본체 같은 배포 대상 프로젝트
- 라이브러리 (react, lodash 같은) 의 lock 파일은 작성자 내부용일 뿐, 사용자에게 영향 주지 않음

</div>

## <span class="term">npm install vs npm ci:</span> 어느 쪽을 써야 하나

<span class="term">npm ci</span> 는 Continuous Integration install (또는 Clean Install) 의 약자로, npm 5.7 (2018) 에 추가된 비교적 새로운 명령이에요. <span class="key">CI/CD 파이프라인 · Docker 빌드 · 프로덕션 배포 같은 자동화 환경을 위해 만들어진 전용 설치 명령</span>. 로컬 개발에서는 잘 안 보지만 GitHub Actions YAML 이나 Dockerfile 을 열면 자주 나옵니다.

### 두 명령의 차이

<table class="compare-table">
  <thead>
    <tr><th>항목</th><th><span class="term">npm install</span></th><th><span class="term">npm ci</span></th></tr>
  </thead>
  <tbody>
    <tr><td><strong>용도</strong></td><td>로컬 개발</td><td>CI/CD · Docker · 배포</td></tr>
    <tr><td><strong>lock 이 없으면?</strong></td><td>새로 생성</td><td>에러로 중단</td></tr>
    <tr><td><strong>lock 과 package.json 이 불일치하면?</strong></td><td>lock 을 갱신해서 맞춤</td><td>에러로 중단 (어느 쪽도 안 바꿈)</td></tr>
    <tr><td><strong>기존 node_modules/ 는?</strong></td><td>변경분만 반영 (점진적)</td><td>통째로 삭제 후 재설치 (항상 clean)</td></tr>
    <tr><td><strong>lock 파일 수정?</strong></td><td>필요하면 함</td><td><strong>절대 안 함</strong> (read-only)</td></tr>
    <tr><td><strong>속도</strong></td><td>의존성 해석 필요, 변동적</td><td>빠름 (lock 에 트리가 이미 있음)</td></tr>
    <tr><td><strong>결과의 재현성</strong></td><td>시점에 따라 다를 수 있음</td><td><strong>100% 동일</strong></td></tr>
  </tbody>
</table>

### ⚠️ 꼭 기억할 것: <span class="term">npm install</span> 은 **조용히** lock 을 수정합니다

위 표의 "lock 파일 수정? → 필요하면 함" 이 대수롭지 않아 보이지만, <span class="key">실제로 가장 많은 혼란의 원인</span> 이에요. <span class="term">npm install</span> 은 <span class="key">아무 알림 없이</span> package-lock.json 을 바꿔 버립니다. 이걸 모르면 다음 같은 상황에서 이해가 안 돼요.

#### 상황 ①. "내가 lock 을 건드리지도 않았는데 git diff 에 잔뜩 떠 있네?"

```bash
$ npm install       # 그냥 프로젝트 받아서 한 번 돌렸을 뿐
$ git status
  modified:   package-lock.json     ← 뭐야?!
```

그사이 registry 에 최신 버전이 나와서 ^ 범위 해석 결과가 바뀌었거나, 예전 lock 의 특정 항목이 최신 npm 에서 살짝 다른 형태로 재기록되었거나. <span class="key">내 의도와 무관하게 lock 이 갱신</span> 돼요.

#### 상황 ②. "로컬에선 되는데 CI 에선 빌드가 깨져요"

<div class="callout">

- 로컬에서: <span class="term">npm install</span> → lock 이 갱신됨 → 하지만 <span class="key">커밋을 안 함</span> → 로컬에선 새 lock 으로 돌아감
- CI 에서: 커밋된 (갱신 전) lock 으로 <span class="term">npm ci</span> → 다른 버전이 깔려 에러

</div>

<span class="term">npm install</span> 뒤에 git status 를 확인하는 습관이 없으면 이런 차이를 놓쳐요.

#### 상황 ③. "패키지 하나 추가했을 뿐인데 lock 이 200줄 변경됨"

<span class="term">npm install new-pkg</span> 를 하면 new-pkg 뿐 아니라 <span class="key">기존 패키지의 transitive 의존성까지</span> 재해석되면서 lock 이 연쇄적으로 갱신될 수 있어요. 정상이지만 코드 리뷰에서 "이 변경이 의도한 건가?" 의 혼란을 일으킴.

<div class="callout-warm">

실전 대응법:

1. <span class="term">npm install</span> 뒤에는 <span class="key">항상 git status · git diff package-lock.json 확인</span>. 의도하지 않은 변경이 있으면 커밋 전에 이해하거나 되돌리기
2. 의도적 변경이면 <span class="key">반드시 커밋</span>. "내 머신에서만 살아 있는 lock 변경" 은 상황 ② 의 고전적 원인
3. <span class="key">CI 에선 무조건 <span class="term">npm ci</span></span>. CI 가 실수로 lock 을 바꾸거나, 커밋되지 않은 로컬 lock 변경을 덮어쓰지 못하게

</div>

한마디로. <span class="key"><span class="term">npm install</span> 은 lock 을 "관리" 하지만 "조용히" 관리합니다</span>. 침묵의 갱신이 git 레포 상태를 어떻게 바꾸는지 항상 확인하세요.

### 왜 CI 에서 <span class="term">npm install</span> 이 아니라 <span class="term">npm ci</span> 인가

자동화 환경에서 <span class="term">npm install</span> 이 일으킬 수 있는 문제들:

<div class="callout">

- <span class="key">빌드 시점에 lock 이 슬쩍 업데이트됨</span>: 누군가 package.json 만 바꾸고 lock 갱신을 깜빡했을 때 CI 가 "친절하게" lock 을 고쳐 버려요. 결과적으로 재현성이 깨짐
- <span class="key">기존 node_modules/ 와 섞임</span>: 캐시된 node_modules/ 에 남은 이전 상태가 간섭할 수 있음
- <span class="key">lock 없이도 진행 가능</span>: lock 이 실수로 빠져도 "대충" 설치되고 말아서 문제가 늦게 발견됨

</div>

<span class="term">npm ci</span> 는 이런 모호함을 전부 막아요:

<div class="callout-accent">

- lock 이 없거나 package.json 과 안 맞으면 즉시 에러. 문제를 빨리 드러냄
- node_modules/ 를 통째로 삭제 후 재설치: 이전 상태와 격리
- lock 을 절대 수정 안 함: 재현성 보장

</div>

### 실전 권장

<div class="callout">

- <span class="key">로컬 개발</span> → <span class="term">npm install</span>: 새 패키지 추가 · 업데이트 · lock 갱신이 자연스럽게 반영되어야 하니까
- <span class="key">CI/CD · Docker · 배포 서버</span> → <span class="term">npm ci</span>: 재현성 100%, 빠름, 실수로 lock 이 갱신되는 일 없음

</div>

GitHub Actions 에서 흔한 패턴:

```yaml
- name: Install dependencies
  run: npm ci               # npm install 이 아니라 ci
```

Dockerfile 에서도 마찬가지:

```dockerfile
COPY package.json package-lock.json ./
RUN npm ci                  # 이미지 빌드 재현성 확보
COPY . .
```

### 파이썬으로 치면

uv sync (유연) vs uv sync --frozen (엄격) 의 구분과 같은 자리예요. --frozen 은 lock 과 맞지 않으면 에러로 중단하고 lock 을 절대 수정하지 않음. <span class="term">npm ci</span> 와 동일한 철학.

  <tbody>
    <tr><td>범위 선언</td><td>pyproject.toml / requirements.txt</td><td>package.json</td></tr>
    <tr><td>정확 버전 고정 (lock)</td><td>uv.lock · poetry.lock · pdm.lock</td><td>package-lock.json</td></tr>

  </tbody>

파이썬 쪽에선 uv.lock · poetry.lock 이 <span class="key">커밋하는 게 기본</span> 이듯이, npm 도 package-lock.json 커밋이 기본이에요. 구조적으로 같은 해결책.

<div class="note-box">
참고. 순수 pip + requirements.txt 만 쓰면 lock 의 역할이 희미해지는데, 그래서 uv · poetry 같은 도구가 등장한 거예요. npm 은 처음부터 lock 을 내장하고 있어서 이 문제는 덜합니다.
</div>

## AI 엔지니어 입장에선 이 글이 왜 중요한가

여기까지 읽으면서 "내가 AI CLI 를 <span class="key">쓰기만</span> 하는데 이 lock 파일 이야기가 왜 필요하지?" 싶을 수 있어요. 케이스를 두 갈래로 나눠 드립니다.

### 케이스 A: Claude Code 같은 AI CLI 를 **사용** 만 하는 경우

<span class="term">npm install -g @anthropic-ai/claude-code</span> 또는 <span class="term">npx @anthropic-ai/claude-code</span> 로 쓰기만 한다면:

<div class="callout">

- package-lock.json 을 <span class="key">직접 만질 일 없음</span>
- <span class="term">npm install -g</span> · npx 경로에는 lock 개념이 안 걸림 (lock 은 "프로젝트" 단위 개념인데, 사용자는 프로젝트를 만들고 있지 않으니까)
- <span class="key">이 글은 배경 지식 정도로만</span> 읽고 넘어가도 OK

</div>

### 케이스 B: **본인이 npm 프로젝트를 직접 관리** 하기 시작한 순간부터

아래 상황 중 하나라도 해당되면 <span class="key">lock 파일은 반드시 숙지해야 할 요소</span> 예요.

<div class="callout-warm">

- Claude Code fork · plugin · MCP 서버를 <span class="key">빌드 · 개발</span> 하는 경우
- <span class="key">Node.js 기반 백엔드 서비스</span> 를 짜는 경우 (AI 에이전트가 호출하는 API, 데이터 수집 서버 등)
- <span class="key">팀 프로젝트 레포</span> 에 기여하면서 <span class="term">npm install</span> 을 돌리는 경우
- <span class="key">Docker 이미지</span> 에 Node 기반 코드를 넣어 배포하는 경우

</div>

요즘 AI 엔지니어는 <span class="key">파이썬 백엔드 + Node 기반 프론트엔드/CLI</span> 를 섞어 쓰는 경우가 많아요. 그 순간부터 lock 파일은 여러분의 <span class="key">재현성 책임 영역</span> 에 들어옵니다.

### 최소한 이 세 가지만 기억하세요

어떤 npm 프로젝트를 만지든, 모르고 넘어가면 팀 전체에 피해를 줄 수 있는 핵심 세 가지.

<div class="callout-accent">

1. <span class="key">package-lock.json 은 무조건 Git 에 커밋</span> 한다. .gitignore 에 넣지 않는다
2. <span class="key">손으로 편집하지 않는다</span>. npm 이 관리하도록 둔다
3. <span class="key"><span class="term">npm install</span> 뒤에는 git status 로 lock 변경을 확인</span> 한다. 의도하지 않은 변경이면 커밋하지 말거나, 이해한 뒤 커밋

</div>

이 세 가지만 지켜도 lock 관련 문제의 <span class="key">90%</span> 를 피합니다. CI 에 <span class="term">npm ci</span> 를 쓰면 나머지 10% 까지 커버돼요.

</div>
