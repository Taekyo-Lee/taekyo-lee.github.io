저는 원래 RF 하드웨어 엔지니어였어요. 안테나·신호 쪽 일을 하다가 AI 가 재미있어 보여서 공부를 시작했고, 그때 처음으로 파이썬을 손에 잡았습니다. PyTorch, transformers, scikit-learn. <span class="term">pip</span> install 한 줄이면 끝나는 세계가 어느 순간 익숙해졌고, 한동안은 그 안에서 별 불편 없이 살았습니다. 그런데 ChatGPT 이후로 터미널에서 <span class="term">npx</span> some-tool, <span class="term">npm install -g some-ai-cli</span> 같은 명령을 칠 일이 부쩍 늘었어요. Claude Code 도 Codex 도 Cursor CLI 도, <span class="term">MCP</span> 서버들도 다 <span class="term">npm</span> 으로 배포되더라고요.

문제는 그 옆에서 자꾸 모르는 단어가 튀어나온다는 거였습니다. <em>Node, <span class="term">Node.js</span>, <span class="term">JavaScript</span>, <span class="term">TypeScript</span>, <span class="term">ECMAScript</span>, <span class="term">V8</span>…</em> 어렴풋이 들어보긴 했는데 서로 무슨 관계인지 자신 있게 설명할 수가 없었어요. 심지어 누가 "JavaScript 는 Java 의 스크립트 버전이에요?" 라고 물으면 잠깐 멈칫할 정도였죠.

이 글은 그때의 저 같은 사람을 위한 정리입니다. <span class="key">AI 모델 자체 (학습·파인튜닝·추론) 는 여전히 파이썬 중심입니다.</span> 다만 <span class="key">AI 를 "도구로 쓰는" 레이어 (CLI·에이전트 프런트엔드·MCP 서버) 는 Node 위에 모이고 있어요.</span> 그 세계가 뭐로 구성돼 있는지를 한 자리에 두고, 가능한 한 **파이썬 용어로 옮겨** 보겠습니다.

## <span class="term">JavaScript</span> 는 원래 어디서 돌던 언어인가

<span class="key">오직 웹 브라우저 안</span> 이었어요. 브라우저가 HTML 을 화면에 그리는 중간에 &lt;script&gt; 태그에 든 코드를 실행하는 용도였죠.

각 브라우저마다 <span class="term">JavaScript</span> 를 실행하는 내부 엔진이 따로 있습니다.

<table class="compare-table">

  </tbody>
</table>

여기서 "엔진" 은 <span class="term">JavaScript</span> 소스를 읽어 실제로 실행하는 프로그램이에요. 파이썬으로 치면 이렇게 보면 됩니다.

<div class="callout-accent">
<strong><span class="term">CPython</span></strong> = <span class="term">Python</span> 언어의 <strong>대표 구현체(인터프리터)</strong><br/>
<strong><span class="term">V8</span></strong> = <span class="term">JavaScript</span> 언어의 <strong>구현체 중 하나</strong> (Chrome 의 것)
</div>

2009년 전까지 <span class="term">JavaScript</span> 는 브라우저에 갇혀 있었어요. 서버, CLI 도구, 파일 처리 같은 "시스템 프로그래밍" 영역은 <span class="term">JavaScript</span> 의 세계가 아니었죠. 그 벽을 뒤집은 게 <span class="term">Node.js</span> 입니다.

## <span class="term">Node.js</span> 는 무엇인가

<span class="term">Node.js</span> 는 Chrome 의 <span class="term">V8</span> 엔진을 <span class="key">브라우저에서 떼어 와 독립 실행 런타임으로 만든 프로젝트</span> 예요. 거기에 "브라우저 밖 세상을 다루는 API" (파일 I/O, 네트워킹, 프로세스 관리 같은 것들) 를 얹었습니다.

### 왜 만들어졌나: 2009년의 서버 풍경

옛날 이야기 하나 해볼게요. 2009년 당시 인기 있던 웹 서버 방식은 이랬습니다.

<div class="note-box">
잠깐, 웹 서버 는 브라우저가 아닙니다. 정반대 자리예요. 사용자가 브라우저에서 어떤 사이트 주소를 치면, 그 요청을 <span class="key">받아서 응답을 보내주는 프로그램</span> 이 웹 서버입니다. 어딘가의 컴퓨터에서 항상 돌고 있어요. 비유하자면 <span class="key">사용자가 손님</span>, 브라우저는 손님이 주문을 전달하는 통로, <span class="key">웹 서버가 그 주문을 받아 음식을 만드는 주방</span> 입니다.
</div>

<div class="callout">
<strong>"손님 한 명 올 때마다 직원 한 명을 붙여준다."</strong><br/>
접속자가 오면 그 사람 전용 일꾼을 하나 만드는 방식. 문제는 접속자가 늘면 직원 수가 폭발한다는 점이었어요 (1만 명 접속 = 직원 1만 명). 서버 메모리가 감당 못 해서 뻗거나 느려집니다.
</div>

더 웃긴 건, 이 직원들이 <span class="key">거의 모든 시간을 멀뚱히 서 있다</span> 는 거였어요. 웹 서버가 하는 일은 대부분 **기다리는 것** 이거든요. *"DB 가 답해줄 때까지", "파일 읽기가 끝날 때까지", "외부 API 응답이 올 때까지"*. 1만 명의 직원이 각자 자기 손님 음식 기다리며 그냥 서 있는 풍경이었던 거죠.

<div class="callout-warm">
<strong>해법은 이미 알려져 있었습니다.</strong> <span class="key">직원 한두 명이 여러 테이블을 돌면서</span> 관리하는 방식이죠. 테이블 A 주문 받아 주방에 넘기고 → 바로 B 로 가서 주문 받고 → A 음식 나오면 다시 A 로 가서 서빙. 직원은 계속 움직이지만 숫자는 적어요. 프로그래밍 용어로는 event-driven async I/O 라고 부르는데, 이름은 거창해도 결국 이 "직원이 여러 테이블 돌리는" 방식입니다.
</div>

그런데 왜 하필 <span class="term">JavaScript</span> 였을까요? 다른 언어들도 가능은 했지만 각자 벽이 있었어요.

<table class="compare-table">
  <thead>
    <tr><th>언어</th><th>벽</th></tr>
  </thead>
  <tbody>
    <tr><td><strong><span class="term">Python</span></strong></td><td>2009년 당시엔 표준 async 도구가 없어서 "스레드 한 명당 손님 하나" 옛 방식밖에 없었음. 그마저 GIL 때문에 진짜 동시 처리도 안 됐음.</td></tr>
    <tr><td><strong>Java</strong></td><td>event-driven 가능은 했지만 async 코드가 너무 장황함 (콜백 지옥).</td></tr>
    <tr><td><strong>C / C++</strong></td><td>속도는 최고지만 표준 생태계가 없어서 event loop 부터 직접 만들어야 함.</td></tr>
    <tr><td><strong><span class="term">JavaScript</span></strong></td><td>태생부터 event-driven async. 브라우저에서 화면이 멈추면 안 되니 처음부터 "직원 한 명이 여러 테이블을 돈다" 방식이 기본 규칙.</td></tr>
  </tbody>
</table>

브라우저에서는 사용자가 버튼을 누르든 이미지가 로딩되든 <span class="key">화면이 절대 멈추면 안 되잖아요.</span> 그래서 <span class="term">JavaScript</span> 는 처음부터 *"전부 event-driven, 전부 async, 한 명이 여러 일을 번갈아 처리"* 가 기본 규칙이었거든요. 이미 "직원 여러 테이블 돌리기" 방식이 몸에 배어 있었던 셈입니다.

Ryan Dahl (<span class="term">Node.js</span> 창시자) 이 여기서 착안했어요: <em>"이 <span class="term">JavaScript</span> 를 브라우저에서 꺼내 서버 쪽으로 가져가면 이 문제가 자연스럽게 풀리지 않을까?"</em> 그렇게 <span class="key">2009년에 Node.js 가 태어납니다.</span>

### 엔진과 런타임은 같은 말인가

<span class="key">아니요, 다른 개념이에요.</span>

<div class="callout-accent">
<strong>엔진</strong> = 언어 자체를 돌리는 실행 코어. 파싱, JIT 컴파일, GC, 기본 자료형(숫자·문자열·배열·객체·함수) 정도까지만 담당. <strong><span class="term">V8</span> 만으로는 파일 하나도 못 엽니다.</strong><br/><br/>
<strong>런타임</strong> = 엔진 + 그 위에 얹힌 API 세트. "이 환경에서 뭘 할 수 있는지" 를 결정.
</div>

구체적인 예로 <strong>파일 읽기</strong> 를 들어볼게요.

<table class="compare-table">
  <thead>
    <tr><th>런타임</th><th>파일 읽기 방법</th></tr>
  </thead>
  <tbody>
    <tr><td><strong><span class="term">Node.js</span></strong></td><td>require('fs').readFileSync('a.txt') — Node 가 fs API 를 제공해줍니다.</td></tr>
    <tr><td><strong>브라우저 (Chrome)</strong></td><td>위 코드는 작동 안 해요. 브라우저엔 fs 가 없거든요. 대신 fetch('/a.txt') 같은 네트워크 API 만 가능합니다.</td></tr>
  </tbody>
</table>

<span class="key">엔진(V8) 은 같은데 런타임이 다르면 할 수 있는 일이 달라진다.</span> 이게 핵심이에요.

<div class="note-box">
업계에서는 "Node 런타임", "Node 환경", "Node 실행 환경(execution environment)" 모두 같은 뜻으로 써요. 이 글에서는 <strong>"런타임"</strong> 으로 통일합니다.
</div>

### 파이썬과의 대응

파이썬에서는 이 구분이 상대적으로 흐릿해요. <span class="term">CPython</span> 이 <strong>엔진(인터프리터/VM)</strong> 역할과 <strong>런타임(표준 라이브러리)</strong> 역할을 한 덩어리로 묶어서 배포하기 때문입니다.

<table class="compare-table">
  <thead>
    <tr><th>역할</th><th><span class="term">Python</span></th><th><span class="term">JavaScript</span></th></tr>
  </thead>
  <tbody>
    <tr>
      <td>엔진 (언어 실행 코어)</td>
      <td><span class="term">CPython</span> 의 bytecode 인터프리터</td>
      <td><span class="term">V8</span> (Chrome·Node), <span class="term">SpiderMonkey</span> (Firefox), <span class="term">JavaScriptCore</span> (Safari)</td>
    </tr>
    <tr>
      <td>런타임 = 엔진 + 표준 API</td>
      <td><span class="term">CPython</span> + 표준 라이브러리 (os, socket, json, …)</td>
      <td><span class="term">Node.js</span> = <span class="term">V8</span> + fs · http · process · … (브라우저, Deno, Bun 도 각각의 런타임)</td>
    </tr>
    <tr>
      <td>실행 명령</td>
      <td>python <span class="term">app.py</span></td>
      <td><span class="term">node</span> <span class="term">app.js</span></td>
    </tr>
  </tbody>
</table>

이게 등장하면서 <span class="term">JavaScript</span> 는 <span class="key">브라우저용 장난감에서 범용 프로그래밍 언어로 승격</span> 됐어요. 오늘날 <span class="term">Node.js</span> 위에서 돌고 있는 것들은 모두 브라우저 안이 아니라 <strong>서버 머신이나 내 컴퓨터 위에서 도는 프로그램</strong> 입니다.

<table class="compare-table">
  <thead>
    <tr><th>분류</th><th>대표 예</th><th>비고</th></tr>
  </thead>
  <tbody>
    <tr>
      <td><strong>웹 서버</strong></td>
      <td>Express, <span class="term">Next.js</span>, NestJS</td>
      <td>파이썬의 Django/Flask/FastAPI, ML 데모용 Streamlit/Gradio 와 같은 자리. <strong>언어만 <span class="term">JavaScript</span> 로 바뀐 것</strong>.</td>
    </tr>
    <tr>
      <td><strong>Electron 앱</strong></td>
      <td>Slack, Discord, <strong>VS Code</strong>, Figma, Notion</td>
      <td>"앱 안에 작은 브라우저(<span class="term">Chromium</span>) 를 통째로 넣는" 방식. UI 는 미니 Chrome 이, 시스템 일은 <span class="term">Node.js</span> 가. 각 앱이 자기 전용 Chrome 을 하나씩 돌려서 <span class="key">RAM 을 많이 먹는</span> 게 단점.</td>
    </tr>
    <tr>
      <td><strong>CLI 도구</strong></td>
      <td>Prettier, ESLint, <span class="term">TypeScript</span> 컴파일러, Webpack</td>
      <td>개발자 일상 도구의 큰 부분.</td>
    </tr>
    <tr>
      <td><strong>AI CLI 도구</strong></td>
      <td><strong>Claude Code, Codex, Cursor CLI, 대부분의 <span class="term">MCP</span> 서버</strong></td>
      <td>이 글이 시작된 이유.</td>
    </tr>
  </tbody>
</table>

즉 <span class="term">Node.js</span> 를 설치한다는 건 <span class="term">JavaScript</span> 생태계로 들어가는 관문을 여는 셈이에요. 요즘 AI 개발자 도구를 쓰려면 파이썬을 설치하듯 Node 도 한 번은 깔아둬야 합니다.

## <span class="term">TypeScript</span> 는 따로 배워야 하나

<span class="term">TypeScript</span> 는 <span class="key">JavaScript 에 정적 타입을 얹은 소스 언어</span> 예요 (Microsoft, 2012). 파이썬의 타입 힌트에 비유하면, mypy 같은 검사를 <strong>컴파일 단계에서 강제로</strong> 돌린 뒤 순수 <span class="term">JavaScript</span> 로 변환하는 방식입니다.

핵심은 하나예요. <span class="key">CLI 사용자가 실제로 돌리는 건 JavaScript 입니다.</span> <span class="term">TypeScript</span> 는 그 CLI 를 만드는 쪽에서 쓰는 것이고, 배포 시점엔 .js 로 컴파일되어 공개 패키지 레지스트리에 올라가요. Claude Code 도 마찬가지예요. 소스는 .ts 지만 여러분 컴퓨터에서 실행되는 건 .js 입니다.

그러니 AI CLI 를 쓰려고 <span class="term">TypeScript</span> 를 따로 배울 필요는 없어요. 소스를 들여다볼 때 .ts 파일을 마주치면 *"아, 빌드 전 원본이구나"* 정도로 읽고 넘어가시면 됩니다.

## 곁다리 1: Electron 은 왜 <span class="term">Chromium</span> 을 골랐나

위에서 Electron 앱이 "앱 안에 작은 브라우저를 통째로 넣는" 방식이라고 했어요. 그럼 그 브라우저로 왜 하필 <span class="term">Chromium</span> 을 골랐는지 (다른 브라우저도 있는데) 잠깐 풀어볼게요.

작은 교정부터. <span class="term">Chromium</span> 은 엔진이 아니라 "브라우저 그 자체" 입니다.

<div class="callout">
<strong><span class="term">Chromium</span></strong> = Google 이 주도하는 <strong>오픈소스 브라우저 프로젝트</strong><br/>
<strong>Chrome</strong> = <span class="term">Chromium</span> 에 Google 고유 기능을 얹은 상용 버전<br/>
<span class="term">Chromium</span> <strong>안의 엔진</strong> = Blink (HTML/CSS 렌더링) + <span class="term">V8</span> (<span class="term">JavaScript</span> 실행)
</div>

즉 "Electron 이 <span class="term">Chromium</span> 을 썼다" = <span class="key">크롬과 거의 같은 브라우저를 앱 안에 통째로 가져왔다</span> 는 뜻이에요.

한 줄 답: <span class="key">"법적으로 가장 자유롭고, 기술적으로 가장 앞서 있고, Node.js 와 엔진을 공유할 수 있는" 유일한 선택지</span> 였기 때문입니다.

<table class="compare-table">
  <thead>
    <tr><th>요소</th><th><span class="term">Chromium</span> (Blink + <span class="term">V8</span>)</th><th>WebKit (Safari)</th><th>Gecko (Firefox)</th></tr>
  </thead>
  <tbody>
    <tr><td>라이선스</td><td><strong>BSD — 상용 제품에 통째로 넣기 자유</strong></td><td>Apple 중심 개발</td><td>MPL — 까다로움</td></tr>
    <tr><td>발전 속도</td><td>빠름 (Google + 커뮤니티)</td><td>느림 (Apple 릴리스 주기)</td><td>중간</td></tr>
    <tr><td>임베딩 친화도</td><td>매우 좋음 (CEF 프레임워크 존재)</td><td>OS(macOS) 의존성 강함</td><td>가능은 하나 손 많이 감</td></tr>
    <tr><td><span class="term">JavaScript</span> 엔진</td><td><span class="term">V8</span> — 가장 빠르고 성숙</td><td><span class="term">JavaScriptCore</span></td><td><span class="term">SpiderMonkey</span></td></tr>
  </tbody>
</table>

가장 결정적인 이유는 <span class="key">Node.js 도 V8 을 쓴다는 점</span> 이었어요. Electron 이 <span class="term">Chromium</span> 을 고름으로써 자연스럽게 UI 쪽 <span class="term">JavaScript</span> 도, 시스템 쪽 <span class="term">JavaScript</span> 도 <strong>같은 <span class="term">V8</span> 위에서 실행</strong> 되는 구조가 됩니다. 만약 Gecko 를 골랐다면 <span class="term">V8</span> 과 <span class="term">SpiderMonkey</span> <strong>두 엔진이 공존</strong> 하는 앱이 돼서 메모리·유지보수가 훨씬 무거워졌을 거예요.

지금은 사실상 독점이에요. <span class="key">2020년엔 Microsoft Edge 마저 자체 엔진(EdgeHTML) 을 버리고 Chromium 으로 갈아탔습니다.</span> 주류 브라우저 중 <span class="term">Chromium</span> 기반이 아닌 건 <strong>Firefox 뿐</strong> 이에요.

## 곁다리 2: HTML / CSS / <span class="term">JavaScript</span> 의 관계

이 글 곳곳에서 웹·브라우저·Electron 이야기를 하면서 <strong>HTML / CSS / <span class="term">JavaScript</span></strong> 라는 세 단어가 같이 자주 나왔어요. 이 셋이 뭐고 어떻게 역할이 나뉘는지 짧게 정리합니다.

<table class="compare-table">
  <thead>
    <tr><th>기술</th><th>담당</th><th>한 줄 비유</th></tr>
  </thead>
  <tbody>
    <tr><td>HTML (HyperText Markup Language)</td><td><strong>구조·내용</strong> — 페이지에 무엇이 있는가</td><td>뼈대</td></tr>
    <tr><td>CSS (Cascading Style Sheets)</td><td><strong>스타일</strong> — 어떻게 보이는가 (색·폰트·레이아웃)</td><td>옷과 화장</td></tr>
    <tr><td><span class="term">JavaScript</span></td><td><strong>행동</strong> — 상호작용·동작·로직</td><td>움직임</td></tr>
  </tbody>
</table>

구체 예. "로그인 버튼" 을 만든다고 해보면.

<div class="callout">
버튼이 <strong>있다</strong> → HTML<br/>
그 버튼이 <strong>파란색, 둥근 모서리</strong> → CSS<br/>
버튼을 누르면 <strong>서버에 요청 보내고 알림이 뜸</strong> → <span class="term">JavaScript</span>
</div>

브라우저가 웹 페이지를 여는 과정도 대략 이 순서예요.

1. HTML 을 받아 <strong>DOM 트리</strong> 로 파싱 (페이지의 뼈대)
2. CSS 를 받아 각 요소에 스타일을 입힘
3. <span class="term">JavaScript</span> 가 실행되면서 그 DOM·CSS 를 <strong>실시간으로 조작</strong>

셋 다 "프로그래밍 언어" 일까요?

<table class="compare-table">
  <thead>
    <tr><th>기술</th><th>분류</th><th>설명</th></tr>
  </thead>
  <tbody>
    <tr><td>HTML</td><td>마크업 언어</td><td>변수·로직 없이 "구조를 선언" 하는 문법.</td></tr>
    <tr><td>CSS</td><td>스타일 시트 언어</td><td>조건부 규칙(미디어 쿼리) 은 있지만 계산·로직 없음.</td></tr>
    <tr><td><span class="term">JavaScript</span></td><td><strong>진짜 프로그래밍 언어</strong></td><td>변수·함수·루프·조건·클래스 전부 있음.</td></tr>
  </tbody>
</table>

파이썬 쪽엔 이런 구분이 없어요. 예컨대 <strong>Streamlit / Gradio</strong> 앱을 만들 때, 컴포넌트 선언(HTML 역할), 스타일 지정(CSS 역할), 버튼 동작(<span class="term">JavaScript</span> 역할) 이 <span class="key">하나의 Python 코드 안에 묶여</span> 있죠. 내부적으로 Streamlit 이 HTML/CSS/JS 로 변환해서 브라우저에 보내지만 개발자는 그걸 의식할 필요가 없습니다.

## 그럼 <span class="term">Python</span> 은 왜 여전히 강력한가

여기까지 읽으면 이런 의문이 들 수 있어요. <em>"<span class="term">JavaScript/Node.js</span> 가 이렇게 좋으면 그냥 거기로 다 옮기면 되는 거 아닌가? <span class="term">Python</span> 은 왜 아직도 대세지?"</em>

결론부터 말하면, 두 생태계는 <span class="key">경쟁 관계가 아니라 분업 관계</span> 예요. 각자 압도적으로 잘하는 영역이 따로 있습니다.

### <span class="term">Python</span> 이 여전히 지배하는 영역

### 왜 이렇게 갈라졌나

- <strong>NumPy</strong> 는 2006년부터, <strong>SciPy</strong> 는 2001년부터 있었어요: <span class="term">Node.js</span> 가 탄생한 2009년 시점에 이미 <span class="term">Python</span> 은 과학 계산의 표준이었습니다.
- 학계·연구실이 관성적으로 <span class="term">Python</span> 을 썼고, 현대 딥러닝 붐 (2012년 AlexNet 이후) 이 터지면서 <span class="term">Python</span> 은 더 가속됐어요.
- <span class="term">JavaScript</span> 는 반대로 브라우저에서 출발 → 웹 서버 → CLI 로 확장: <span class="key">숫자 계산 쪽은 태생적으로 강점이 없었음.</span>
- 그래서 <strong>"모델은 <span class="term">Python</span>, 개발자 도구는 Node"</strong> 라는 분업이 자연스럽게 굳어졌습니다.

1. <strong><span class="term">Python</span> 을 버릴 필요 전혀 없음.</strong> 모델·데이터·수치 계산은 계속 Python.<br/>
2. <strong><span class="term">Node.js</span> 생태계 "해독 능력" 은 갖추는 게 좋음.</strong> AI 개발자 도구가 거기에 있어서 설치·실행·트러블슈팅은 해야 해요.<br/>
3. 대부분의 실무에서는 <span class="key">두 언어를 "함께"</span> 쓰게 됩니다. <span class="term">Python</span> 으로 모델을 만들고, <span class="term">Node.js</span> 기반 CLI 를 열어 그 모델을 사용하는 식.
</div>

> <span class="term">Python</span> 은 <strong>과학·수치·AI 모델의 제왕</strong>, <span class="term">Node.js</span> 는 <strong>개발자 도구·CLI·웹 UI 의 제왕</strong>. 경쟁자가 아니라 <span class="key">위아래 층에서 협력하는 스택</span> 입니다.
