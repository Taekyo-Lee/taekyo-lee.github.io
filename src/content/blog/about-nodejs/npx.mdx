Claude Desktop 의 MCP 설정 파일을 처음 열었을 때, 거의 모든 서버 항목이 똑같은 모양이었어요. "command": "npx". 도대체 왜 매번 npx 인가? 답을 알면 단순합니다. <span class="key">MCP 생태계가 npx 의 대표적인 사용처</span> 이기 때문이에요.

## 한 줄 답

<div class="callout-accent">
<span class="term">npx &lt;패키지&gt;</span> 는 <span class="key">실행 가능한 패키지를</span> <span class="key">전역 설치하지 않고도</span> 그 자리에서 바로 실행합니다 (라이브러리는 대상이 아니에요. 자세한 건 바로 아래에서). 내부적으로는 레지스트리에서 내려받아 임시 캐시에 두고 실행한 뒤 끝. 파이썬의 uvx (또는 pipx run) 와 사실상 같은 개념이에요. AI 엔지니어에게는 특히 중요. <span class="key">MCP 서버 실행</span> 의 표준 패턴이 npx 입니다.
</div>

## 핵심: npx 는 CLI 실행 전용입니다

넘어가기 전에 반드시 짚어둘 포인트.

<div class="callout-warm">
npx 는 <span class="key">"CLI 가 있는 패키지"</span> 에만 의미가 있습니다. CLI 가 없는 패키지 (= 라이브러리) 에는 npx 를 쓸 이유 자체가 없고, 시도해도 에러가 나요.
</div>

npx 의 동작 자체가 <span class="key">"패키지가 제공하는 CLI 진입점을 실행하는 것"</span> 이에요. CLI 진입점이란 package.json 의 bin 필드에 선언된 실행 파일 (<a href={`${import.meta.env.BASE_URL}blog/bin-folder/`}>(10/14) node_modules/.bin/ — 심볼릭 링크와 shebang 한 줄의 협력</a> 에서 상세). bin 이 없는 패키지는 실행할 대상 자체가 없으니 npx 는 무의미해집니다.

  <thead>
    <tr><th>패키지 성격</th><th>bin 필드</th><th>예시</th><th>npx</th></tr>
  </thead>
  <tbody>
    <tr><td><strong>CLI 애플리케이션</strong></td><td>있음</td><td>create-react-app, prettier, MCP 서버, Claude Code</td><td><strong>이게 npx 의 유일한 용도</strong></td></tr>
    <tr><td><strong>라이브러리</strong></td><td>없음</td><td>react, lodash, axios, SDK 들</td><td><strong>무의미</strong> — <span class="term">npx react</span> 는 에러. 라이브러리는 코드에서 import 해서 쓰는 것</td></tr>
  </tbody>
</table>

<div class="callout-accent">
핵심 질문으로 바꾸면. "이 패키지 터미널에서 <span class="key">실행</span> 하는 거예요, 아니면 코드에서 <span class="key">import</span> 하는 거예요?"

- 실행 쪽이면 npx 나 <span class="term">npm install -g</span> 가 자연스럽고
- import 쪽이면 <span class="term">npm install</span> (로컬) 이 자연스러워요

(세부적으로는 "프로젝트 안에서만 쓰는 CLI 를 devDependencies 로 깔아 npm scripts 에서 호출" 같은 자리도 있어서 분류가 더 갈리는데, 큰 그림은 일단 위 두 갈래로 잡으면 됩니다.)
</div>

이 구분은 이전 글 ("CLI 는 -g, 라이브러리는 로컬") 과 정확히 같은 축이에요. npx 는 그 축에서 <span class="key">"CLI 를 -g 로 깔지 않고 한 번만 돌리는 대안"</span> 으로 추가된 선택지일 뿐입니다. 라이브러리는 npx 의 대상이 된 적이 없어요.

## npx ≠ 전역 설치: 자주 헷갈리는 지점

"npx 로 실행했으니 전역에 설치된 거 아닌가?" 하는 오해가 많은데, 둘은 다른 개념이에요. 결과가 "CLI 를 어디서나 실행 가능" 이라는 점만 같고, 그 외엔 전부 다릅니다.

    <tr><td><strong>제거 방법</strong></td><td><span class="term">npm uninstall -g &lt;pkg&gt;</span></td><td>캐시 비우기 (<span class="term">npm cache clean --force</span>) 또는 그냥 둠</td></tr>

<div class="callout">

직관적 차이:

- <span class="term">npm install -g</span> = <span class="key">"한 번 깔고 계속 쓸게요"</span>: 매일 쓰는 CLI 에 적합
- npx = <span class="key">"매번 새로 불러서 한 번만 쓸게요"</span>: 가끔 쓰거나, 매번 다른 버전이 필요하거나, 관리하기 번거로운 양일 때 (<span class="key">MCP 서버가 정확히 이 자리</span> — 수십·수백 개 서버를 일일이 전역 설치하긴 부담스러우니까)

</div>

실전 선택 기준:

<div class="callout-warm">

- Claude Code 를 매일 쓴다 → <span class="term">npm install -g @anthropic-ai/claude-code</span> (매 실행마다 캐시 체크·진입점 해석 오버헤드를 내는 것보다 한 번 깔아 두는 게 효율적)
- MCP 서버 20 개를 설정했다 → <span class="key">전부 npx</span>: -g 로 깔면 20 개를 일일이 설치 + 각각의 업데이트 관리를 해야 하지만, npx 는 설정 파일에 패키지명만 적어 두면 끝

</div>

파이썬 대응을 함께 두면 이 구분이 선명해요.

## 기본 사용법

```bash
npx <패키지>                   # 최신 버전으로 한 번 실행
npx <패키지>@1.0.42            # 특정 버전 실행
npx -y <패키지>                # 설치 확인 프롬프트를 자동으로 yes
```

예:

```bash
npx create-react-app my-app    # React 프로젝트 초기 뼈대를 만들어 주는 도구
                               # → 현재 폴더에 my-app/ 을 만들고 표준 React 구조 자동 생성
                               # (django-admin startproject 와 같은 자리)

npx prettier --check .         # 코드 포매터 (파이썬의 black / ruff format 과 같은 자리)
                               # --check = 수정하지 말고 포맷 검사만
                               # . = 현재 디렉토리의 모든 파일 대상
```

끝나면 시스템에 <span class="key">설치된 패키지는 없고</span>, 임시 캐시만 남습니다 (&#126;/.npm/_npx/ 아래).

## 내부 동작: 네 단계

<span class="term">npx &lt;패키지&gt;</span> 를 치면:

<div class="callout">

1. <span class="key">로컬 탐색</span>. 현재 프로젝트의 node_modules/.bin/ 에 같은 이름이 있는지 먼저 확인. 있으면 그걸 실행 (레지스트리 안 감). 이래서 프로젝트 내에서 <span class="term">npx jest</span> 같은 명령이 빠르게 돕니다
2. <span class="key">전역 탐색</span>. 시스템 전역 설치에 있는지 확인
3. <span class="key">레지스트리에서 받기</span>. 둘 다 없으면 npm registry 에서 내려받아 임시 캐시 (&#126;/.npm/_npx/&lt;해시&gt;/) 에 풂
4. <span class="key">실행 후 정리</span>. 패키지의 CLI 진입점 (package.json 의 bin 필드, <a href={`${import.meta.env.BASE_URL}blog/bin-folder/`}>(10/14) node_modules/.bin/ — 심볼릭 링크와 shebang 한 줄의 협력</a> 참고) 을 실행. 끝나도 전역 설치는 안 됨. 다음 실행 때도 캐시 재사용만

</div>

즉 npx 는 "있으면 쓰고, 없으면 잠깐 빌려와서 실행만 하고 돌려주는" 방식이에요.

## 왜 쓰는가: 네 가지 실전 시나리오

### ① 일회성 실행: 전역에 깔기는 아까움

잠깐 한 번만 쓰고 말 도구를 전역에 설치하고 관리하긴 부담스러울 때. 예: "지금 이 프로젝트에 Prettier 를 적용해 볼까?" 하는 탐색 단계.

```bash
npx prettier --check .
```

마음에 들면 그때 <span class="term">npm install -D prettier</span> 로 프로젝트 devDependencies 에 넣으면 됩니다.

### ② 버전 고정: 특정 버전을 즉시 실행

전역에 특정 버전을 고정해서 쓰기 애매할 때. 예: "이 스크립트는 typescript@5.3 으로 돌려야 함".

```bash
npx typescript@5.3 --noEmit
```

전역 설치는 한 버전밖에 못 담지만, npx 는 매번 원하는 버전을 부를 수 있어요.

### ③ Scaffolding (프로젝트 생성) 도구

"이건 한 번만 실행되고 끝" 의 전형.

```bash
npx create-react-app my-app
npx create-next-app
```

프로젝트를 만들고 나면 이 도구는 더 이상 쓸 일이 없으니 설치할 필요가 없어요.

### ④ MCP 서버 실행: AI 엔지니어의 기본 패턴

<span class="key">이게 AI 엔지니어가 실전에서 가장 많이 보는 npx 사용처</span> 예요. Claude Code · Cursor · Claude Desktop 같은 도구에서 MCP 서버를 등록할 때 설정 파일에 이런 모양이 들어갑니다.

```json
{
  "mcpServers": {
    "some-server": {
      "command": "npx",
      "args": ["-y", "@some-org/mcp-server"]
    }
  }
}
```

이 설정은 "MCP 서버가 필요할 때마다 <span class="key">npx 로 해당 서버 패키지를 내려받아 실행</span> 하라" 는 뜻이에요. -y 플래그는 첫 실행 시 "설치해도 되냐?" 는 프롬프트를 자동으로 yes 처리.

<div class="callout-warm">
<span class="key">왜 <span class="term">npm install -g</span> 가 아니라 npx 인가</span>. MCP 생태계는 서버가 수십·수백 개로 늘어나고 업데이트도 잦기 때문에 전역 설치로 관리하기 번거로워요. npx 를 쓰면 설정만으로 끝. 서버 쪽에서 버전을 올리면 사용자는 캐시만 갱신되면서 자연스럽게 따라갑니다.
</div>

## -y 플래그

npx 가 "이 패키지를 받아 실행해도 되냐?" 고 물어보는 확인 프롬프트가 있어요 (대화형 실행 때). -y 는 그걸 자동으로 yes 처리. <span class="key">MCP 설정처럼 사람이 답할 수 없는 환경에선 사실상 필수</span> 입니다.

## 파이썬과의 대응

  <thead>
    <tr><th>npm</th><th>Python</th></tr>

    <tr><td><span class="term">npx &lt;패키지&gt;</span>@&lt;버전&gt;</td><td>uvx --from &lt;패키지&gt;==&lt;버전&gt; &lt;커맨드&gt;</td></tr>

개념은 거의 1:1. uvx · pipx run 이 MCP · 에이전트 쪽에서 증가하는 것도 같은 흐름이에요. <span class="key">"설치 없이 잠깐 실행"</span> 이라는 수요가 커지면서 두 생태계 모두 비슷한 해법을 만든 것.

## 주의점

<div class="callout">

- <span class="key">네트워크가 필요</span>: 첫 실행 때는 레지스트리에서 받아와야 하니 오프라인에선 실패합니다. 캐시가 있으면 이후엔 네트워크 없이도 돌지만, 회사 방화벽·폐쇄망에선 이 점을 신경 써야 해요
- <span class="key">실행 때마다 약간의 지연</span>: 전역 설치 + 직접 실행보다는 조금 느림 (캐시 체크·진입점 해석 때문). 1~2초 수준이라 대부분 문제가 안 되지만, 매우 빈번하게 호출되는 경로라면 <span class="term">npm install -g</span> 쪽이 유리할 수 있음
- <span class="key">버전이 자동으로 변할 수 있음</span>: <span class="term">npx &lt;패키지&gt;</span> 는 최신 버전을 쓰려고 시도해요. 재현성이 중요하면 <span class="term">npx &lt;패키지&gt;</span>@&lt;정확한버전&gt; 으로 고정할 것

</div>

## 한 줄 요약

<div class="callout-accent">
npx 는 <span class="key">"설치 없이 잠깐 실행"</span>. 레지스트리에서 받아 임시 캐시에 두고 돌린 뒤 끝. 파이썬의 uvx 와 같은 자리예요. <span class="key">AI 엔지니어의 실전 용도는 대부분 MCP 서버 실행</span> 이고, 그 밖에도 scaffolding · 일회성 도구 · 버전 고정 실행에 많이 씁니다.
</div>

다음 글에서는 package-lock.json 이 정확히 무엇이고, 왜 "내 컴퓨터에선 되는데" 의 절반 이상을 이 한 파일이 막아주는지를 풀어 봅니다.
