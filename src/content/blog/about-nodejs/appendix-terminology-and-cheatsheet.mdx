npm 생태계를 돌아다니다 보면 <span class="key">비슷해 보이는 이름들</span> 때문에 자주 헷갈려요. 한자리에 모아 정리합니다.

### 한 문장 구분법

<div class="callout">

- node 는 <span class="key">"JS 를 실행"</span> 하는 것
- npm 은 <span class="key">"패키지를 관리"</span> 하는 것
- npx 는 <span class="key">"잠깐 실행"</span> 하는 것
- yarn · pnpm · bun 은 <span class="key">npm 의 경쟁자</span> (호환되긴 함)

## 의존성 네 종류: dependencies 계열

package.json 에 들어갈 수 있는 의존성 필드들이에요. 대부분의 프로젝트는 앞의 두 개만 씁니다.

<table class="compare-table">
  <thead>
    <tr><th>필드</th><th>의미</th><th>언제 설치되나</th></tr>
  </thead>
  <tbody>
    <tr><td>dependencies</td><td><strong>런타임</strong> 에 필요한 패키지</td><td>항상. transitive 하게 따라감</td></tr>
    <tr><td>devDependencies</td><td><strong>개발 · 빌드 · 테스트</strong> 시에만 필요</td><td>npm install (플래그 없이), --production 시 제외, <strong>남이 의존으로 가져가면 안 따라감</strong></td></tr>
    <tr><td>peerDependencies</td><td>"이 패키지를 쓸 사람은 <strong>당신 프로젝트에 X 를 따로 설치해야</strong> 합니다" 선언</td><td>자동 설치 안 됨 (npm 7+ 는 자동 설치 시도). 플러그인 · React 컴포넌트 라이브러리 패턴</td></tr>
    <tr><td>optionalDependencies</td><td>설치 실패해도 <strong>전체 설치는 성공</strong> 처리</td><td>플랫폼별 네이티브 바이너리 같은 조건부 의존성에 쓰임</td></tr>
  </tbody>
</table>

### peerDependencies 가 뭔지 간단히

예: 내가 <span class="key">React 컴포넌트 라이브러리</span> 를 만든다면, react 를 dependencies 에 넣으면 안 돼요. 사용자가 이미 자기 프로젝트에 React 를 쓰고 있을 텐데, 내 라이브러리가 또 설치하면 <span class="key">두 벌의 React</span> 가 공존해서 작동이 깨져요. 그래서:

```json
"peerDependencies": {
  "react": ">=18.0.0"
}
```

"내 라이브러리는 당신 프로젝트의 React 를 <span class="key">빌려 쓸 거에요</span>. 버전이 18 이상이기만 하면 됩니다" 라는 의미. Python 생태계엔 <span class="key">직접 대응이 없는</span> 개념.

### optionalDependencies: 언제 쓰이나

주로 <span class="key">플랫폼별 네이티브 바이너리</span> (예: macOS 용 바이너리가 Linux 에서 설치 실패하는 경우) 를 우아하게 건너뛰기 위함. 일반 AI 엔지니어 입장에선 거의 만날 일 없어요.

## 여러 패키지 매니저가 공존하는 이유

"왜 npm 하나만 쓰지 yarn, pnpm, bun 이 있지?". 역사적 맥락이에요.

<div class="callout">

- <span class="key">2010년대 초반</span>: npm 초기 버전이 느리고 (직렬 다운로드) 불안정함. node_modules 트리가 중복 · 충돌 많음
- <span class="key">2016</span>: Facebook 이 yarn 출시. 병렬 다운로드 · deterministic 한 lock 파일 (yarn.lock) · offline 캐시. npm 에 압력을 줌
- <span class="key">2017</span>: npm 5 가 대응. package-lock.json 도입, 속도 개선. 많은 기능이 yarn 과 수렴
- <span class="key">그 이후</span>: pnpm (2017, 디스크 절약형), bun (2022, 속도 혁신) 등 새 도구 등장

</div>

각자 해결하는 문제가 다릅니다.

- pnpm: 수십 개 프로젝트의 node_modules 합산이 수십 GB 가 되는 문제. 공용 store + symlink 로 해결
- bun: 전체 툴체인의 속도 (설치 · 번들 · 실행 등) 를 통합적으로 가속

### 그래서 뭘 써야?

<div class="callout-accent">
일단 <span class="key">npm 만 알아도 충분합니다</span>. 이유:

- 모든 AI CLI · MCP 서버의 설치 지침이 npm install -g · npx 를 기본으로 씀
- yarn · pnpm · bun 은 package.json 을 그대로 읽어서 어느 것을 설치하든 결과가 호환
- 속도 · 용량 차이는 프로젝트가 <span class="key">매우 커졌을 때</span> 체감되는 것: AI CLI 사용자 수준에선 체감 차이 거의 없음

</div>

"어떤 매니저를 골라야 하나?" 고민하지 말고, 필요한 순간 (회사 레포가 bun 을 쓰거나 pnpm-lock.yaml 만 있는 등) 에 그때 배우면 돼요.

## 개념 차이: 헷갈리지 말 것

### ① 격리 모델

<div class="callout">

- <span class="key">Python</span>: venv 를 명시적으로 만들어 활성화 (uv venv · source .venv/bin/activate). 활성화 안 하면 시스템 Python 을 씀
- <span class="key">npm</span>: 폴더별 node_modules/ 가 자동 격리. 활성화 개념 없음. 현재 디렉토리의 node_modules 가 자동으로 쓰임

</div>

### ② Transitive 의존성의 dev 처리

<div class="callout">

- <span class="key">Python (pip 기본)</span>: dev 그룹의 transitive 구분이 약함
- <span class="key">npm</span>: 구조적으로 명확. 4 번 글의 규칙 3 (내 패키지의 devDependencies 는 남에게 안 따라감)

</div>

### ③ 빌드의 의미

<div class="callout">

- <span class="key">Python</span>: 컴파일 · wheel 생성 정도가 "빌드". 라이브러리 배포는 소스 그대로 올리는 경우도 흔함

### ④ 스크립트 · 태스크 러너

<div class="callout">

- <span class="key">Python</span>: Makefile · justfile · tox 등 외부 도구로 해결. pyproject.toml 의 [project.scripts] 는 패키지가 노출할 CLI 만 담당
- <span class="key">npm</span>: package.json 의 scripts 가 내부 태스크 러너 + bin 이 외부 CLI 선언. 한 파일에 둘 다

</div>
