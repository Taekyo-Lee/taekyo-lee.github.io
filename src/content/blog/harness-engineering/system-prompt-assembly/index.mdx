한 번의 입력에서 system 블록은 <span class="key">**네 개의 서브 블록**</span> 이 순서대로 이어붙은 한 array 예요. 각 서브 블록마다 **수명** 이 달라요. 어떤 건 **세션 내부에선 동일** 하고, 어떤 건 **LLM 호출마다 새로 빌드** 돼요.

<div class="callout">
📝 <strong>note</strong> — 이 *수명* 분류는 technical 하게는 LLM API 의 <em>prompt cache 정책</em> 과 직결돼요. *모든 세션 거의 동일* 한 서브 블록은 cache prefix 에 한 번 등록되면 이후 LLM 호출이 그 위에서 hit 돼서, <em>응답이 빨라지고 요금도 줄어들어요</em>. *매 LLM 호출 다른* 서브 블록은 cache 가 안 잡혀서 매번 full price 고요.
</div>

이번 포스트가 다루는 건 **셋**이에요.

1. **네 개의 서브 블록** 이 어떤 콘텐츠로 채워지는지
2. 그 안의 **21 개 slot** (서브 블록을 더 잘게 쪼갠 단위, *정적 7 개 + 동적 14 개*) 이 어떤 순서로 쌓이는지
3. 그 섹션을 **정적과 동적** 두 부분으로 가르는 **경계선**

이 셋이 system 블록의 <span class="key">**골격**</span> 이에요.

## 네 개의 서브 블록

네 서브 블록이 어떻게 다른지를 한 표로 잡고 시작해요.

- **서브 블록 #1: attribution header** — Claude Code 의 billing/tracking 한 줄 (`x-anthropic-billing-header: cc_version=...`). Claude Code 빌드가 같으면 *모든 세션 거의 동일*.
- **서브 블록 #2: system prompt prefix** — Anthropic 의 base persona 한 줄 (`You are Claude Code, Anthropic's official CLI for Claude.`). 모든 세션에서 동일.
- **서브 블록 #3: static system blocks** — *모든 세션에서 거의 동일한* 7 개 slot 묶음. base persona 의 행동 규칙들.
- **서브 블록 #4: dynamic system blocks** — *세션마다 다른* 14 개 slot. environment 정보, settings, MCP, gitStatus trailer 등. 같은 세션 안에서도 매 LLM 호출마다 다를 수 있음.

서브 블록 #3 과 #4 사이에 한 marker 가 위치하고 있어요. <span class="key">**`SYSTEM_PROMPT_DYNAMIC_BOUNDARY`**</span>. 이 한 줄이 **변하지 않는 부분** 과 **변할 수 있는 부분** 을 가르는 구분선이에요.

## 정적과 동적의 경계

21 개 slot 중 <span class="key">**slot 7 과 slot 8 사이**</span> 에 한 marker 가 위치해요. **`SYSTEM_PROMPT_DYNAMIC_BOUNDARY`**. 이 한 줄이 **변하지 않는 부분** 과 **변할 수 있는 부분** 을 가르는 경계예요. `splitSysPromptPrefix` (`utils/api.ts:321`) 가 array 를 둘로 나눕니다.

### 정적 부분의 의미

sections 1-7 은 **모든 세션에서 거의 동일**해요. 같은 model 을 쓰는 모든 LLM 호출이 이 7 개 slot 의 **byte-identical 한 prefix** 를 공유합니다. <span class="key">**base persona + 행동 규칙**</span> 의 자리. Anthropic 이 한 번 정해 두고 거의 안 바꾸는 영역이에요.

<div class="callout">
📝 <strong>'<span style={{ color: '#dc2626' }}>거의</span>' 동일하다고?</strong> — 정적 부분도 세션마다 변할 수 있는 자리가 셋 있어요.

- **`outputStyle` 토글**: intro 라인 다시 쓰기 + `# Doing tasks` 제거 가능 (4편 자세히)
- **Claude Code 빌드 update**: `cc_version` 변경 + Anthropic 의 base prompt 텍스트 미세 조정 가능
- **model 변경**: 일부 sub-section 이 model 별로 미세 다름 (예: `frc` 같은 model-dependent slot)

세 trigger 가 흔하진 않으니 *거의* 라는 단어를 썼지만, <strong style={{ color: '#dc2626' }}>언제든 동일은 아니라는 점</strong> 기억해 두세요.
</div>

### 동적 부분의 의미

sections 8-21 은 **매 세션마다** (어쩌면 **매 LLM 호출마다**) 달라질 수 있어요. cwd 가 다르고, settings 가 다르고, MCP 서버가 다르고, scratchpad path 가 다름. <span class="key">**환경 · 설정 · 콘텐츠**</span> 의 자리예요.

<div class="callout">
💡 <strong>TIP</strong> — slot 17 (<code>summarize_tool_results</code>) 은 <em>settings 로 끌 수 없는 메타 지시</em> 예요. 매 LLM 호출에 항상 위치해서 모델에게 <em>self-summarization</em> 을 일관되게 요청해요.

> *When working with tool results, write down any important information you might need later in your response, as the original tool result may be cleared later.*

한 줄짜리라 token 부담은 무시할 정도지만, fleet 전체로 보면 의미 있게 누적되는 자리예요.
</div>
