System Block (<span class="k">gen_ai.system_instructions</span>) 을 열어 보면 가장 아래에 항상 이런 블록이 있어요 (git project 폴더에서 실행한 경우):

[*System Block*](../system-prompt-assembly/) 의 어셈블리 그림으로 보면 *slot 21 (trailer)* 자리. <span class="k">appendSystemContext</span> 가 dynamic blocks (slots 8-20) 다음에 통째로 붙입니다. 별 거 아닌 듯 짧지만 (git project 라면) <span class="key">모든 trace 에 항상</span> 보여요. **이번 글의 주인공**.

## 다섯 줄. 어디서 왔나

<span class="k">getGitStatus()</span> 가 **5 개의 git 명령** 을 <span class="key">동시에</span> 돌려서 만듭니다.

| 명령 | 채우는 줄 |
|---|---|
| <span class="mod">git branch --show-current</span> | <span class="k">Current branch:</span> |
| <span class="mod">git remote show origin</span> (HEAD ref) | <span class="k">Main branch (you will usually use this for PRs):</span> |
| <span class="mod">git status --short</span> | <span class="k">Status:</span> (2K char cap, 넘으면 truncation marker 추가) |
| <span class="mod">git log --oneline -n 5</span> | <span class="k">Recent commits:</span> |
| <span class="mod">git config user.name</span> | <span class="k">Git user:</span> (없으면 줄 자체를 omit) |

## *one-shot snapshot*. 의도된 정적

위 블록의 첫 줄을 다시 봅시다:

> *Note that this status is a snapshot in time, and will not update during the conversation.*

이게 **load-bearing** 한 줄이에요. *"trust 하지 마"* 라고 모델에게 명시. 왜?

claude **프로세스 시작 시 한 번만 계산** 해서 trailer 자리에 넣어두면 <span class="key">그 프로세스 안에선 끝까지 안 바뀌어요</span>. 5 turn 후에 모델이 보는 gitStatus 도 *글자 그대로 같음*. 만약 매 turn 새로 계산한다면 파일 한 줄만 바뀌어도 trailer 가 달라지고, 그러면 prompt prefix 도 달라져요. <span class="key">cache 가 깨지고 매 turn full price</span>.

그러니 *snapshot* 으로 한 번만 넣고, **"필요하면 직접 돌려"** 라는 escape hatch 를 따로 줍니다. <span class="key">대략의 위치 감만 잡는 용도는 snapshot 으로, 정확한 현재 상태가 필요한 순간엔 모델이 직접 가져오기</span>.

<div class="callout">
<strong><span class="mod">Current&nbsp;branch:</span> 가 다르면 다른 *프로세스 invocation* (대개 다른 session)</strong>

gitStatus 는 *claude 프로세스 시작 시 한 번 capture* 되어 그 프로세스 안에선 안 바뀝니다. 그러니 같은 사용자의 두 trace 에서 *Current branch:* 가 다르게 찍혀 있다면, 두 trace 가 *서로 다른 프로세스 invocation 이라는 쉬운 힌트 중 하나*. 보통은 다른 session 이기도 해서 trace 들을 *각 작업 단위로 묶어 볼 때* 의외로 쓸 만해요.

다만 <span class="mod">claude --resume</span> 으로 이어 받는 경우 sessionId 는 보존되지만 새 프로세스라 gitStatus 가 다시 capture 됩니다. *같은 session 안에서도 두 branch 가 가능* 한 유일한 예외.
</div>

## subagent 변형. *일부는 gitStatus 가 빠집니다*

[*Subagents*](../subagents/) 에서 자세히 보지만 미리 한 번 짚으면, *여러 가능한 subagent 중에서* built-in <span class="k">Explore</span> 와 <span class="k">Plan</span> subagent 는 *부모 세션* (subagent 를 spawn 한 main agent) 으로부터 systemContext 를 <span class="key">gitStatus 뺀 채로</span> 상속받아요.

이유는 <span class="k">runAgent.ts</span> 의 코드 주석에 또렷이 설명되어 있어요. Explore 와 Plan 은 **읽기 전용 검색 subagent** 라 git 상태 정보가 본인 일에 거의 안 쓰여요. 게다가 부모로부터 물려받을 gitStatus 는 *부모 프로세스 시작 시점의 snapshot* 이라 어차피 낡은 값. 정말 필요한 순간엔 subagent 가 직접 <span class="mod">git status</span> 한 번 돌려서 그 자리에서 신선한 값을 받으면 그만입니다. 이 작은 결정 하나로 <span class="key">Anthropic 전체 사용 단위에서 매주 상당량의 토큰</span> 이 절약된다는 게 주석의 마지막 줄.

Explore subagent 의 trace 에서 system prompt 끝에 <span class="k">gitStatus:</span> 가 *없으면* 정상이에요. 있으면 strip 로직이 안 돈 거.

## System Block 내 위치. *왜 trailer 인가*

[*System Block*](../system-prompt-assembly/) 의 골격에서 gitStatus 가 *동적 부분의 마지막 trailer* 자리에 가는 이유가 또렷해져요.

- **프로세스마다 fresh 함**. 같은 repo 라도 가끔 branch 가 바뀌니, session 간 cache 공유 어려움 → 정적 부분에는 못 들어감
- **프로세스 안에선 stable**. 일단 들어가면 안 바뀜 → 한 번 넣어두면 그대로 둬도 됨
- **Anthropic 의 base 가 결정 못 함**. 사용자의 cwd · branch · status 에 의존 → 동적 영역
- **프로세스의 *어떤 단계* 든 prefix 의 일부* 이긴 해야 함**. 동적 부분이 적당한 자리

여기까진 *왜 동적* 의 이유. 그럼 *동적 안에서도 왜 마지막 (trailer)* 인가, 두 가지 자연스러운 이유가 또 있어요:

- **종류가 달라요**. 동적 슬롯들 (memory, MCP server instructions, scratchpad 등) 은 *harness 가 결정* 한 정보. gitStatus 는 *사용자 작업 디렉토리 sniff*. 책임이 다르니 별도 trailer 로 분리하면 시각적으로 또렷.
- **append 의 단순함**. <span class="k">appendSystemContext</span> 는 *어셈블리가 끝난 뒤 끝에 한 줄 덧붙이는 단일 목적 함수*. 동적 한가운데에 끼우려면 ordering 로직과 결합돼야 해요. trailer 는 *기존 어셈블리를 건드리지 않고 한 블록 추가* 라 변경 표면이 가장 작음.

그러니 <span class="k">appendSystemContext</span> 가 어셈블리의 **마지막 단계** 에서 한 줄로 추가합니다. 정적 prefix 깨지 않고, 동적 부분의 끝에 붙어 <span class="key">프로세스-안 안정성</span> 을 누리는 자리.

## tracing 의 작은 단서들

| 찾을 것 | 의미 |
|---|---|
| <span class="k">gitStatus:</span> 블록 자체가 없음 | git project 폴더가 아니거나, [*Settings 와 MCP*](../settings-and-mcp/) 에서 본 <span class="k">includeGitInstructions</span> 같은 settings 로 git 정보를 제외했음 |
| <span class="k">gitStatus:</span> 블록은 있는데 <span class="k">Git user:</span> 줄 없음 | <span class="mod">git config user.name</span> 이 설정 안 되어 있음 |
| <span class="k">Status:</span> 끝에 <span class="mod">… (truncated …)</span> marker | 작업 중인 파일이 2K 자 넘게 변경되어 잘림 (5 git 명령 표 참조) |
| 같은 사용자 trace 인데 두 trace 의 <span class="k">Current branch:</span> 가 다름 | 두 프로세스 invocation. 보통 다른 session 이지만 <span class="mod">--resume</span> 으로 이어 받았다면 같은 session 일 수도 있음 |
| Explore / Plan subagent 의 system prompt 에 <span class="k">gitStatus:</span> 가 보임 | gitStatus 제외 로직이 작동 안 한 거예요. 의외의 시그널, 디버깅 단서 |

## Take-home

이 글의 정리:

<blockquote>
<p>gitStatus 는 System Block의 <em>마지막 trailing block</em>. 5 git 명령이 동시에 돌아 한 snapshot 을 만들고, <span class="key">프로세스 시작 시 한 번 들어간 채로 그 프로세스 끝까지 가요</span>.</p>
<p>위에서 본 <em>"snapshot 이니까 그대로 믿지 말고 정확한 정보가 필요하면 직접 git status 돌려"</em> 라는 한 줄이 핵심이에요. <span class="key">대략의 감은 snapshot 으로, 정확한 상태는 모델이 직접 가져오기</span>. 그리고 동적 영역 안에서도 <em>맨 끝 (trailer)</em> 에 가는 이유는, gitStatus 가 <em>사용자 작업 디렉토리 정보</em> 라 다른 동적 슬롯들 (memory · MCP 같은 harness 정보) 과 종류가 다르고, <em>한 번에 추가하는 게 가장 단순</em> 해서.</p>
<p>같은 디자인 패턴 (<span class="key">작은 단서만 주고 정확한 정보는 모델이 직접 가져오기</span>) 이 시리즈 다른 글에서도 반복돼요. <a href="../settings-and-mcp/"><em>Settings 와 MCP</em></a> 의 scratchpad 디렉토리 경로, <a href="../sessions-transcripts-and-memory/"><em>세션의 기억</em></a> 에서 보게 될 compaction 후 transcript 파일 경로 도 같은 모양.</p>
</blockquote>
