## 7 가지 큰 correction. *문서와 trace 가 어긋난 자리*

Phase 2 검증의 진짜 가치가 여기에 모입니다. 문서(가이드 초안) 가 *어떻게 쓰여 있었는지* 와 실제 trace 가 *어떻게 보였는지* 가 어긋난 자리 8 개. 각각이 시리즈의 어느 편에서 발견됐는지를 함께 메모.

## 작은 발견들. *trace 디버깅에서 자주 만나는 한 줄들*

위 8 개가 *큰* correction 이라면, 그 외 *작은 발견* 도 의외로 많아요. 일관된 telemetry pattern 을 만드는 자리들 위주로 정리.

### <span class="k">system_reminders</span> 는 *omit*, not *empty*

새 reminder 가 없는 turn 에선 attribute 자체가 사라져요. 빈 string 이 아니라 *키가 metadata 에 없음*. *"reminder 가 들어갔나"* 의 빠른 검사:

| <span class="k">system_reminders</span> 상태 | 의미 |
|---|---|
| 있고 + 텍스트 | 새 reminder 가 들어간 turn |
| 없음 (key 자체) | 이 turn 엔 새 reminder 없음 |

같은 패턴이 <span class="k">#&nbsp;claudeMd</span> 의 disable 검사에도 적용됨 ([*Long-term Memory*](../permanent-memory-three-channels/)).

### subagent 의 <span class="k">tool_result</span> 안에 *agentId + SendMessage hint + &lt;usage&gt;* 다 있음

[*Subagents*](../subagents/) 에서 본 디테일. 부모 LLM 이 *텍스트로* 받는 정보:

```
agentId: aa20a239aca893577
(use SendMessage with to: 'aa20a239aca893577' to continue this agent)
<usage>total_tokens: 6127, tool_uses: 1, duration_ms: 5449</usage>
```

multi-agent 흐름이 *prompt instruction* 으로 wired (rigid API affordance 가 아님). cost 정보가 모델에 도달해 budget-aware 한 다음 결정 가능.

### compactor 는 *main-thread system prompt 를 그대로 쓰고* + 9-section template 을 final user message 로

[*세션의 기억*](../sessions-transcripts-and-memory/) 의 compactor mechanism. specialized system prompt 가 아니라, *main-thread 와 같은 prompt + 한 user message 안에 9-section extraction template* 이 들어감. anti-tool-use 가드까지 prepend (<span class="mod">Respond with TEXT ONLY. Do NOT call any tools. Tool calls will be REJECTED...</span>).

### post-compact 시 *transcript pointer escape hatch*

absolute path 가 prompt 에 들어감:

```
If you need specific details from before compaction, read the full
transcript at: /home/jet/.claude/projects/.../38ef7c82-...jsonl
```

소프트 compaction (요약), 하드 retrieval (원본) 의 escape hatch 가 같이 제공. 같은 패턴 (작은 hint + 모델이 직접 fetch) 이 [*gitStatus*](../git-context/) 의 truncation marker 에서도 보임 (<span class="mod">If you need more information, run "git status" using BashTool</span>).

### hook 의 auto-generated name 은 <span class="k">:matcher</span> suffix 포함

[*Hooks*](../hooks-context-injection/) 의 발견. <span class="k">PreToolUse:Bash</span> 같은 형태로 prompt 에 들어감. trace grep 패턴에 영향. 이 디테일을 모르면 *"왜 hook 이 안 걸려"* 디버깅에 시간이 빠짐.

### <span class="k">cc_version</span> suffix 가 call type 마다 다름

[*Subagents*](../subagents/) 의 부수 발견. 같은 process, 같은 binary 인데 parent 호출과 subagent 호출이 서로 다른 cc_version suffix 를 갖음 (e.g., parent <span class="k">dd8</span>, subagent <span class="k">f17</span>). 의심: content hash 일 가능성.

### dev mode 의 <span class="k">summarize_tool_results</span> (slot 17) 은 *settings 로 못 끔*

[*System Block*](../system-prompt-assembly/) 과 [*Settings 와 MCP*](../settings-and-mcp/) 의 한 줄. *"When working with tool results, write down any important information you might need later in your response, as the original tool result may be cleared later."* 는 모든 trace 에 항상. self-summarization 의 always-on 메커니즘.

## 시리즈가 보여준 *디자인 모티프* 셋

8 편을 다 읽고 다시 종합하면, 몇 가지 디자인 모티프가 시리즈 전반에 반복돼요.

### 1. *Thin advertise / Fat invoke*. 세 군데에서 반복

- **Skills** ([*Skills 와 Slash Commands*](../skills-and-slash-commands/)): 매 turn 옅은 한 줄 카탈로그, 호출 시 body 한 번 inline
- **MCP tools** ([*Settings 와 MCP*](../settings-and-mcp/)): ToolSearch defer mode. 매 turn ToolSearch 하나만, 호출 시 schema lazy load
- **Subagents** ([*Subagents*](../subagents/)): 부모는 agent 의 description / name 만 봄, body 는 별개 세션의 system prompt

같은 lazy load 원칙. *advertise 는 cheap, invoke 는 expensive*.

### 2. *Soft mechanism + Hard escape hatch*. 여러 군데에서 반복

- **gitStatus** ([*gitStatus*](../git-context/)): 2K cap + truncation marker + *"BashTool 로 직접 돌려"* 한 줄
- **Compaction** ([*세션의 기억*](../sessions-transcripts-and-memory/)): summary inline + transcript pointer (.jsonl 의 absolute path)
- **Scratchpad** ([*Settings 와 MCP*](../settings-and-mcp/)): dir 경로 한 줄만 prompt 에 + 안의 파일은 모델이 read 해야

소프트한 첫 단계 + 모델이 필요시 직접 fetch 하는 명시 escape. *최소 비용으로 충분한 정보 + 필요하면 더 가져가*.

### 3. *Per-turn delta vs Session-stable*. cache 전략의 단일 원칙

- **Session-stable** (cacheable): static system prefix (sections 1-7), <span class="k">#&nbsp;claudeMd</span> (content 가 같으면 prefix-stable), tool 정의
- **Per-turn delta** (uncached): dynamic system blocks, latest user input, attachments, hook injected reminders

cache prefix 의 절단점이 이 두 카테고리를 가르는 한 marker. 시리즈 전반의 cost 분석이 이 한 분류 위에서 다 정렬됨.

## 시리즈를 마치며

이 시리즈를 시작할 때 [*Harness Engineering과 Context Engineering*](../harness-vs-context-engineering/) 의 cliffhanger 는:

> *"이 코드 분석해줘"* 같은 짧은 한 줄을 던졌는데 trace 를 열어 보면 12,000 토큰. 그 사이를 채운 토큰들은 어디서 왔을까요?

답은 8 편에 걸쳐 지도 위에 다 그려졌어요. *14개의 source 가 3 top-level keys 의 각 자리에 위치합니다*. 각 source 는 *왜 거기에* 가는지. *cache 효율*, *isolation*, *update 빈도*, *모델의 attention 위치* 등. 설계의 선택이 있어요.

진짜 가치는 그 다음에 있다고 봅니다. 이론으로 그린 이 지도와 *실제 trace 의 모양* 이 어긋나는 자리들. Phase 2 의 40+ 발견 중 위 7 개의 큰 correction 이 그 어긋남의 elemental 리스트예요.

<div class="surprise">
<span class="label">⚡ 마지막 한 줄</span> 시리즈가 끝나도 trace 디버깅의 모서리는 계속 남아요. 가이드의 *다음 phase* 는 <em>verification 자체를 telemetry 개선으로 잇는 것</em>. per-attachment token count, cache-buster identification, subagent ↔ parent trace linkage 같은 *구체적인 개선 작업들* 입니다. 이 시리즈는 *지도* 까지만. 그 위에서의 항해는 trace 마다 새로 시작됩니다.
</div>

## Take-home

이 시리즈의 정리:

<blockquote>
<p>Claude Code 가 LLM 에 보내는 한 번의 입력. 그 안에 14 개의 context source 가 4 가지 자리 (시스템 프롬프트 정적/동적 부분, tools 블록, user-message slot 0 의 합성 reminder, 사용자 명시 호출/hook injection 의 latest user message) 에 분산돼 위치합니다.</p>
<p>같은 디자인 모티프가 여러 source 에서 반복돼요. <span class="key">thin advertise / fat invoke</span> (Skills · MCP · Subagents), <span class="key">soft mechanism + hard escape hatch</span> (gitStatus · compaction · scratchpad), <span class="key">per-turn delta vs session-stable</span> 의 cache 절단점.</p>
<p>이론으로 그린 지도와 실제 trace 가 어긋나는 7 개의 자리가 있습니다. resume 시 옛 reminder 사라짐, post-compact 의 messy wire format, slash 의 XML 메타 헤더, line-prefix <span class="mod">!cmd</span> 의 침묵, <span class="mod">@include</span> additive, OpenAI shim coalesce, <span class="mod">outputStyle</span> 의 4 효과. 이 7 개가 trace 디버깅에서 가장 자주 만나는 <em>모서리들</em>.</p>
<p>시리즈가 <em>지도</em> 였다면, 그 위에서의 항해는 다음 phase. 더 자세한 telemetry, 더 정밀한 cost attribution, 더 명료한 cross-trace linkage. <span class="key">verification 의 끝이 아니라 telemetry 개선의 시작</span>.</p>
</blockquote>

여기까지 와 주신 분, 고맙습니다. 시리즈를 따라 trace 를 같이 들여다보면서 *"아 그래서 이게 여기 있는 거구나"* 의 작은 순간이 한두 번이라도 있었다면, 이 글들의 목적은 충분히 달성됐어요.

다음 시리즈는 *telemetry 개선 작업* 자체를 다룹니다. 발견한 8 개의 telemetry gap 을 Claude Code 코드에 어떻게 wire 하는지의 실전. 곁에 두는 도구가 아니라 *만지는 도구* 로의 한 걸음 더.
