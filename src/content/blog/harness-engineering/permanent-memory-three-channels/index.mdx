<div class="callout-accent">
<strong>💡 Long-term memory 란?</strong>

한 session 의 *working memory* (context window) 가 끝나도 *session 을 넘어 보존되어 다음 turn 에도 자동으로 model 에 도달* 하는 영구 기억이에요.

Claude Code 에서는 **filesystem 의 실제 파일** 로 구현됐어요. 두 source 가 있습니다:

- *사람이 직접 적은 instructions*: <span class="key">CLAUDE.md</span> (Managed / User / Project / Local 4 종 scope)
- *모델이 스스로 작성한 노트*: <span class="key">MEMORY.md</span> + <span class="key">topic 노트</span>들

CLAUDE.md 와 MEMORY.md 는 매 turn 자동으로 LLM 에 도달하고, topic 노트의 본문은 모델이 필요할 때 retrieval 로 가져옵니다 (자세한 mechanic 은 본문에서).
</div>

### Long-term Memory 의 source 들

앞에서 봤었던 내용 기억나시나요?

<blockquote>
*"내가 쓴 CLAUDE.md 는 당연히 system prompt 안에 들어가는 거 아니야?"* 라고 생각하기 쉽죠. 그런데 실제로는 그렇지 않아요. <span class="key">CLAUDE.md · MEMORY.md · 오늘 날짜</span> 처럼 harness 가 끌어오는 컨텍스트는 System Block 이 아니라, Messages 배열의 첫 번째 user message 에 합성으로 prepend 되는 <span class="k">&lt;system-reminder&gt;</span> 블록 안에 들어갑니다.
</blockquote>

<div class="callout-accent">
<strong>💡 Messages Block 의 두 memories</strong>

## CLAUDE.md

### Full Hierarchical Search

그러면 이 파일들이 어디에 있는지를 harness 는 어떻게 찾아낼까요. Managed 와 User scope 는 고정 경로 (<span class="mod">/etc/claude-code/...</span>, <span class="mod">&#126;/.claude/...</span>) 라 직접 조회로 끝나지만, **Project 와 Local scope 는 디렉토리를 거슬러 올라가며 탐색** 합니다.

방식은 단순해요. **현재 작업 디렉토리에서 출발해 파일시스템 root 까지 한 단계씩 거슬러 올라가면서**, 매 단계마다 네 파일 패턴을 확인합니다:

- <span class="mod">&lt;repo&gt;/CLAUDE.md</span> → Project 로 분류
- <span class="mod">&lt;repo&gt;/.claude/rules/&#42;.md</span> → Project 로 분류
- <span class="mod">&lt;repo&gt;/&lt;any_subfolder&gt;/CLAUDE.md</span> → Project 로 분류
- <span class="mod">&lt;repo&gt;/CLAUDE.local.md</span> → Local 로 분류 <span class="not-rec">⚠ 사용 권장되지 않음</span>

<pre class="walk-diagram">
       /home/jet/repo/src/feature   ← <span class="cwd-tag">cwd (Claude Code 실행 폴더)</span>, 시작
       ↓
       /home/jet/repo/src           ← 여기에도 있나?
       ↓
       /home/jet/repo               ← 여기는?
       ↓
       /home/jet                    ← 여기도?
       ↓
       /                            ← 끝

(↓ 한 방향으로만 검색, 옆 sibling / 아래 subfolder 는 보지 않음)
</pre>

이렇게 위로 한 번 훑은 뒤 수집된 디렉토리 목록을 *root → cwd* 순으로 뒤집어, 그 순서대로 파일을 적재해요. **cwd 에 가까운 파일이 결과 리스트의 뒤쪽에 자리잡습니다.** 모델은 prompt 끝부분에 더 집중하는 경향이 있어서, 적재 순서가 자연스럽게 *attention 우선순위* 와 정렬돼요.

<div class="callout-warm">
<strong>🧪 실험 1 — *precedence* (어느 쪽이 이기는가)</strong>

<span class="mod">&#126;/.claude/CLAUDE.md</span> (User scope) 에 *"my nickname is CAPTAIN AMERICA"*, <span class="mod">&lt;repo&gt;/.claude/CLAUDE.md</span> (Project scope) 에 *"my nickname is IRON MAN"* 을 적어 두고 Claude Code 에게 nickname 을 물었습니다.

대답은 이랬어요: *"CLAUDE.md 에 따르면 CAPTAIN AMERICA 와 IRON MAN 두 가지가 있지만, 우선순위에 따라 IRON MAN 이 맞는 것 같아요."*

두 source 모두 prompt 안에 있어서 모델은 **둘 다 인지** 합니다. 그런데 *어느 쪽을 답으로 줄지* 결정할 땐 cwd 에 더 가까운 (= prompt 끝부분에 더 가까운) Project scope 를 따라가요. 일종의 <strong>mental overwrite</strong> 라고 부를 수 있는 현상. 정보는 보이지만, 결정은 후자에 맡기는 거예요.
</div>

<div class="callout-warm">
<strong>🧪 실험 2 — *visibility* (애초에 보이는가)</strong>

이번엔 같은 이름의 충돌을 피하고, 세 곳에 *서로 다른 사람의 nickname* 을 심었습니다:

- <span class="mod">&#126;/.claude/CLAUDE.md</span> (User scope) → *"Julia's nickname is BLACK ROSE"*
- <span class="mod">&#126;/&lt;repo&gt;/CLAUDE.md</span> (Project, repo root) → *"Tom's nickname is CAPTAIN AMERICA"*
- <span class="mod">&#126;/&lt;repo&gt;/&lt;subfolder&gt;/CLAUDE.md</span> (Project, repo 안 subfolder) → *"Jimmy's nickname is IRON MAN"*

그리고 cwd 를 바꿔가며 두 번 물었어요:

  <thead>
    <tr>
      <th>cwd</th>
      <th>Julia?</th>
      <th>Tom?</th>
      <th>Jimmy?</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td><span class="mod">&#126;/&lt;repo&gt;</span></td>
      <td>BLACK ROSE ✅</td>
      <td>CAPTAIN AMERICA ✅</td>
      <td><em>I don't know</em> ❌</td>
    </tr>
    <tr>
      <td><span class="mod">&#126;/&lt;repo&gt;/&lt;subfolder&gt;</span></td>
      <td>BLACK ROSE ✅</td>
      <td>CAPTAIN AMERICA ✅</td>
      <td>IRON MAN ✅</td>
    </tr>
  </tbody>
</table>

Jimmy 만 cwd 위치에 따라 *보였다 안 보였다* 합니다. **walk 는 cwd 에서 root 쪽으로만 올라갑니다. cwd 의 *아래* (subfolder) 도, *옆* (sibling 디렉토리) 도 살피지 않아요.** 이게 walk 의 단방향성을 보여주는 결정적 증거예요.

그래서:

- *Project scope* 라는 이름이 *"repo 단위 영구 anchor"* 처럼 들리지만, 실제 anchor 는 **cwd 의 ancestor chain** 입니다.
- subfolder 안 CLAUDE.md 가 발효되려면 **그 subfolder (또는 더 아래) 에서 claude-code 를 실행** 해야 함. 단순히 *"repo 안 어디든 두면 되겠지"* 가 아니에요.
- 위 다이어그램의 <span class="mod">&lt;repo&gt;/&lt;any_subfolder&gt;/CLAUDE.md</span> 표기도 이 맥락에서 다시 읽어 주세요. *"repo 안 임의의 subfolder 에 두면 자동 적용"* 이 아니라 *"cwd 가 그 subfolder 이거나 더 아래일 때만 visibility 가 생긴다"* 가 정확.
- *어디서 실행하든 자동으로 attach* 되게 하고 싶다면 <span class="mod">.claude/rules/&#42;.md</span> 의 <span class="k">paths:</span> frontmatter (conditional matching) 가 올바른 도구예요. **walk 가 못 하는 일을 paths-glob 매칭이 메워주는 셈입니다.**
</div>

### .claude/rules/&#42;.md 와 conditional matching

<span class="mod">.claude/rules/&#42;.md</span> 디렉토리 안의 모든 *.md* 파일이 자동으로 memory 로 로드돼요. CLAUDE.md 와 같은 description suffix 를 자기 scope 에서 상속받습니다.

일반 CLAUDE.md 와의 핵심 차이는 frontmatter 의 <span class="k">paths:</span> 필드. **rules 는 *glob 매칭 조건부 로딩* 이 가능**해요.

```markdown
---
paths: src/api/**/*.ts
description: API layer 의 규칙
---
- Always return Result<T, E>
- Use Zod for validation
```

위 rule 은 모델이 <span class="mod">src/api/&#42;&#42;/&#42;.ts</span> 에 매칭되는 파일을 Read / Edit / Write 등으로 *건드릴 때만* messages 에 attach 돼요. 'paths' 가 없는 rule 은 CLAUDE.md 처럼 *항상 로드*.

<div class="callout-warm">
<strong>💡 왜 conditional matching 이 중요한가</strong>

큰 조직 / 모노레포 / 다팀 공유 코드베이스에서는 <span class="mod">.claude/rules/&#42;.md</span> 의 가치가 큽니다.

제가 일하는 환경을 예로 들면, *여러 팀이 함께 작업하는 거대한 monorepo* 에 수많은 파트가 붙어서 협업해요. 각 파트는 보통 *repo 안 자기 subfolder 영역에서만* 작업합니다:

- Frontend 파트 → <span class="mod">&lt;monorepo&gt;/frontend/web-portal/&#42;&#42;</span>
- Payments 파트 → <span class="mod">&lt;monorepo&gt;/backend/payments/&#42;&#42;</span>
- Infra 파트 → <span class="mod">&lt;monorepo&gt;/infra/terraform/&#42;&#42;</span>

각 파트가 자기 영역에 <span class="k">paths:</span> 로 묶인 <span class="mod">.claude/rules/CLAUDE.md</span> 를 두면 세 가지 이점이 자연스럽게 따라와요:

- 모든 rules 를 *항상* 로드하지 않아요. token 비용 누적과 무관 instruction 의 attention 분산 방지
- 작업 컨텍스트에 매칭되는 rule 만 *그때그때* attach → token-efficient + relevance ↑
- <strong>파트별 context isolation 이 자연스럽게 성립</strong>. Frontend 파트의 컨텍스트에 payments domain rule 이 끼어들지 않고, 그 반대도 마찬가지

<strong>🔗 실험 2 (visibility) 와의 연결</strong>

CLAUDE.md 의 walk 는 단방향이라 *cwd 의 ancestor chain* 만 봅니다. 그래서 파트가 자기 subfolder 에서 claude-code 를 실행하면 *상위 공통 CLAUDE.md* 는 자동으로 보이지만, *옆 파트의 subfolder CLAUDE.md* 는 보이지 않아요.

반대로 *어디서 실행하든 자동으로 attach* 가 필요한 cross-cutting rule 은 <span class="mod">.claude/rules/&#42;.md</span> 의 <span class="k">paths:</span> glob 으로 처리됩니다. <strong>walk 가 못 하는 일을 paths-glob 매칭이 메우는 셈</strong>.

두 메커니즘이 서로 다른 곡선을 그려 *파트 context* 와 *cross-cutting rule* 을 깔끔하게 나눠 가져가요.
</div>

### <span class="k">claudeMdExcludes</span> 로 자동 로딩 막기

*특정 CLAUDE.md, .claude/rules, CLAUDE.local.md 파일을 자동 로딩에서 빼고 싶을 때* 는 settings.json 의 <span class="k">claudeMdExcludes</span> 글로벌 패턴 배열을 씁니다.

```json
{
  "claudeMdExcludes": [
    "/some/repo/CLAUDE.md",
    "**/vendored/**/CLAUDE.md"
  ]
}
```

매칭되는 경로는 hierarchical search 결과에서 *드롭* 되어 <span class="k">#&nbsp;claudeMd</span> 안에 들어가지 않아요. Trace 에서는 *정상 빌드에서 보이던 <span class="k">Contents&nbsp;of</span> 헤더 한 줄이 사라진* 모양으로 보입니다.

### <span class="mod">@</span> 로 다른 파일 참조하기

<div class="note-box">
<span class="note-flag">💡 TIP</span> <span class="mod">@</span> 로 다른 파일을 참조하는 건 Claude Code 전반에서 쓰이는 기법이에요. user prompt 에서 <span class="mod">@README.md</span> 로 파일을 첨부하는 그것과 같은 결. CLAUDE.md 에서도 같은 형태로 등장합니다.
</div>

CLAUDE.md 본문 안에 <span class="mod">@./team-rules.md</span> 같은 한 줄을 적으면 그 파일이 함께 합쳐져 들어가요. CLAUDE.md 가 너무 커지면 일부 규칙을 별도 파일로 떼어 두고 거기서 끌어쓰기 좋아요. 지원되는 경로 형태는 원본 CLAUDE.md 파일 기준 상대 (<span class="mod">@./relative.md</span>), home 기준 (<span class="mod">@~/.claude/shared.md</span>), 그리고 절대 경로 (<span class="mod">@/absolute/path.md</span>).

예시. CLAUDE.md 끝에 한 줄 적어 둔다고 해봅시다:

```markdown
## 추가 컨텍스트

@./included.md
```

여기서 included.md 는 임의의 markdown 파일명 (예: *team-rules.md*, *shared-style.md*) 의 placeholder 예요.

directive 를 만났을 때 included.md 의 내용이 그 자리에 *치환되어* 들어가는 건 아니에요. directive 는 본문에 그대로 남고, included.md 는 별도의 <span class="k">Contents&nbsp;of&nbsp;...</span> 엔트리로 *추가* 됩니다. 즉 *additive*, not *transformational*.

trace 에 들어가는 모양:

```
Contents of <repo>/CLAUDE.md (project instructions, ...):
...
## 추가 컨텍스트

@./included.md           ← directive 그대로 살아 있음

Contents of <repo>/included.md (project instructions, ...):
<included.md 의 본문>     ← 바로 다음 엔트리로 추가됨
```

몇 가지 더 짚을 만한 디테일:

- **삽입 위치**: included 파일은 원본 *바로 다음 자리* 에 들어가요. 원본 파일들을 다 모은 뒤 한꺼번에 모아두지 않습니다.
- **타입 상속**: included.md 는 원본의 type 을 그대로 물려받아요. 예를 들어 원본이 Project 면 included.md 도 Project 로 분류되고, 자체 description suffix 를 새로 받지는 않습니다.
- **재귀 깊이**: chain 최대 5 단계. cycle 은 <span class="k">processedPaths</span> 가 이미 본 경로를 추적해 방지.

<div class="sb-tag">What NOT to save + How to save</div>

````text
## What NOT to save in memory

- Code patterns, conventions, architecture, file paths, or project structure — these can be derived by reading the current project state.
- Git history, recent changes, or who-changed-what — `git log` / `git blame` are authoritative.
- Debugging solutions or fix recipes — the fix is in the code; the commit message has the context.
- Anything already documented in CLAUDE.md files.
- Ephemeral task details: in-progress work, temporary state, current conversation context.

These exclusions apply even when the user explicitly asks to save. If they ask you to save a PR list or activity summary, ask what was *surprising* or *non-obvious* about it — that is the part worth keeping.

## How to save memories

Saving a memory is a two-step process:

**Step 1** — write the memory to its own file (e.g., `user_role.md`, `feedback_testing.md`) using this frontmatter format:

\`\`\`markdown
---
name: {{memory name}}
description: {{one-line description — used to decide relevance in future conversations, so be specific}}
type: {{user, feedback, project, reference}}
---

{{memory content — for feedback/project types, structure as: rule/fact, then **Why:** and **How to apply:** lines}}
\`\`\`

**Step 2** — add a pointer to that file in `MEMORY.md`. `MEMORY.md` is an index, not a memory — each entry should be one line, under ~150 characters: `- [Title](file.md) — one-line hook`. It has no frontmatter. Never write memory content directly into `MEMORY.md`.

- `MEMORY.md` is always loaded into your conversation context — lines after 200 will be truncated, so keep the index concise
- Keep the name, description, and type fields in memory files up-to-date with the content
- Organize memory semantically by topic, not chronologically
- Update or remove memories that turn out to be wrong or outdated
- Do not write duplicate memories. First check if there is an existing memory you can update before writing a new one.
````

<div class="sb-tag">When to access + Before recommending + Memory vs persistence</div>

````text
## When to access memories
- When memories seem relevant, or the user references prior-conversation work.
- You MUST access memory when the user explicitly asks you to check, recall, or remember.
- If the user says to *ignore* or *not use* memory: proceed as if MEMORY.md were empty. Do not apply remembered facts, cite, compare against, or mention memory content.
- Memory records can become stale over time. Use memory as context for what was true at a given point in time. Before answering the user or building assumptions based solely on information in memory records, verify that the memory is still correct and up-to-date by reading the current state of the files or resources. If a recalled memory conflicts with current information, trust what you observe now — and update or remove the stale memory rather than acting on it.

## Before recommending from memory

A memory that names a specific function, file, or flag is a claim that it existed *when the memory was written*. It may have been renamed, removed, or never merged. Before recommending it:

- If the memory names a file path: check the file exists.
- If the memory names a function or flag: grep for it.
- If the user is about to act on your recommendation (not just asking about history), verify first.

"The memory says X exists" is not the same as "X exists now."

A memory that summarizes repo state (activity logs, architecture snapshots) is frozen in time. If the user asks about *recent* or *current* state, prefer `git log` or reading the code over recalling the snapshot.

## Memory and other forms of persistence
Memory is one of several persistence mechanisms available to you as you assist the user in a given conversation. The distinction is often that memory can be recalled in future conversations and should not be used for persisting information that is only useful within the scope of the current conversation.
- When to use or update a plan instead of memory: If you are about to start a non-trivial implementation task and would like to reach alignment with the user on your approach you should use a Plan rather than saving this information to memory.
- When to use or update tasks instead of memory: When you need to break your work in current conversation into discrete steps or keep track of your progress use tasks instead of saving to memory. Tasks are great for persisting information about the work that needs to be done in the current conversation, but memory should be reserved for information that will be useful in future conversations.
````

- **<span class="k">originSessionId</span>** — 어느 세션에서 이 메모리가 생성됐는지 기록. *trace ↔ memory 노트* 매핑이 가능해 디버깅에 의외로 쓸모 있어요.

<span class="note-flag">📝 NOTE</span> *"auto memory"* 의 *auto* 는 *모델이 알아서 저장한다* 는 의미라기보다 *Write / Edit 같은 built-in tool 호출이 정책에 따라 자동화돼 있다* 는 의미예요. 발동 trigger 는 (a) 모델의 자율 판단, 또는 (b) user 의 명시적 지시 양쪽 다 가능.
</div>


## 시리즈를 관통하는 한 가지. *Harness 가 다루는 정보* vs *LLM 이 다루는 정보*

[*2편 Claude Code 의 Context Architecture*](../claude-code-context-architecture/) 에서 복선으로 남겨뒀던 질문이 있어요.

<blockquote>
<p><em>변할 수 있는 (dynamic) 컨텐츠라도 어떤 건 <strong>System Block</strong> 에, 어떤 건 <strong>Messages Block</strong> 에 들어가요. 분기가 생기는 이유는?</em></p>
</blockquote>

여기까지 시리즈를 따라온 사람이라면 답이 어렴풋이 보일 거예요. 한 줄로:

<blockquote>
<p><strong>System Block dynamic 에는 <em>LLM 이 <span class="key">Harness Engineering</span> 측면의 결정에 쓸 정보</em> 가, Messages Block reminder 에는 <em>LLM 이 <span class="key">Context Engineering</span> 측면의 결정에 쓸 정보</em> 가 자리잡아요.</strong></p>
</blockquote>

<strong>LLM 이 brain 이고, harness 는 brain 의 명령을 받아 실행하는 손발</strong> 이에요. tool 을 호출할지, memory 를 저장할지, 다음 단계를 어디로 갈지 다 LLM 의 결정이고, harness 는 그 결정을 *실행* 합니다. *LLM 이 결정하기 위해 필요한 정보* 라는 건 같지만, **agent 운영에 관한 결정** 이면 *Harness Engineering* 측면, **응답 생성에 관한 결정** 이면 *Context Engineering* 측면입니다.

### System Block dynamic = harness 가 셋업한 *운영 환경*

[*3편 Claude Code 의 System Block*](../system-prompt-assembly/) 의 dynamic slot 들 (8-21) 을 다시 보면:

- *slot 8 session_guidance*: 이 세션에 켜진 도구 안내
- *slot 9 memory procedure*: 메모리 관리의 taxonomy + when-to-save
- *slot 11 env_info*: cwd, platform, shell, model
- *slot 12 language / 13 output_style*: 응답 스타일 config
- *slot 14 mcp_instructions*: 연결된 MCP 서버 instructions
- *slot 21 gitStatus*: 작업 디렉토리 snapshot

대체로 *"harness 가 이 세션을 이렇게 셋업했어. LLM 너는 이 환경 안에서 동작해."* 라는 셋업 메시지예요. <span class="key">LLM 이 agent 운영을 결정할 때 쓰는 정보</span> 들.

### Messages Block reminder = LLM 의 *reasoning 재료*

반면 매 turn 합성되는 Messages Block 의 <span class="k">&lt;system-reminder&gt;</span> 슬롯에는:

- CLAUDE.md content: 사용자가 적은 instructions · context
- MEMORY.md content (memory note index): retrieval 의 출발점
- 오늘 날짜: 시간 reasoning 의 fact
- hook output, @-file content: LLM 이 직접 다룰 컨텐츠
- skill 정의: LLM 이 invoke 판단할 도구 카탈로그

대체로 *"LLM 너가 직접 읽고 reasoning 에 쓸 재료야"* 라는 컨텐츠. <span class="key">LLM 이 응답 생성을 결정할 때 쓰는 정보</span> 들.

### MEMORY.md 의 dual injection 이 *결정적 confirmation*

이 분기 기준을 가장 깔끔하게 보여주는 사례가 이 글의 주제인 <span class="key">MEMORY.md</span> 입니다. 같은 source 안의 두 종류 정보가 *책임 경계* 에 따라 자동으로 두 자리로 갈라져요.

- *procedure* (어떻게 다룰지) → <strong>harness 의 메모리 운영 정책</strong> → System Block
- *content* (memory note index) → <strong>LLM 이 retrieval 출발점으로 삼을 컨텐츠</strong> → Messages Block

한 source 의 두 종류 정보가 두 자리로 갈라지는 게 어색해 보였지만, 분기 기준이 *책임 경계* 라고 보면 **갈라지지 않을 수 없는** 구조였어요.

### 1편으로 되돌아오기. *Context Engineering ⊂ Harness Engineering*

이 분기가 [*1편 Harness Engineering과 Context Engineering*](../harness-vs-context-engineering/) 의 두 단어를 *구조 차원* 에서 다시 만나게 합니다.

- <strong>Harness Engineering</strong>: 이 세션을 어떻게 셋업할지 (settings · env · MCP · output style · memory 정책 등의 결정)
- <strong>Context Engineering</strong>: 그 셋업이 *매 turn LLM 에 어떤 reasoning 재료를 건네줄지* 큐레이션

Context Engineering 은 Harness Engineering 이 *결정한 채널 (Messages Block)* 에 *무엇이 흐를지* 를 다루는 <strong>하위 활동</strong> 이에요. harness 의 셋업이 바뀌면 ([*4편 Settings 와 MCP*](../settings-and-mcp/) 의 <span class="k">customSystemPrompt</span> 로 slot 1-20 통째로 갈아끼우는 예시처럼) Context Engineering 의 무대 자체가 달라져요.

이 분기 구조는 <span class="key">Context Engineering ⊂ Harness Engineering</span> 이라는 *주류 시각* (1편의 Anthropic / Osmani framing) 을 구조적으로 서포팅합니다. 반대 framing (*Harness ⊂ Context*) 도 통용되지만, 이번 글의 dual-channel 분기는 전자를 받쳐주는 증거예요. 시리즈의 thesis 가 여기서 한 자리로 모입니다.


## Take-home

이 글의 정리:

<blockquote>
<p>Claude Code 의 long-term memory 는 여러 source 에서 흘러 들어와요. <strong>모두 Messages Block 으로 합쳐져</strong> 모델에 도달하고, 어느 source 에서 왔는지는 <strong>description suffix 한 줄</strong> 로 구분됩니다.</p>
<p><span class="key">예외는 MEMORY.md 하나</span>. <em>content</em> 와 <em>policy</em> 가 분리되어 <strong>System Block 과 Messages Block 에 동시에</strong> 들어가요. content (memory note index) 는 다른 source 들과 같은 Messages Block 쪽, policy (어떻게 다뤄야 하는지) 는 cache 잘 되는 System Block 쪽. <strong>MEMORY.md 자체는 <em>index 역할</em> 만</strong> 하고, 실제 memory 노트는 모델이 필요할 때 <strong>retrieval</strong> 로 가져갑니다.</p>
<p>그리고 더 큰 그림. *두 영역 다 결국 LLM 이 받아서 결정에 쓰는 정보* 인데, <strong>System Block dynamic</strong> 자리에 들어가는 건 <em>agent 운영에 관한 결정</em> (Harness Engineering 측면), <strong>Messages Block reminder</strong> 자리에 들어가는 건 <em>응답 생성에 관한 결정</em> (Context Engineering 측면) 에 쓰이는 정보입니다. MEMORY.md 가 두 자리에 동시에 들어간 건 우연이 아니라 *그 분기의 가장 깔끔한 사례* 였어요. <span class="key">Context Engineering 은 Harness Engineering 의 부분집합</span> 이라는 시리즈 thesis 가 이 자리에서 모입니다.</p>
</blockquote>
