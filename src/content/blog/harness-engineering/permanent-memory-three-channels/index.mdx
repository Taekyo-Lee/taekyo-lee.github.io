## Long-term Memory 전반

### 같은 채널, 여러 memory source

[*Context Architecture*](../claude-code-context-architecture/) 의 마지막에 가장 의외라고 한 부분, 기억 나시나요?

<blockquote>
<span class="key">CLAUDE.md · MEMORY.md · 오늘 날짜</span> 처럼 harness 가 끌어오는 컨텍스트는 System Block 이 아니라, <span class="k">&lt;system-reminder&gt;</span> slot 안에 들어갑니다.
</blockquote>

[*System Block*](../system-prompt-assembly/) 부터 [*gitStatus*](../git-context/) 까지 다 봤으니, 이번 편이 바로 *Messages Block* 으로 자리를 옮기는 첫 글이에요. 결론부터 말하면. **여러 종류의 메모리 파일이 Messages Block 의 <span class="k">&lt;system-reminder&gt;</span> slot 으로 흘러 들어와요**. 이름은 "claudeMd" 인데, 실제 콘텐츠는 CLAUDE.md 만이 아닙니다.

<div class="note-box">
<span class="note-flag">📝 NOTE</span> 이 글에서 "다섯 memory source" 라고 정리하는 건 <strong>제가 살펴본 빌드 (v.2.1.88) 기준</strong>이에요. Claude Code 가 빠르게 업데이트되면서 새 채널이 추가될 수 있고, 제가 놓친 memory source 가 있을 가능성도 있어요. 숫자 자체보다 <em>한 채널 안에 여러 memory source 가 합쳐 들어온다</em> 는 구조에 무게를 두고 읽어 주세요.
</div>

이 memory source 들은 각각 누가, 어떤 의도로 쓰느냐가 다릅니다. <span class="term">Managed</span> 는 조직 정책 (관리자가 배포), <span class="term">User</span> 는 내 노트북 전체에 적용되는 내 설정, <span class="term">Project</span> 는 이 레포에 체크인된 팀 공유 규칙, <span class="term">Local</span> 은 이 레포에 한정된 내 비공개 메모, <span class="term">MEMORY.md</span> 은 모델이 직접 관리하는 자기 노트.

이 memory source 들이 같은 slot 안에 모이지만, 어떤 종류인지를 알려주는 단서가 한 줄씩 붙어요. 모델이 *"이 내용은 누가 쓴 거지?"* 를 구분할 수 있게 하는 장치입니다.

## CLAUDE.md

### 디렉토리를 거슬러 올라가는 walk. cwd → root

그러면 이 파일들이 어디에 있는지를 harness 는 어떻게 알아낼까요. <span class="k">getMemoryFiles()</span> 의 동작은 의외로 단순해요. **현재 작업 디렉토리에서 출발해 파일시스템 root 까지 한 단계씩 거슬러 올라가면서**, 매 단계마다 CLAUDE.md, .claude/CLAUDE.md, <span class="mod">.claude/rules/&#42;.md</span>, CLAUDE.local.md 가 있는지 살핍니다.

```
cwd:  /home/jet/repo/src/feature   ← 시작
       ↓
       /home/jet/repo/src        ← 여기에도 CLAUDE.md 있나?
       ↓
       /home/jet/repo            ← 여기는?
       ↓
       /home/jet                 ← 여기도?
       ↓
       /                         ← 끝
```

이렇게 위로 다 모은 뒤 순서를 *root → cwd* 로 뒤집어요. **cwd 에 가까운 파일이 뒤쪽에 자리잡는 거죠**. 모델은 prompt 의 끝부분에 더 집중하는 경향이 있어서, 가까운 파일일수록 자연스럽게 더 높은 우선순위가 실립니다.

<div class="note-box">
<span class="note-flag">📝 NOTE</span> <strong>walk 는 git boundary 를 모릅니다.</strong> <code>/home/jet/repo</code> 에서 시작했다고 해서 <code>/home/jet</code> 에서 멈추지 않아요. 무조건 root 까지 갑니다. 단 <em>nested git worktree</em> (예: <span class="mod">claude -w</span> 로 띄운 worktree) 안에선 walk 는 그대로 root 까지 가지만, load 단계에서 worktree 바깥 main-repo 의 Project-type 파일은 skip — 같은 파일이 두 번 들어가는 걸 방지하는 edge case.
</div>

### <span class="k">#&nbsp;claudeMd</span> 가 비활성화되면 *비워지는 게 아니라 사라집니다*

환경변수 <span class="mod">CLAUDE_CODE_DISABLE_CLAUDE_MDS=1</span> 로 띄우면 Messages Block 의 <span class="k">&lt;system-reminder&gt;</span> 부분이 이렇게 바뀝니다:

```
<system-reminder>
As you answer the user's questions, you can use the following context:

Today's date is 2026-05-10.
```

### <span class="mod">@</span> 로 다른 파일 참조하기

<div class="note-box">
<span class="note-flag">💡 TIP</span> <span class="mod">@</span> 로 다른 파일을 참조하는 건 Claude Code 전반에서 쓰이는 기법이에요. user prompt 에서 <span class="mod">@README.md</span> 로 파일을 첨부하는 그것과 같은 결. CLAUDE.md 에서도 같은 형태로 등장합니다.
</div>

CLAUDE.md 본문 안에 <span class="mod">@./team-rules.md</span> 같은 한 줄을 적으면 그 파일이 함께 합쳐져 들어가요. CLAUDE.md 가 너무 커지면 일부 규칙을 별도 파일로 떼어 두고 거기서 끌어쓰기 좋아요. 지원되는 경로 형태는 원본 CLAUDE.md 파일 기준 상대 (<span class="mod">@./relative.md</span>), home 기준 (<span class="mod">@~/.claude/shared.md</span>), 그리고 절대 경로 (<span class="mod">@/absolute/path.md</span>).

예시. CLAUDE.md 끝에 한 줄 적어 둔다고 해봅시다:

```markdown
## 추가 컨텍스트

@./included.md
```

여기서 included.md 는 임의의 markdown 파일명 (예: *team-rules.md*, *shared-style.md*) 의 placeholder 예요.

**기대**: directive 를 만나면 included.md 의 내용이 그 자리에 *치환* 되어 들어간다.

**실제**: directive 도 그대로 *남고*, included.md 가 별도의 <span class="k">Contents&nbsp;of&nbsp;...</span> 엔트리로 *추가* 됩니다. 즉 *additive*, not *transformational*.

trace 에 들어가는 모양:

```
Contents of <repo>/CLAUDE.md (project instructions, ...):
...
## 추가 컨텍스트

@./included.md           ← directive 그대로 살아 있음

Contents of <repo>/included.md (project instructions, ...):
<included.md 의 본문>     ← 바로 다음 엔트리로 추가됨
```

몇 가지 더 짚을 만한 디테일:

- **삽입 위치**: included 파일은 원본 *바로 다음 자리* 에 들어가요. 원본 파일들을 다 모은 뒤 한꺼번에 모아두지 않습니다.
- **타입 상속**: included.md 는 원본의 type 을 그대로 물려받아요. 예를 들어 원본이 Project 면 included.md 도 Project 로 분류되고, 자체 description suffix 를 새로 받지는 않습니다.
- **재귀 깊이**: chain 최대 5 단계. cycle 은 <span class="k">processedPaths</span> 가 이미 본 경로를 추적해 방지.

## MEMORY.md

### 두 자리에 동시에 들어가요

CLAUDE.md 와는 달리 MEMORY.md 는 *System Block 과 Messages Block* 에 동시에 들어가요. CLAUDE.md 는 사람이 적어 둔 콘텐츠가 모델에 그대로 도달하면 끝이지만, MEMORY.md 는 모델이 harness 의 built-in tool (Write / Edit / Read) 을 직접 호출해 *쓰고 다루는* 노트라서, 콘텐츠 자체 + *어떻게 사용할지의 가이드* 가 짝으로 필요해요.

<div class="callout">
<strong>MEMORY.md 의 dual injection</strong>

| 무엇을 | 어디로 | 왜 거기로 |
|---|---|---|
| MEMORY.md 의 **본문** | <span class="k">&lt;system-reminder&gt;</span> 의 <span class="k">#&nbsp;claudeMd</span> 안 (나머지와 같은 채널) | 모델이 자기 노트의 *내용* 을 읽어야 함 |
| MEMORY.md 를 *어떻게 쓸지* 의 절차 (auto memory 사용 가이드. 어떤 종류를 언제 어떻게 저장할지) | System Block 의 <span class="k">#&nbsp;auto memory</span> 슬롯 | 절차는 모든 turn 동일하니 cache 가능한 System Block 쪽이 효율적 |
</div>

이 분리가 잘 된 이유는 *역할이 다르기 때문* 이에요. 메모리에 무엇을 적었는지 (data) 와 메모리를 어떻게 다뤄야 하는지 (policy) 는 변하는 빈도가 달라요. data 는 자주 바뀌고, policy 는 거의 안 바뀜. 그래서 policy 는 cache 가 잘 되는 System Block 쪽에 두고, data 는 매 turn 새로 prepend 되는 Messages Block 쪽에 둔 겁니다.

### MEMORY.md 와 memory 노트

조금 더 들어가면, Messages Block 으로 흘러 들어오는 *MEMORY.md* 는 사실 메모리 자체가 아니에요. 실제 메모리 내용은 같은 디렉토리의 **memory 노트** 들에 자리잡고, MEMORY.md 는 그 파일들로 가는 한 줄짜리 pointer 모음입니다. 다시 말해 *구체적 memory 들의 index 역할* 만 하는 셈이에요.

실제 메모리 디렉토리의 한 예:

```
~/.claude/projects/<slug>/memory/
├── MEMORY.md                              ← 자동 주입
├── user_setup.md                          ← 사용자 환경 셋업
├── feedback_act_dont_reconfirm.md         ← 재확인 자제, 바로 실행
├── feedback_terse_responses.md            ← 간결한 응답 선호
├── project_hooks_parity.md                ← Hook 동작 점검 노트
├── project_release_status.md              ← 릴리스 상태 추적
├── project_langfuse_input.md              ← Langfuse Input 패널 관찰
├── project_interactive_fix.md             ← Interactive 모드 수정 기록
├── project_interactive_hang.md            ← Interactive hang 디버깅
└── project_vscode_freeze_fix.md           ← VSCode freeze 해결법
```

각 memory 노트의 *타입* 은 파일명이 아니라 frontmatter 의 <span class="k">type:</span> 필드로 결정됩니다. 4 가지 (user / feedback / project / reference) 가 정의되어 있고, 각각 *언제 저장하고 어떻게 활용할지* 의 policy 가 다르게 붙어 있어요 (아래 verification 의 <span class="k">#&nbsp;auto&nbsp;memory</span> 블록 안 *Types of memory* 섹션에 타입별로 <span class="k">when_to_save</span> / <span class="k">how_to_use</span> / <span class="k">body_structure</span> / <span class="k">examples</span> 가 정의됨). 즉 type 은 단순 라벨이 아니라 *그 메모리에 적용되는 reasoning rule* 의 키.

예를 들어 위 dir tree 의 feedback_act_dont_reconfirm.md 의 frontmatter:

```yaml
---
name: Don't reconfirm an action the user has already asked for
description: When the user issues a clear directive ("do X", "wire it in", "go"), execute it; don't end with "want me to proceed?"
type: feedback
---
```

분류 기준은 <span class="mod">type: feedback</span> 한 줄이에요. 파일명은 사람이 보기 좋은 라벨일 뿐, 모델은 frontmatter 의 <span class="k">type</span> 만 보고 분류합니다.

trace 의 <span class="k">#&nbsp;claudeMd</span> 에 자동으로 들어가는 건 *MEMORY.md 한 파일뿐*. 나머지 memory 노트들은 모델이 필요할 때 built-in tool 인 Read / Grep 으로 직접 불러서 봐요. 매 turn 가벼운 MEMORY.md 한 줄로 *"이런 메모가 있다"* 만 알리고, 관련 있는 토픽만 retrieval 로 풀어가는 패턴이에요.

다시 말해 Claude Code 의 long-term memory 시스템은 **세 layer** 로 분리돼 있어요. *policy* (System Block, cache 가능), *MEMORY.md* (Messages Block, per-turn 자동 주입), *memory 노트* (on-demand retrieval). CLAUDE.md 가 *모든 내용을 매 turn 다 prepend* 하는 무거운 모델이라면, MEMORY.md 는 *policy 와 pointer 만 항상, 본 내용은 필요할 때만* 의 더 정제된 모델이에요.

### 실제 trace 에서 본 두 자리

실제로 한 turn 의 trace 를 캡처해 보면 *두 자리에 동시에 들어간다* 가 한눈에 보여요. 그리고 *MEMORY.md 만 자동, memory 노트 는 retrieval* 도 같이 검증됩니다.

<span class="key">System Block 의 동적 슬롯</span> (<span class="k">gen_ai.system_instructions</span> 패널에서 보임). memory 의 *어떻게 다뤄야 하는지* 가 여기 들어가요.

````text
## What NOT to save in memory

- Code patterns, conventions, architecture, file paths, or project structure — these can be derived by reading the current project state.
- Git history, recent changes, or who-changed-what — `git log` / `git blame` are authoritative.
- Debugging solutions or fix recipes — the fix is in the code; the commit message has the context.
- Anything already documented in CLAUDE.md files.
- Ephemeral task details: in-progress work, temporary state, current conversation context.

These exclusions apply even when the user explicitly asks to save. If they ask you to save a PR list or activity summary, ask what was *surprising* or *non-obvious* about it — that is the part worth keeping.

## How to save memories

Saving a memory is a two-step process:

**Step 1** — write the memory to its own file (e.g., `user_role.md`, `feedback_testing.md`) using this frontmatter format:

\`\`\`markdown
---
name: {{memory name}}
description: {{one-line description — used to decide relevance in future conversations, so be specific}}
type: {{user, feedback, project, reference}}
---

{{memory content — for feedback/project types, structure as: rule/fact, then **Why:** and **How to apply:** lines}}
\`\`\`

**Step 2** — add a pointer to that file in `MEMORY.md`. `MEMORY.md` is an index, not a memory — each entry should be one line, under ~150 characters: `- [Title](file.md) — one-line hook`. It has no frontmatter. Never write memory content directly into `MEMORY.md`.

- `MEMORY.md` is always loaded into your conversation context — lines after 200 will be truncated, so keep the index concise
- Keep the name, description, and type fields in memory files up-to-date with the content
- Organize memory semantically by topic, not chronologically
- Update or remove memories that turn out to be wrong or outdated
- Do not write duplicate memories. First check if there is an existing memory you can update before writing a new one.
````

<div class="sb-tag">When to access + Before recommending + Memory vs persistence</div>

````text
## When to access memories
- When memories seem relevant, or the user references prior-conversation work.
- You MUST access memory when the user explicitly asks you to check, recall, or remember.
- If the user says to *ignore* or *not use* memory: proceed as if MEMORY.md were empty. Do not apply remembered facts, cite, compare against, or mention memory content.
- Memory records can become stale over time. Use memory as context for what was true at a given point in time. Before answering the user or building assumptions based solely on information in memory records, verify that the memory is still correct and up-to-date by reading the current state of the files or resources. If a recalled memory conflicts with current information, trust what you observe now — and update or remove the stale memory rather than acting on it.

## Before recommending from memory

A memory that names a specific function, file, or flag is a claim that it existed *when the memory was written*. It may have been renamed, removed, or never merged. Before recommending it:

- If the memory names a file path: check the file exists.
- If the memory names a function or flag: grep for it.
- If the user is about to act on your recommendation (not just asking about history), verify first.

"The memory says X exists" is not the same as "X exists now."

A memory that summarizes repo state (activity logs, architecture snapshots) is frozen in time. If the user asks about *recent* or *current* state, prefer `git log` or reading the code over recalling the snapshot.

## Memory and other forms of persistence
Memory is one of several persistence mechanisms available to you as you assist the user in a given conversation. The distinction is often that memory can be recalled in future conversations and should not be used for persisting information that is only useful within the scope of the current conversation.
- When to use or update a plan instead of memory: If you are about to start a non-trivial implementation task and would like to reach alignment with the user on your approach you should use a Plan rather than saving this information to memory.
- When to use or update tasks instead of memory: When you need to break your work in current conversation into discrete steps or keep track of your progress use tasks instead of saving to memory. Tasks are great for persisting information about the work that needs to be done in the current conversation, but memory should be reserved for information that will be useful in future conversations.
````

```
Contents of /home/user/.claude/projects/-home-user-workspace-claude-context-engineering/memory/MEMORY.md (user's auto-memory, persists across conversations):

- [Sandbox purpose](sandbox_purpose.md) — 블로그 캡처와 context-engineering 동작 검증용 sandbox
```

명확히 표기하면:

- **MEMORY.md** = 위 dump 의 *한 파일* (한 줄짜리 pointer 목록, 즉 *구체적 memory 들의 index 역할*). 매 turn 자동으로 Messages Block 의 <span class="k">#&nbsp;claudeMd</span> 안에 들어옵니다.
- **memory 노트** = sandbox_purpose.md 같은 *별도 파일들* (sandbox 디렉토리에 함께 자리잡지만 별개 파일). **이 파일들의 본문은 trace 어디에도 등장하지 않습니다.** 모델이 필요할 때 built-in tool 인 Read / Grep 으로 직접 열어볼 뿐.


## Take-home

이 글의 정리:

<blockquote>
<p>Claude Code 의 long-term memory 는 여러 source 에서 흘러 들어와요. 모두 Messages Block 의 <span class="k">#&nbsp;claudeMd</span> 한 슬롯으로 합쳐져 모델에 도달하고, 어느 source 에서 왔는지는 description suffix 한 줄로 구분됩니다.</p>
<p>예외는 MEMORY.md 하나. <em>content</em> 와 <em>policy</em> 가 분리되어 System Block 과 Messages Block 에 동시에 들어가요. content (실제 메모) 는 다른 source 들과 같은 Messages Block 쪽, policy (어떻게 다뤄야 하는지) 는 cache 잘 되는 System Block 쪽. 그리고 MEMORY.md 자체는 *index 역할* 만 하고, 실제 memory 노트는 모델이 필요할 때 retrieval 로 가져갑니다.</p>
</blockquote>
