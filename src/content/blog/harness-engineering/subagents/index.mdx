## 같은 markdown, 정반대 행선지

7편의 Skills 와 이번 편의 Subagents 는 디스크 위에서 매우 비슷하게 생겼어요. frontmatter + body 의 markdown 파일.

```
.claude/skills/<name>/SKILL.md       ← 7편의 자산
.claude/agents/<name>.md             ← 이번 편의 자산
```

같은 cascade (managed → user → project → plugin) 로 발견. 같은 frontmatter parser. 둘 다 사용자가 명시적으로 호출하지 않으면 모델 context 에 *body* 가 들어오지 않음 (advertise 만).

그런데 *호출되는 순간* 행선지가 정반대로.

## agent 파일의 frontmatter 가 훨씬 길어요

이 차이의 첫 단서. agent 파일의 frontmatter 는 SKILL.md 보다 *훨씬* 길어요. *그 새 세션의 모든 것을 결정해야* 하니까.

```yaml
---
name: explore
description: Read-only search agent. Use to find files, grep symbols, etc.
tools: Read, Glob, Grep, Bash
disallowedTools: Edit, Write
model: haiku
maxTurns: 25
isolation: worktree
---

You are a focused, read-only search assistant. ...
```

여기 보이는 모든 키 (`tools`, `disallowedTools`, `model`, `maxTurns`, `isolation`) 가 *그 subagent 가 살 환경* 을 결정합니다. 부모와 다른 model, 다른 tool 셋, 다른 turn cap, 다른 cwd (worktree).

### 핵심 — subagent 의 system prompt 는 *stripped*

부모 세션의 system prompt 가 ~12K 토큰이라면, subagent 의 system prompt 는 *agent 파일의 body 만*. 보통 1K 미만이에요. 부모의 *intro*, *# Doing tasks*, *# Tone and style*, *# Session-specific guidance* 같은 sections 가 전혀 없어요. 3편에서 본 어셈블리가 *통째로 안 적용*.

이게 왜 의미있나. *각 subagent 가 자기만의 좁은 prompt 로 동작* 하도록 하는 isolation. *parent 의 무거운 sytem prompt 를 모든 subagent 가 다시 실어 나르지 않게* 하는 비용 절약. 그리고 *subagent 가 부모의 noise 와 격리* 되도록 하는 context hygiene.

### Explore / Plan 의 추가 stripping — 5편의 한 디테일 다시

5편에서 잠깐 본 디테일을 다시 짚어요. built-in `Explore` 와 `Plan` 은 부모의 *systemContext* 로부터 *gitStatus 빠진 채* 로 상속받아요.

```ts
// tools/AgentTool/runAgent.ts:401-408
const resolvedSystemContext =
  agentDefinition.agentType === 'Explore' ||
  agentDefinition.agentType === 'Plan'
    ? systemContextNoGit
    : baseSystemContext
```

코드 주석: *"Read-only search agent 는 parent-session-start gitStatus 가 dead weight. 필요하면 git status 직접 돌림. fleet-wide ~1-3 Gtok/week 절약."*

같은 식으로 `omitClaudeMd: true` 도 Explore/Plan 에 적용 — 부모의 claudeMd bundle 을 안 받아옴. *"34M+ Explore spawns × 5-15 Gtok 절약"*.

핵심: Subagent 가 *추상화 비용을 isolation 으로 보상* 하는 자산이지만, *너무 무거우면 의미가 깎임*. 그래서 read-only agent 들은 더 stripped 됨.

## 부모는 subagent 의 결과를 *text* 로 받아요

subagent 가 작업을 마치면 그 결과가 부모의 trace 에서 `tool_result` 로 돌아옵니다. 그 `tool_result` 안에 의외의 디테일이 다 들어 있어요:

```
[Tool result]
The file `notes.md` contains a mention of "verification beaver."
... agentId: aa20a239aca893577 (use SendMessage with to: 'aa20a239aca893577'
to continue this agent)
<usage>
total_tokens: 6127
tool_uses: 1
duration_ms: 5449
</usage>
```

세 가지가 같이 옵니다:
1. **subagent 의 final reply**. 그 자체가 *부모 LLM 의 다음 turn 의 input* 의 일부가 됨
2. **`agentId`**. 이 subagent 와 *다시 대화* 하고 싶으면 이 ID 로 SendMessage
3. **`<usage>`**. 이 subagent 가 얼마나 비쌌는지 (`total_tokens`, `duration_ms`)

마지막 게 흥미로워요. *부모 LLM 이 subagent 의 비용을 직접 본다*. 다음 결정에 *budget-aware* 하게 행동 가능. *"이 subagent 가 5초 + 6K 토큰 썼네, 다음엔 좀 다른 방식으로 가자"*.

<div class="callout-warm">
<strong>multi-agent 호출은 *prompt instruction* 으로 wired</strong>

Claude Code 의 multi-agent 흐름은 *tool schema 의 affordance* 가 아니라 *문장으로 모델에게 알려주는 방식* 으로 만들어져 있어요. <span class="mod">"use SendMessage with to: '...' to continue this agent"</span> 라는 한 줄이 *모델이 읽어야 할 instruction*. 이게 harness 전반의 디자인 철학 — rigid API 보다 *모델이 읽을 수 있는 prompt content* 로 capability 를 노출.
</div>

### Agent wrapper span — 한 분석 단위

부모 trace 에서 `Agent` tool call 이 끝나면, 단 하나의 wrapper span 이 남는데 그 안에 subagent 의 *모든 정보* 가 다 packed:

```
agentId: a66b9237c505653c6
agentType: general-purpose
content: [{type: 'text', text: '...'}]    ← subagent 의 reply
totalTokens: 6184
totalDurationMs: 3337
totalToolUseCount: 1
usage.input_tokens: 6112
usage.output_tokens: 72
```

부모의 분석 단위는 *Agent call 한 개 = wrapper span 한 개*. 안의 LLM 호출 / tool 실행이 몇 번이었는지는 그 안에 합산. cost monitoring 관점에서 깔끔.

## isolation 의 비용 — 두 cache prefix + 두 LLM 호출

같은 thin/fat 패턴이지만 7편의 Skills 보다 Subagents 가 *더 무거워요*. 호출 시 비용:

| | Skills (7편) | Subagents (이 편) |
|---|---|---|
| **호출 시 prefix cache** | 부모의 prefix 한 개 (그대로 쓰임) | **두 개** (부모의 prefix + subagent 의 새 prefix) |
| **호출 시 LLM 호출 횟수** | 한 번 더 (body 가 user message 가 된 입력) | **최소 두 번** (subagent 의 첫 LLM 호출 + 부모의 synthesis) |
| **격리 정도** | 같은 conversation 안 | 완전히 별개 LLM 세션 |
| **부모와의 정보 흐름** | 직접 inline | tool_use / tool_result 왕복 |
| **trace 모양** | 같은 turn 의 user message | 별개의 trace 트리 |

이 무거움을 *isolation* 으로 사는 거예요. *subagent 의 tool noise (Read 50K, Grep 결과 등) 가 부모의 prefix cache 를 오염시키지 않도록*. *부모의 무거운 system prompt 가 subagent 에게 거꾸로 짐이 되지 않도록*.

## Phase 2 의 한 디테일 — gpt-4o-mini 가 *Explore* 를 *general-purpose* 로 dispatch

7편에서도 잠깐 짚었지만, 이 글의 본 주제니 다시. Phase 2 trace 18-23 에서 user prompt 가 *"Use the Explore subagent to find ..."* 였는데, gpt-4o-mini 가 dispatch 한 건 `general-purpose` agent 였어요. 모델이 *"Explore"* 를 *동사 hint* 로 해석한 거예요. 명시적으로 *type 이름* 을 받아 쓰지 못함. Claude Opus 라면 정확히 `Explore` 로 갔을 가능성이 큽니다.

이게 trace 에서 의외로 자주 만나는 패턴이에요. *"왜 Explore 가 안 잡혔지"* → *"모델이 그냥 general 로 보냈구나"*. 같은 harness 라도 *모델 quality 가 instrumentation 의 정확도에 영향*.

또한 Explore 가 dispatch 안 됐으니 *gitStatus-strip / claudeMd-strip 의 positive case* 도 trace 에서 못 봤어요 (general-purpose 는 strip 안 함). Phase 2 의 *남은 검증 항목* 한 줄이에요.

## Subagent 의 mental model

| | Skills (7편) | Subagents (이 편) |
|---|---|---|
| destination | 같은 conversation 의 user message | 별개 세션의 system prompt |
| 격리 | 없음 (같은 prefix) | 강함 (별개 prefix + 별개 messages) |
| 비용 | body 한 번 user message | 두 prefix + 두 LLM 호출 |
| isolation 의 가치 | 없음 (Skills 는 isolation 목적이 아님) | 부모 conversation 보호 + subagent 좁은 prompt |
| 적합한 use case | 자주 쓰는 user 입력의 alias | 부모 conversation 을 오염시키지 않고 좁은 작업 isolation |

같은 markdown 이 *어떤 디렉토리에 있느냐* 만 다른데, 디자인의 *목적* 자체가 다르니 비용도 가치도 다른 두 abstraction. 5편의 thin/fat 분류로 보면 *같은 advertise/invoke 패턴* 이지만 invoke 비용의 *상한* 이 훨씬 큰 자산.

## Take-home

이 글의 정리:

<blockquote>
Subagents 는 7편의 Skills 와 *같은 thin/fat 패턴* 이지만, *fat invoke* 의 destination 이 정반대. 같은 conversation 의 user message 가 아니라 <span class="key">별개 LLM 세션의 system prompt</span>.
<br/><br/>
새 세션의 system prompt 는 *agent 파일의 body 만*. 부모의 12K 짜리 system prompt 가 통째 안 들어가요. *isolation* 의 가치를 *두 prefix + 두 LLM 호출* 의 비용으로 사는 abstraction.
<br/><br/>
부모는 subagent 의 결과를 *tool_result 안의 텍스트* 로 받아요. 그 안에 final reply 외에도 <span class="key">agentId, SendMessage hint, <span class="mod">&lt;usage&gt;</span> block</span> 까지 다 inline. 부모 LLM 이 *subagent 의 비용을 직접 보고 budget-aware 결정* 가능.
<br/><br/>
Explore / Plan 같은 read-only agent 는 *부모의 claudeMd 와 gitStatus 도 stripped*. <em>"34M+ spawns × 5-15 Gtok 절약"</em> 의 결정. fleet-scale 에서 의미 있는 비용 차이.
</blockquote>

다음 글은 *messages 영역* 의 또 다른 자리 — *시간 축* 으로 가는 메커니즘들. *transcript replay* (resume) 와 *session memory* (background 보강). 한 session 의 정보가 어떻게 보존되는가의 두 메커니즘.
