## 같은 Command 객체, 다른 advertise 방식

코드 안쪽에서 보면 Slash command 와 Skill 은 *같은 `Command` 객체* 예요. 어느 디렉토리에 있느냐만 달라요.

| 디렉토리 | `loadedFrom` | 모델에 advertise? |
|---|---|---|
| `~/.claude/commands/` (user) | `commands_DEPRECATED` | description 있을 때만 |
| `<repo>/.claude/commands/` (project) | `commands_DEPRECATED` | description 있을 때만 |
| `~/.claude/skills/<name>/SKILL.md` | `skills` | 항상 |
| `<repo>/.claude/skills/<name>/SKILL.md` | `skills` | 항상 |
| Plugin-bundled | `plugin` | 항상 |
| Built into the binary | `bundled` | 항상 |

같은 frontmatter parser, 같은 body 처리 (argument substitution + shell execution), 같은 invocation flow. 모든 게 공유.

그런데 *모델에 도달하는 모양* 이 두 단계로 분리돼요. Stage 1 *advertise*, Stage 2 *invoke*. 시리즈 전반의 디자인 모티프 — *thin advertise, fat invoke*.

이 한 분리가 *수십 개의 skill 을 가지고 있어도 평소 비용은 8000 chars 에 묶임* 이라는 효율을 만들어요. 호출하는 한두 개만 그 turn 에 fat 비용. 호출 안 하면 0.

## Stage 1 — `Skill` 도구 + `skill_listing` attachment

매 turn 보내는 `tools` 배열에 항상 `Skill` 이라는 도구 하나가 정의되어 있어요. 그 도구의 description 안에 *"사용 가능한 skill 들은 system-reminder 메시지에 나열됩니다"* 한 줄. 실제 list 는 거기 안 있고, **별도 attachment** 로 매 turn 0 마다 user message 안에 흘러 들어와요.

trace 의 첫 user-message 안에 이런 블록이 잡힙니다:

```
<system-reminder>
The following skills are available for use with the Skill tool:

- pdf: Use this skill whenever the user wants to do anything with PDF files…
- batch: Run multiple operations in parallel…
- /commit: Create a git commit with a generated message
…
</system-reminder>
```

<div class="callout-accent">
<strong>Stage 1 의 핵심은 *얇음*</strong>

각 entry 가 한 줄 description (250 chars 제한). 수십 개 skill 이 있어도 매 turn 8000 chars 안에 다 들어가야 함 (<span class="mod">SLASH_COMMAND_TOOL_CHAR_BUDGET</span> 또는 context window 의 약 1%). 안 들어가면 <span class="mod">formatCommandsWithinBudget</span> 가 trim 또는 names-only fallback.
</div>

### turn 0 only — `skill_listing` 의 delta 의미

이 attachment 는 *turn 0 에만 풀로* 보내고, 이후엔 새 skill 이 mid-session 으로 들어왔을 때만 delta 로. trace 에선 turn 0 의 `attributes.system_reminders` 에 skill_listing 이 있고, turn 1 부터 그 attribute 자체가 사라져요 (omit, not empty).

이 *omit, not empty* 가 trace 패턴 한 줄:

| `system_reminders` 상태 | 의미 |
|---|---|
| 있고 + 텍스트 | 이 turn 에 새 reminder 가 들어감 |
| 키 자체 부재 | 이 turn 엔 새 reminder 없음 |

같은 패턴이 6편의 `# claudeMd` 검사에서도, 10편의 hook 발동 검사에서도 반복.

### Stage 1 의 몇 가지 필터링 규칙

`getSkillToolCommands` 가 *모델에 보일 set* 으로 필터링:

- `disableModelInvocation: true` 인 skill 은 advertise 에서 제외 — *user-only*
- `cmd.source === 'builtin'` 인 CLI 명령 (`/help`, `/clear` 등) 도 제외 — UI 명령
- `.claude/commands/<name>.md` 파일이 *description 없음* 이면 model 에 보일 listing 에서 빠짐. *user 만 직접 칠 수 있음*

이게 *legacy* `.claude/commands/` 와 *modern* `.claude/skills/` 의 차이 중 하나. skill 은 *frontmatter description* 이 일종의 schema. 없으면 model 에 안 보임.

## Stage 2 — Invoke 시 body 가 user message 로

`/commit "fix: ..."` 를 친 순간 일어나는 일을 단계별로:

```
1. parser: "/commit \"fix: ...\"" → name="commit", args="\"fix: ...\""
2. resolve: <repo>/.claude/commands/commit.md 발견
3. body 읽기 + frontmatter 분리
4. argument substitution: $0, $1, ... $ARGUMENTS
5. shell execution: ```! cmd ``` 또는 !`cmd`
6. metadata 헤더 prepend (<command-name>, <command-message>, ...)
7. user message 로 wrap → 다음 turn 의 입력
```

호출 안 하면 *0 토큰*. 호출하면 *body 한 번* user message 비용으로 끝남. 시리즈 통틀어 가장 깔끔한 lazy load.

### shell execution 의 두 패턴 — 의외의 함정

body 안에 shell 명령을 박아두고 실행 결과를 prompt 안에 넣을 수 있어요. 두 패턴이 있습니다:

```markdown
# Block form
Diff:
` ` ` !
git diff --cached
` ` `

# Inline form
Diff: !`git diff --cached`
```

(*위 backtick 들은 실제로 인접해 있어요. 가독성 위해 띄움.*)

이 두 패턴만 인식돼요. *line prefix `!cmd`* 는 안 됨.

<div class="surprise">
<span class="label">⚡ SURPRISE</span> Phase 2 trace 15 에서 확인한 의외의 사실. <span class="key">line prefix `!git status`</span> 는 shell 실행이 <strong>안</strong> 됩니다. 그냥 literal text 로 LLM 에 도달해요. bash/sh 의 history-expansion 을 떠올리면 자연스럽지만, 이 harness 의 patterns 는 <span class="mod">``` !</span> (block) 또는 <span class="mod">!`</span> (inline) 둘만 매칭함. 한 번 헷갈리면 *"왜 내 git diff 가 안 나오지"* 디버깅에 시간이 갑니다.
</div>

### metadata 헤더 — 모델에게 *"이건 슬래시였어"* 알려주기

호출된 turn 의 user message 첫 부분에 이런 태그가 붙어요:

```
<command-message>commit</command-message>
<command-name>/commit</command-name>
<command-args></command-args>

<...body 본문...>
```

이게 왜 필요한가. **모델이 다시 Skill tool 을 호출하는 걸 막기 위해서** 입니다. 이 태그가 없으면 모델이 *"어, /commit 이라는 게 있네, Skill tool 로 호출해 볼까"* 하면서 무한 루프에 빠질 수 있어요. 태그를 보면 *"이미 expand 된 거구나, 그냥 본문 따라가자"* 로 인식.

가이드 문서 초안의 가설은 *"Slash command /foo loaded"* 같은 free-form 텍스트가 별도 user message 로 prepend 라고 했어요. 실제로는 위처럼 *XML 태그* + *같은 user message 안에 prepend*. Phase 2 trace 15 의 발견.

## 같은 패턴이 시리즈 곳곳에서

이 *thin advertise / fat invoke* 가 Skills + Slash 만의 것이 아니에요. 시리즈 전반에서 세 군데에서 같은 골격이 반복:

| 자산 | thin advertise | fat invoke |
|---|---|---|
| **Skills + Slash** (이 글) | skill_listing attachment (한 줄씩) | body 가 user message 로 inline |
| **MCP tools** (4편) | ToolSearch defer mode 의 deferred_tools_delta | ToolSearch 호출 후 schema inline |
| **Subagents** (8편) | agent listing (한 줄씩) | agent body 가 *별개 세션* 의 system prompt 로 |

같은 lazy load 원칙. *advertise 는 cheap, invoke 는 expensive*. 어떤 자산을 *항상 박아둘 가치 가 없는 것* 일 때 이 패턴이 등장.

## 4편의 ToolSearch 와 비교

4편에서 잠깐 본 ToolSearch defer mode 와 정확히 같은 골격이에요.

| | Skills (이 글) | MCP tools (4편) |
|---|---|---|
| advertise 방식 | `<system-reminder>` skill_listing | `<system-reminder>` deferred_tools_delta |
| advertise 위치 | turn 0 의 user message | 매 turn 의 user message attachment |
| invoke trigger | `/<name>` 또는 Skill tool 호출 | ToolSearch 도구 호출 |
| 호출 시 도달 | body 가 user message 로 | tool schema 가 그 turn inline 으로 |

같은 *meta-tool + delta-attachment* 패턴. *Skill* 이 *ToolSearch* 보다 한 단계 위 추상화 — Skill 은 *prompt body* 를 들고, ToolSearch 는 *tool schema* 를 들고 옴.

## Phase 2 의 한 디테일 — gpt-4o-mini 의 instruction-following

기록한 사실 한 가지. Phase 2 trace 18-23 에서 user prompt 가 *"Use the Explore subagent to find ..."* 였는데, 모델 (gpt-4o-mini) 이 dispatch 한 건 `general-purpose` agent 였어요. *"Explore"* 를 *동사 hint* 로 해석. 명시적인 *type 이름* 을 받아 쓰지 못함.

이게 왜 의미 있는가. 같은 harness 라도 *모델 quality 가 instrumentation 의 정확도에 영향*. trace 디버깅에서 *"왜 Explore 가 안 잡혔지"* → *"모델이 그냥 general 로 보냈구나"* 의 패턴이 의외로 자주.

이 디테일은 8편 (Subagents) 에서 자세히 다루지만, *Skill 호출 시에도 같은 종류의 모델 quality 이슈* 가 있을 수 있어요. *가짜 trigger 단어* 만 잡고 실제 invoke 까지 안 가는 경우.

## Take-home

이 글의 정리:

<blockquote>
Skills 와 Slash Commands 는 *같은 Command 객체*, 어느 디렉토리에 있느냐만 다름. 모델에 도달하는 모양은 *두 단계로 분리* 되어 — <span class="key">Stage 1 advertise (thin)</span> + <span class="key">Stage 2 invoke (fat)</span>.
<br/><br/>
Stage 1 의 자리는 *user message 의 <span class="k">&lt;system-reminder&gt;</span>* 안의 한 줄 카탈로그. turn 0 에만 풀로, 이후 delta. <span class="key">attribute 자체가 *없다* 가 *비었다* 보다 강한 신호</span>.
<br/><br/>
Stage 2 의 자리는 *호출된 그 turn 의 user message*. body 가 argument 치환 + shell 실행 후 inline. shell 패턴은 <span class="mod">``` !</span> block 또는 <span class="mod">!`cmd`</span> inline 둘만. line prefix <span class="mod">!cmd</span> 는 literal text 로 통과.
<br/><br/>
같은 *thin/fat* 패턴이 4편의 ToolSearch (MCP tools) 와 8편의 Subagents 에서 반복. <em>advertise 는 cheap, invoke 는 expensive</em> 가 시리즈 전반의 디자인 모티프.
</blockquote>

다음 글은 같은 *명시 호출* 자산이지만 격리 정도가 훨씬 큰 자산. <span class="term">Subagents</span>. 같은 markdown body 인데 이번엔 *부모의 messages* 가 아닌 *별개 LLM 세션의 system prompt* 가 됩니다.
