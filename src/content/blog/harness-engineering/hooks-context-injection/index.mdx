<div class="series-banner">
📚 시리즈 <strong>Context Engineering of Claude Code</strong> 의 열 번째 글입니다. 지금까지 본 모든 context source 는 *harness 가 어떤 파일을 어떻게 읽어 prompt 에 박을지* 를 다뤘다면, 이번 글은 그걸 <em>뒤집어요</em>. 사용자가 직접 prompt 안에 바이트를 박을 수 있는 단 하나의 메커니즘, hooks.
</div>

## "harness 가 결정한다" 의 뒤집기

지금까지 본 13개의 context source 는 모두 *harness 가 결정* 하는 것이었어요. *어떤 CLAUDE.md 를 읽을지*, *MEMORY.md 를 어떻게 truncate 할지*, *어떤 settings 키가 어디로 가는지*. 결정 권한은 Claude Code 의 코드 안에. 사용자는 그 룰 안에서 파일을 만들 뿐.

<span class="term">Hooks</span> 는 그걸 정확히 뒤집어요. 사용자가 작성한 *외부 shell command* (또는 SDK callback) 가 매 turn (또는 특정 event) 마다 실행되고, 그 stdout 또는 JSON return 이 *LLM 의 다음 user message 안에 바이트를 박습니다*.

<div class="callout-warm">
<strong>hook 이 유일한 mid-flight injection 통로</strong>

CLAUDE.md, MEMORY.md, gitStatus, output style, MCP instructions. 모두 *session 시작 시* (또는 매 turn 의 정해진 어셈블리 시점에) 위치합니다. 사용자가 mid-session 에 *동적으로* prompt 안에 뭔가를 추가하고 싶다면 hook 만이 답이에요.
</div>

## 두 injection mode. stdout vs additionalContext

같은 hook 이라도 inject 방식이 두 개. 모양은 다르지만 *결과는 똑같이* <span class="k">&lt;system-reminder&gt;</span> 블록으로 user message 안에 위치해요.

<div class="modes">
  <div class="mode-card">
    <span class="mode-tag">Mode 1</span>
    <div class="mode-name">stdout → hook_success</div>
    <div class="mode-desc">hook 이 stdout 에 텍스트를 출력하면 그게 통째로 prompt 에 위치합니다. 단 <span class="mod">SessionStart</span> 와 <span class="mod">UserPromptSubmit</span> 두 event 만 이 path 가 동작.</div>
  </div>
  <div class="mode-card">
    <span class="mode-tag">Mode 2</span>
    <div class="mode-name">JSON return → additionalContext</div>
    <div class="mode-desc">hook 이 stdout 의 마지막 줄에 특정 JSON shape 을 emit 하면 그 안의 <span class="mod">additionalContext</span> 필드가 prompt 에 들어감. 거의 모든 event 에서 동작.</div>
  </div>
</div>

### Mode 1. stdout

settings.json:

```json
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "matcher": "",
        "hooks": [
          { "type": "command", "command": "echo 'today is meeting day'" }
        ]
      }
    ]
  }
}
```

trace 의 <span class="k">system_reminders</span> attribute 안에 이런 모양으로 위치합니다:

```
<system-reminder>
UserPromptSubmit hook success: today is meeting day
</system-reminder>
```

위에 적은 대로 *<span class="k">SessionStart</span> 와 <span class="k">UserPromptSubmit</span> 둘만* 이 path 를 honor 해요. <span class="k">PreToolUse</span> 가 stdout 으로 *"about to run Bash"* 같은 걸 출력해도. 모델은 안 봅니다. logging 으로만 잡힘.

### Mode 2. JSON return

stdout 의 *마지막 줄* 에 이 shape 의 JSON 을 출력:

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "additionalContext": "extra context for the model"
  }
}
```

이 path 는 거의 모든 event 에서 동작해요. trace 의 모양:

```
<system-reminder>
PreToolUse:Bash hook additional context: extra context for the model
</system-reminder>
```

두 path 가 같은 hook 에서 *공존* 할 수 있어요. 자유 형식 stdout + 마지막 줄에 JSON → 두 블록이 동시에 들어감.

## hookName 의 자동 생성 패턴. <span class="k">:matcher</span> suffix

Phase 2 trace 36-40 에서 발견된 의외의 디테일.

settings 에서 hook name 을 명시하지 않으면 harness 가 자동 생성하는데, *tool-related event 의 경우 matcher 가 같이 들어갑니다*:

| event class | 자동 생성된 hookName | 예 |
|---|---|---|
| 비-tool event (UserPromptSubmit, SessionStart) | event 이름 그대로 | <span class="k">UserPromptSubmit</span> |
| tool-related (PreToolUse, PostToolUse) | <span class="mod">&lt;event&gt;:&lt;matcher&gt;</span> | <span class="k">PreToolUse:Bash</span>, <span class="k">PostToolUse:Edit&#124;Write</span> |

그래서 trace 의 <span class="k">&lt;system-reminder&gt;</span> 안에 들어가는 literal text 가 <span class="mod">PreToolUse hook additional context: ...</span> 가 아니라 <span class="mod">PreToolUse:Bash hook additional context: ...</span> 예요.

<div class="surprise">
<span class="label">⚡ SURPRISE</span> trace 를 grep 으로 *"hook 이 firing 했나"* 검사할 때 <span class="key">정규식에 <span class="mod">:&lt;matcher&gt;</span> suffix 를 허용</span> 해야 합니다. <span class="mod">^PreToolUse hook</span> 만 보면 tool-related hook 들을 다 놓쳐요. 처음 한 번 헷갈리면 *"왜 내 hook 이 fire 안 되는데 다 잘 돌아가는 거지"* 디버깅에 시간이 빠집니다.
</div>

## 어디에 들어가나. 실제 trace 의 모양

<span class="k">UserPromptSubmit</span> hook 이 fire 하면, 그 다음 LLM 호출 turn 의 user-message 안에 위치합니다. 그 자리는 prepend 된 합성 reminder 와 사용자의 literal prompt 사이.

OpenAI shim 환경에선 모두 한 user message 로 coalesced 되어 이런 모양:

```
[index 0. user message]
<system-reminder>...skill_listing...</system-reminder>

<system-reminder>...claudeMd + currentDate...</system-reminder>

<system-reminder>UserPromptSubmit hook success: today is meeting day</system-reminder>

(사용자가 친 literal prompt)
```

<span class="k">PreToolUse</span> / <span class="k">PostToolUse</span> 의 경우는 좀 더 흥미로워요. *tool 의 result 와 같은 turn* 의 user message 안에 위치해요:

```
[Tool call: Bash] {"command": "pwd"}
[Tool result] /home/jet/workspace
<system-reminder>PreToolUse:Bash hook additional context: PRE-TOOL-SENTINEL...</system-reminder>
<system-reminder>PostToolUse:Bash hook additional context: POST-TOOL-SENTINEL...</system-reminder>
```

여러 hook 이 같은 turn 에 fire 하면 모두 같은 user message 안에 <span class="mod">\n\n---\n\n</span> 로 join 되어 한꺼번에 들어감.

## hook 의 <span class="mod">isMeta: true</span>. transcript 에 안 적힘

[*세션의 기억*](../sessions-transcripts-and-memory/) 의 transcript 가이드와 connecting 되는 한 가지: hook 으로 inject 된 reminder 들도 *<span class="mod">isMeta: true</span>* flag 를 답니다. 결과:

- LLM 에는 user message 로 도달
- on-disk transcript 에는 *기록 안 됨* (meta filter)
- <span class="mod">--resume</span> 시 *replay 안 됨*

그러니 hook 으로 inject 한 정보는 **본질적으로 per-turn delta**. *"한 번 보여주고 끝"*. 영구 기억이 필요하면 hook 으로 inject 하지 말고 CLAUDE.md / MEMORY.md 에 적어야 해요.

<div class="callout-accent">
<strong>blocking PreToolUse. 도구 호출을 막을 수 있어요</strong>

PreToolUse hook 이 JSON return 으로 <span class="mod">&#123; "decision": "block", "reason": "..." &#125;</span> 를 emit 하면, 그 tool 호출 자체가 차단되고 reason 이 모델의 user message 안에 <span class="k">&lt;system-reminder&gt;</span> 로 위치해요. <em>"이 도구는 못 써, 이런 이유로"</em> 를 모델에게 mid-flight 으로 알리는 path. 정책 가드레일을 코드 변경 없이 wire 할 수 있는 강력한 메커니즘.
</div>

## hook 이 trace 에서 보이는 모양. 두 surface

### 1. hook span 자체

매 hook firing 은 <span class="mod">span.type: "hook"</span> 의 별도 span 으로 잡힙니다. attributes:

- <span class="k">hook_event</span>, <span class="k">hook_name</span> (auto-generated 또는 custom)
- <span class="k">num_hooks</span>, <span class="k">hook_definitions</span>
- <span class="k">duration_ms</span>, <span class="k">num_success</span>, <span class="k">num_blocking</span>, <span class="k">num_non_blocking_error</span>, <span class="k">num_cancelled</span>

이게 *"hook 이 돌았다"* 를 알려주는데, **inject 된 content 자체는 hook span 에 없어요**.

### 2. 다음 LLM-call span 의 <span class="k">system_reminders</span>

inject 된 *바이트* 는 그 hook 다음에 firing 하는 LLM-call span 의 <span class="k">attributes.system_reminders</span> 에서 봅니다.

| 패턴 | 어떤 hook 이 firing 했나 |
|---|---|
| <span class="mod">&lt;hookName&gt; hook success: &lt;stdout&gt;</span> | SessionStart 또는 UserPromptSubmit 의 stdout path |
| <span class="mod">&lt;hookName&gt; hook additional context: &lt;content&gt;</span> | 어떤 event 든 additionalContext path |
| <span class="mod">&lt;hookName&gt; hook blocking error from command: ...</span> | PreToolUse 의 blocking decision |

이 두 surface 의 분리 (*hook span 은 outcome, LLM span 은 content*) 가 trace 디버깅의 명료함을 만들어요. *"hook 이 돌았는데 model 이 못 봤다"* 면 둘 사이의 어떤 path 가 끊겼는지 단서가 또렷.

## hook 의 mental model

| | Hook | CLAUDE.md | MEMORY.md | Slash command |
|---|---|---|---|---|
| 결정 주체 | 사용자 (외부 process) | 사용자 (markdown) | 모델 (auto-write) | 사용자 (명시 호출) |
| 호출 시점 | event 자동 fire | 매 turn auto | 매 turn auto | 명시 호출 |
| 영속성 | per-turn (transcript 에 미기록) | persistent | persistent | per-turn |
| 가장 좋은 use case | 동적 per-turn 정보 (오늘의 oncall, 환경 변화) | 항상 적용되는 규칙 | 모델이 발견한 사실의 누적 | 자주 쓰는 user 입력의 alias |

hook 의 sweet spot: **per-turn 으로 동적, 모델이 한 번만 보면 되고, 빠른 shell command 또는 SDK callback 으로 source 되는 정보**. 영구도 아니고 user 입력의 alias 도 아닌 *그 사이의 영역*.
