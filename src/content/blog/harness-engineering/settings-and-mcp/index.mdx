## settings 키 대부분은 LLM 에 안 닿습니다

Claude Code 의 동작 옵션은 settings.json 이라는 JSON 파일로 들어갑니다. 모양은 이런 식이에요:

```json
{
  "model": "claude-sonnet-4-6",
  "language": "korean",
  "outputStyle": "Explanatory",
  "permissions": {
    "allow": ["Bash(npm test)", "Read"],
    "deny": ["Bash(rm:*)"]
  },
  "hooks": {
    "PostToolUse": [ /* ... */ ]
  },
  "autoMemoryEnabled": true,
  "claudeMdExcludes": ["**/vendored/**/CLAUDE.md"],
  "theme": "dark",
  "apiKeyHelper": "/usr/local/bin/get-key.sh",
  "telemetry": { /* ... */ }
  // ...수십 개 더
}
```

이런 키가 수십 개. 그런데 <span class="key">**settings.json 은 사실 한 자리에 있는 파일이 아니에요**</span>. 네 layer 에 흩어져 있고, 거기에 CLI flag 까지 합쳐져 한 settings 객체로 머지됩니다. 각 자리는 목적이 다릅니다.

- <span class="key">**managed**</span> (조직 admin 정책): 머신 단위 강제, user/project override 차단
- <span class="key">**user**</span> <span class="mod">~/.claude/settings.json</span>: 개인 기본값
- <span class="key">**project**</span> <span class="mod">&lt;repo&gt;/.claude/settings.json</span> (체크인): 팀 표준
- <span class="key">**local**</span> <span class="mod">&lt;repo&gt;/.claude/settings.local.json</span> (gitignored): 머신 특화 / 비밀
- <span class="key">**flag**</span> <span class="mod">--setting key=value</span> (CLI): 일회성 override

정확한 우선순위는 바로 아래 머지 표에서 봅니다.

이 모두를 합쳐도, *실제로 prompt 에 들어가 LLM 에 도달하는 키* 는 **손에 꼽힐 정도로 적어요**. 제가 살펴본 빌드 (cc_version 2.1.88) 기준 다섯 개. 나머지는 모두 <span class="key">*pure harness engineering*</span> 영역 (권한 게이트, theme, hook 실행, telemetry, 인증 등). [*Harness Engineering과 Context Engineering*](../harness-vs-context-engineering/) 에서 본 framing 으로 보면 context engineering 에 닿지 않는 자리들이에요.

이 글 전반에선 그 키들이 무엇이고 [*System Block*](../system-prompt-assembly/) 의 어떤 slot 에 어떻게 들어가는지 짚어봅니다.

### 여섯 layer 의 priority 와 머지 순서

키마다 도달처가 다르지만 머지 규칙은 공통이에요. 여섯 layer 가 priority 높은 순 (*숫자 작을수록 우위*) 으로:

<div class="cascade">
<span class="priority">1 (최고)</span>  <span class="layer">policySettings</span>   /etc/claude-code/managed-settings.json (admin 배포, override 차단)<br/>
<span class="priority">2</span>          <span class="layer">flagSettings</span>     CLI flag (--setting key=value)<br/>
<span class="priority">3</span>          <span class="layer">localSettings</span>    &lt;repo&gt;/.claude/settings.local.json (gitignored)<br/>
<span class="priority">4</span>          <span class="layer">projectSettings</span>  &lt;repo&gt;/.claude/settings.json (체크인)<br/>
<span class="priority">5</span>          <span class="layer">userSettings</span>     ~/.claude/settings.json<br/>
<span class="priority">6 (최저)</span>  <span class="layer">pluginSettings</span>   ~/.claude/plugins/&lt;plugin&gt;/settings.json (allowlist 키만)
</div>

같은 키가 여러 layer 에 있으면 priority 높은 (위쪽) 쪽이 이깁니다. <span class="key">policySettings 가 최고</span> 인 게 핵심이에요. admin 정책이 user/project override 를 무력화해서 enterprise 환경에서 보안 정책을 강제할 수 있어요.

<div class="callout-warm">

**직접 한 번 돌려보기**

위 priority 표가 맞는지 보려면 *같은 키* 를 *다른 layer* 에 *다른 값* 으로 넣고 trace 에서 어느 값이 살아남는지 보면 됩니다. language 키가 가장 깔끔해요. system prompt 의 <span class="k">#&nbsp;Language</span> block 에 setting 값이 그대로 들어가거든요.

setup:

- userSettings (<span class="mod">~/.claude/settings.json</span>, priority 5): <span class="mod">&#123; "language": "korean" &#125;</span>
- projectSettings (<span class="mod">&lt;repo&gt;/.claude/settings.json</span>, priority 4): <span class="mod">&#123; "language": "japanese" &#125;</span>

priority 가 맞다면 project (4) 가 user (5) 보다 우위라 *japanese* 가 들어가야 해요. 세션 시작 후 trace 의 system prompt 에서 <span class="k">#&nbsp;Language</span> block 발췌:

````text
# Language
Always respond in japanese. Use japanese for all explanations, comments, and communications with the user. Technical terms and code identifiers should remain in their original form.
````

<span class="key">**japanese 가 들어갔어요.**</span> projectSettings 가 userSettings 를 이긴다는 머지 동작이 trace 차원에서 확인됩니다.

</div>

<div class="callout-accent">

**예외 하나: autoMemoryDirectory**

위 priority 규칙을 안 따르는 키가 하나 있어요. autoMemoryDirectory 는 *projectSettings 에서만 의도적으로 무시* 됩니다.

이 키가 결정하는 건 *Claude Code 의 auto-memory 파일이 저장될 디렉토리 경로*. 즉 이 값을 set 할 수 있다는 건 *Claude Code 가 사용자 머신의 어디에 파일을 쓸지 결정* 할 수 있다는 뜻이에요. 만약 이게 projectSettings 에서도 정상 적용됐다면, 악성 repo 가 .claude/settings.json 에 <span class="mod">"autoMemoryDirectory": "~/.ssh"</span> 한 줄을 commit 해 두는 것만으로, 그 repo 를 clone 한 사용자가 Claude Code 를 켜면 *본인도 모르게 ~/.ssh 안에 memory 파일이 쓰이는* 사고가 가능해요. 이걸 막으려고 *projectSettings 라는 한 layer 만* 이 키를 무시하는 안전장치 (나머지 layer 는 정상 동작). 소스에서 memory dir 결정 함수에 *"intentionally excluded"* 주석으로 명시.

</div>

### LLM 에 *안* 닿는 (헷갈리기 쉬운) 키들

settings 안에서 *prompt 와 무관* 한 키들. 이름만 보면 LLM 에 영향을 줄 것 같지만 실제론 harness 내부 plumbing:

| Key | 왜 harness-only 인가 |
|---|---|
| model | API client 가 모델 선택에만 쓰는 값. prompt 본문엔 등장 안 함 |
| <span class="mod">permissions: &#123; allow / deny / ask &#125;</span> | tool 사용 권한 게이트. prompt 내용엔 영향 없음 |
| hooks | 어떤 hook 을 등록할지의 *선언* 일 뿐. hook 자체는 context 를 주입할 수 있지만 ([*Hooks*](../hooks-context-injection/)), 이 키 값은 prompt 에 안 들어감 |
| theme / vimEnabled / editorMode | TUI 렌더링·입력 모드 전용. LLM 호출과 무관 |
| mcpServers | 서버 *목록 자체* 는 prompt 에 안 들어감. 연결된 서버의 tool schema 와 instructions 는 별도 채널로 흐름 (글 후반 MCP 섹션 참조) |

## system block 통째로 우회하는 두 입구

지금까지 본 settings 키들은 *Claude Code 가 정의한 21 slot 안에서* 미세 조정하는 도구였어요. 그런데 settings.json 으로는 손댈 수 없는 두 옵션이 <span class="key">**21 slot 자체를 무력화하거나 확장**</span> 합니다. 두 입구로 노출돼요:

- **CLI flag**: <span class="mod">claude --system-prompt "..."</span>, <span class="mod">claude --append-system-prompt "..."</span> (또는 -file 변형). <span class="mod">claude -p</span> 같은 non-interactive batch 모드에서 자주 같이 씀
- **SDK / programmatic 호출**: <span class="mod">@anthropic-ai/claude-agent-sdk</span> 의 호출 옵션 customSystemPrompt, appendSystemPrompt

CLI flag 와 SDK 옵션은 *같은 메커니즘의 두 표면* 일 뿐이에요. 둘 다 결국 query engine 에 같은 두 값으로 전달됩니다. 아래는 *옵션 이름 (SDK 명명) 기준* 으로 본질을 봅니다.

### customSystemPrompt: 21 slot 본체 갈아끼우기

set 하면 **[*System Block*](../system-prompt-assembly/) 의 slot 1~20 (sub-block #3 + #4 의 본체) 이 전부 사라지고** 사용자가 넘긴 string 이 그 자리에 들어가요. *Claude Code 의 행동 규칙 + 환경 정보 + 메타 지시 + auto memory + language/outputStyle 같은 다른 settings 의 효과* 가 모두 빠짐.

단 다음 셋은 그대로 유지:

- **sub-block #1 (attribution header)**: 빌링 메타. 항상 prepend
- **sub-block #2 (정체성 한 줄)**: *"You are Claude Code, Anthropic's official CLI for Claude."* 같은 one-line 정체성 prefix. CLI sysprompt prefix 라는 별도 채널로 prepend
- **slot 21 (gitStatus trailer)**: appendSystemContext 가 별도 함수로 부착 (interactive 세션 기준)

즉 사용자 customization 은 *21 slot 의 본체만* 갈아끼우고, *Anthropic 의 정체성·빌링 메타와 환경 trailer* 는 carve-out 으로 보존돼요.

CLI 에서: <span class="mod">claude -p --system-prompt "당신은 보안 감사 전문 agent..."</span> 한 줄로 trigger 가능.

대표 use case: Claude Code 의 *tool runtime* (Read · Edit · Bash · Skill · Subagent 등) 만 빌려서 *완전히 다른 페르소나의 agent* 를 빌드. reviewer · evaluator · domain-specific assistant 처럼 base persona 가 아예 다른 specialized agent 들이 보통 이 패턴이에요.

### appendSystemPrompt: 끝에 한 덩어리 덧붙이기

default sections 가 다 깔린 *뒤에* 한 덩어리 append. dynamic 영역의 끝자리라 cache 영향이 적어요 (정적 prefix 의 cache hit 이 그대로 유지).

CLI 에서: <span class="mod">claude -p --append-system-prompt "이 세션의 git commit 에 [TICKET-123] prefix..."</span> 한 줄.

대표 use case: Claude Code 의 default 동작은 그대로 두고, *한 세션 또는 한 프로젝트 한정* 의 추가 정책만 한 줄 얹기. CI/CD 또는 automation 스크립트에서 자주 등장.

### 비교 와 trace 신호

| 옵션 | 동작 | cache 영향 | 주 사용처 |
|---|---|---|---|
| customSystemPrompt | slot 1~20 본체 대체 (sub-block #1, #2, slot 21 trailer 는 보존) | static prefix miss + dynamic 본체 새 빌드 | custom agent 빌더, automation 스크립트 |
| appendSystemPrompt | 21 slot 뒤에 한 덩어리 추가 | 정적 prefix cache hit 유지 | 세션 한정 정책 추가, CI/CD 자동화 |

trace 의 system prompt panel 에서 *slot 1~20 자리에 낯선 한 덩어리만* 있으면 customSystemPrompt. *기본 21 slot 다 있고 끝에 한 덩어리 더* 붙어 있으면 appendSystemPrompt.

<div class="surprise">
<span class="label">⚡ 직접 해봤어요</span> CLI 한 줄로 진짜 동작해요. <span class="mod">claude --system-prompt "당신은 ai agent입니다."</span> 실행 후 trace 의 system_instructions:

````text
x-anthropic-billing-header: cc_version=2.1.88; cc_entrypoint=cli;

You are Claude Code, Anthropic's official CLI for Claude.

당신은 ai agent입니다.

gitStatus: This is the git status at the start of the conversation. ...
Current branch: main
Recent commits:
60f5aea Add strutil package: library, CLI, tests
...
````

원래 자리에 있던 *Claude Code 의 행동 규칙 + 환경 정보 + 메타 지시* 가 모두 빠지고, 그 자리에 사용자 한 줄만 들어갔어요. 단 sub-block #1 (attribution header), sub-block #2 (정체성 한 줄), gitStatus trailer 는 보존.

즉 **Claude Code 의 system block 본체 (slot 1~20) 는 Anthropic 이 정해 둔 한 모양이 아니라, 호출 시점에 갈아끼울 수 있는 자리**. *"내가 쓰는 Claude Code = 결정된 모양"* 이라는 인상을 깨는 발견이에요. 🎯 Claude Code 의 runtime · tool 생태계 위에 *완전히 다른 agent* 를 얹는 모든 가능성의 출발점.
</div>

## outputStyle 의 네 가지 효과: 정적 부분까지 깨는 한 키

위 키들 중 *가장 의외* 인 건 <span class="key">outputStyle</span> 이에요. 일단 settings.json 에선 이렇게 한 줄로 적어요:

```json
{
  "outputStyle": "Explanatory"
}
```

값은 Anthropic 이 미리 만든 builtin 세 가지 (default · Explanatory · Learning) 중 하나, 또는 직접 만든 custom style 의 frontmatter name. 이 한 줄이 단순히 동적 슬롯 하나를 추가하는 게 아니라 *system prompt 의 네 자리* 를 동시에 수정해요. 그 네 자리를 차례로 봅니다.

### 결과: language 보다 outputStyle 이 훨씬 비싼 이유

[*System Block*](../system-prompt-assembly/) 에서 잠깐 본 비대칭이 여기서 본격적으로 풀려요.

LLM 에 보내는 system prompt 안에는 *매 호출마다 같은 정적 부분* 과 *매번 다를 수 있는 동적 부분* 이 섞여 있어요. 정적 부분은 *prompt cache* 가 한 번 잡아 두면 이어지는 호출들이 그 위에서 hit 돼서 응답이 빠르고 저렴해져요. 그런데 settings 값을 바꿀 때 그 *정적 부분까지 손이 가서 cache 를 무효화하는지* 여부가 키마다 다릅니다.

| 설정 변경 | 정적 부분 영향? | 동적 부분 영향? |
|---|---|---|
| <span class="mod">language: "korean"</span> | 아니요 | 예 (slot 12 추가) |
| <span class="mod">outputStyle: "Explanatory"</span> | **예** (효과 1, 2) | 예 (효과 3, 4) |

language 를 바꿔도 정적 부분은 그대로라 cache 가 살아남아요. 반면 outputStyle 을 바꾸면 정적 부분까지 깨져서, 다음 호출은 *전체를 처음부터 새로 빌드해야 합니다*.

<div class="surprise">
<span class="label">⚡ SURPRISE</span> outputStyle 은 시리즈에서 본 settings 키 중 *유일하게 정적 부분까지 깨는 키* 예요. language 처럼 동적 부분만 추가하는 다른 키들과 달리, outputStyle 만 효과 1, 2 로 정적 부분 (slot 1, 3) 자체를 수정합니다. 즉 어떤 outputStyle 로 시작하느냐에 따라 base persona 의 시작 모양이 달라지고, *cache 의 큰 부분을 처음부터 새로 빌드* 하게 돼요. <span class="key">cache 측면에서 가장 비싼 settings 키</span>가 outputStyle 인 이유.
</div>

### custom outputStyle 의 .md 파일: 어디 두고, 어떤 이름으로 보이는가

builtin 세 가지 (default · Explanatory · Learning) 외에, **사용자가 직접 만든 custom outputStyle** 도 쓸 수 있어요. settings.json 의 outputStyle 값이 builtin 이름이 아니면, harness 가 *같은 이름의 markdown 파일* 을 찾아서 그 본문을 system prompt 의 slot 13 에 박아 넣습니다.

<span class="note-flag">출처</span> builtin 목록 (Default / Explanatory / Learning) 은 공식 [Output styles 문서](https://code.claude.com/docs/en/output-styles) 의 *"Built-in output styles"* 섹션 + Claude Code 소스 constants/outputStyles.ts:39-58 (<span class="k">OUTPUT_STYLE_CONFIG</span> 객체) 양쪽에서 확인.

custom outputStyle 의 markdown 파일 모양은:

```markdown
---
name: Strict Reviewer
description: ...
keep-coding-instructions: true
---

본문 (LLM 에게 줄 prompt body)
```

이 파일이 위치할 수 있는 자리는 세 군데:

- **managed**: <span class="mod">/etc/claude-code/.claude/output-styles/*.md</span> (admin 배포, OS별 시스템 경로)
- **project**: <span class="mod">&lt;repo&gt;/.claude/output-styles/*.md</span> (체크인, 팀 공유)
- **user**: <span class="mod">~/.claude/output-styles/*.md</span> (개인)

같은 이름의 style 이 여러 자리에 있으면 *managed > project > user* 순으로 이김 (위에서 본 settings priority 와 같은 패턴).

trace 의 <span class="k">#&nbsp;Output&nbsp;Style:</span> 헤더에 들어가는 *이름* 은 **frontmatter 의 name 필드** 가 우선, filename 은 fallback. 즉 filename 이 strict-reviewer.md 라도 frontmatter 가 *Strict Reviewer* 면 trace 헤더에는 <span class="k">#&nbsp;Output&nbsp;Style:&nbsp;Strict&nbsp;Reviewer</span> 가 들어가요.

<div class="callout-warm">

**직접 한 번 돌려보기**

위 frontmatter name 우선 규칙이 실제로 trace 에서 어떻게 보이는지 확인해 봤어요. 두 파일을 만들고:

```bash
$ cat .claude/output-styles/strict-reviewer.md
---
name: Strict Reviewer
description: Reviews code with strict, thorough standards
---

You are a strict code reviewer. Be critical and thorough. Point out edge cases,
error handling gaps, and stylistic issues. Do not write code unless explicitly asked.

$ cat .claude/settings.json
{
  "outputStyle": "Strict Reviewer"
}
```

filename 은 *strict-reviewer* (kebab-case), frontmatter 의 name 은 *Strict Reviewer* (공백 + 대문자). 다르게 잡았어요. 세션 시작 후 trace 의 system_instructions 에서 slot 13 영역 발췌:

````text
# Output Style: Strict Reviewer
You are a strict code reviewer. Be critical and thorough. Point out edge cases,
error handling gaps, and stylistic issues. Do not write code unless explicitly asked.
````

<span class="key">**filename 은 'strict-reviewer' 였지만 trace 헤더는 'Strict Reviewer'**</span>. frontmatter name 이 이긴다는 게 trace 차원에서 확인됩니다.

</div>

## scratchpad slot 15: 평소엔 비어 있는 한 자리

[*System Block*](../system-prompt-assembly/) 의 동적 slot 표에서 slot 15 (scratchpad) 은 *기본 환경에선 보통 부재* 라고만 짧게 적었어요. 무슨 자리인지 잠깐 풀어 봅니다.

이 슬롯이 활성화되면 system prompt 에 다음 같은 한 블록이 추가돼요:

```
# Scratchpad Directory

IMPORTANT: Always use this scratchpad directory for temporary files
instead of `/tmp` or other system temp directories:
`/tmp/claude-1000/abc-def-.../<sessionId>/scratchpad`
...
```

내용은 단순해요. *세션마다 만들어지는 전용 임시 디렉토리의 경로* 와 *"임시 파일은 여기에 쓰세요"* 라는 지시. 그 디렉토리는 세션이 끝나면 자동으로 삭제되고, 다른 사용자는 접근 못 하게 권한이 잠겨 있어요.

다만 **이 기능은 Anthropic 내부에서 시험 중인 단계라 일반 사용자 환경에선 꺼져 있어요**. 그래서 trace 에 slot 15 가 안 보이는 게 정상. 평소엔 모델이 임시 파일을 만들 일이 있으면 그냥 시스템 기본 임시 디렉토리 (/tmp 같은 곳) 에 쓰게 됩니다 (사용자가 직접 보긴 어렵고, trace 에서나 확인 가능).

한 가지 짚을 점은, <span class="key">**이 슬롯이 알리는 건 디렉토리의 경로 한 줄 뿐**</span> 이라는 거예요. 그 안에 어떤 파일이 들어 있는지는 system prompt 에 안 들어옵니다. 모델이 궁금하면 Read 도구로 직접 들여다봐야 해요.

## MCP. 설정 파일은 안 들어가도, MCP 서버는 세 채널로 흘러 들어와요

자리를 옮깁니다. .mcp.json 자체는 설정 파일이라 LLM 에 안 닿아요. 그런데 *연결된 MCP 서버* 가 일단 켜지면 *세 채널* 로 동시에 흘러 들어옵니다.

```json
{
  "mcpServers": {
    "filesystem": { "command": "npx", "args": ["@mcp/filesystem", "/path"] },
    "playwright": { "command": "npx", "args": ["@playwright/mcp"] },
    "github":     { "url": "https://api.github.com/mcp", "headers": {...} }
  }
}
```

연결된 후 prompt 에 도달하는 세 채널:

<div class="callout-warm">

세 채널의 Block 분포 (A 가 Messages + Tools 양다리, B 는 System, C 는 Messages) 를 한 단계 더 들여다보면 *두 측면* 으로 깔끔하게 묶여요.

- 채널 A (tool 정의) + 채널 B (서버 instructions) → *LLM 이 **agent 운영** 을 결정할 때 쓰는 정보* (<span class="key">Harness Engineering</span> 측면)
- 채널 C (@-mentioned resource) → *LLM 이 **응답 생성** 을 결정할 때 쓰는 정보* (<span class="key">Context Engineering</span> 측면)

같은 MCP 서버 라도 *어떤 결정에 쓰일지* 에 따라 자리가 갈리는 거예요. 이 분기 기준의 본격적 정리는 [*6편 Long-term Memory*](../permanent-memory-three-channels/) 에서.
</div>

세 채널 중 *제일 무거운 건 채널 A* (tool 정의) 예요. 큰 MCP 서버 (수십~100 개의 tool 을 가진) 가 연결되면 매 turn 수만 토큰이 캐시 prefix 에 추가될 수 있어요. 그래서 Claude Code 는 처음부터 *defer* (필요할 때만 schema 를 받기) 를 default 로 채택했고, 그 메커니즘이 *ToolSearch* 입니다.

### ToolSearch: 무거운 tool 정의를 미루는 mode

기본 동작이에요. MCP tool 의 schema 는 *처음부터 한꺼번에 등록되지 않고*, <span class="k">ToolSearch</span> 라는 도구 하나만 turn 0 부터 등록돼요. 매 turn 에는 *tool 이름과 짧은 description 만* (전체 schema 가 아니라) 담은 attachment 한 덩어리가 흘러 들어옵니다. 모델이 *특정 tool 이 필요할 것 같다* 싶으면 그때 <span class="k">ToolSearch</span> 를 호출해서 그 tool 의 schema 를 그 turn 안에 받아요.

끄고 싶다면 환경변수 <span class="mod">ENABLE_TOOL_SEARCH=false</span> 로 *standard mode* 강제. 그러면 모든 MCP tool 의 schema 가 turn 0 부터 한꺼번에 등록되는 옛 동작.

단, 개별 MCP tool 이 메타데이터에 <span class="mod">alwaysLoad: true</span> 플래그를 set 해 두면 그 tool 만 turn 0 부터 schema 가 등록돼요. MCP 서버 작성자가 *"내 이 tool 은 너무 자주 쓰여서 defer 하지 마"* 라고 신호를 보내는 escape hatch.

<div class="callout-warm">

**직접 한 번 돌려보기**

<span class="mod">@modelcontextprotocol/server-everything</span> 서버 (15 개 tool 노출) 를 등록하고 trace 캡처:

<span class="k">gen_ai.tool.definitions</span> (tools panel) 에는 **9 개**만 들어 있어요. ToolSearch + Bash · Read · Edit · Write · Glob · Grep · Skill · Agent. **<span class="mod">mcp__everything__&#42;</span> 가 하나도 없음**.

대신 user message 의 <span class="mod">&lt;available-deferred-tools&gt;</span> attachment 안에 들어 있어요:

````text
... (Claude Code 내부 deferred tool 15 개)
mcp__everything__echo
mcp__everything__get-annotated-message
mcp__everything__get-env
... (총 15 개 mcp__everything__*)
````

<span class="key">**MCP tool 들이 turn 0 의 schema 에 안 들어가고, 이름·짧은 description 만 advertise**</span>. defer 가 default 동작인 게 trace 차원에서 확인돼요.

흥미로운 보너스. deferred pool 에는 MCP 외에도 *Claude Code 자신의 자주 안 쓰일 만한 내부 tool* (NotebookEdit, WebFetch, WebSearch, EnterPlanMode 류, Task* 6 종 등) 도 같이 들어 있어요. *MCP 전용 메커니즘* 이 아니라 *cache 비용을 줄이기 위한 일반 패턴* 임을 보여줘요.

</div>

<div class="surprise">
<span class="label">⚡ SURPRISE</span>

위 evidence 의 deferred attachment 를 다시 보면, *tool 이름 한 줄씩만* 들어가 있어요. **description 도 없고, parameter schema 도 없음**. 모델은 이름만 보고 *"이 tool 이 내가 지금 필요한 도구인가?"* 를 판단해야 합니다. 호출 가치 있다 싶으면 그제야 ToolSearch 로 schema 를 받아 와요.

직관적으론 *"이름만 보고 어떻게 잘 골라?"* 싶지만, 두 가지가 받쳐 줍니다. 이름 자체가 정보를 담고 있고 (예: <span class="mod">mcp__github__create_pr</span>, <span class="mod">mcp__playwright__browser_navigate</span> 처럼 *서버명 + 동사구* 형태), ToolSearch 를 *keyword 검색* 처럼 써서 후보를 탐색할 수도 있어요 (<span class="mod">browser</span> 같은 키워드로 브라우저 관련 도구 추리기).

이 *얇게 알리고, 호출 시에만 자세히* 패턴은 뒤에 다룰 Skills 의 토큰 절약 방식과 같은 디자인이에요.
</div>

<span class="label">💡 PRO TIP</span>

**MCP server 작성자는 tool 이름과 description 을 *서로 다른 역할로* 정성껏 짜야 해요.**

defer mode 의 호출 사이클은 두 단계로 나뉘어요:

- **이름** = <span class="key">발견성</span>. <span class="k">&lt;available-deferred-tools&gt;</span> 리스트엔 이름만 보이니까, 이름이 모호하면 모델이 *"이 tool 이 필요할 수도 있겠다"* 라는 의심조차 안 함 → 검색 시도 자체가 없음
- **description** = <span class="key">검색 적중</span>. 모델이 keyword 로 ToolSearch 를 호출하면 내부적으로 *name + description + searchHint* 모두에서 매칭. 이름이 약해도 description 에 좋은 keyword 가 있으면 잡힐 수 있음

즉 두 가지 다 잘 짜야 *발견되고 + 호출되는* 사이클이 돌아가요.

| 좋은 이름 | 나쁜 이름 |
|---|---|
| <span class="mod">create-pr</span>, <span class="mod">browser_navigate</span>, <span class="mod">search-issues</span> | <span class="mod">do_action</span>, <span class="mod">process_data</span>, <span class="mod">handle_x</span> |
| 짧고 명료한 *동사구* + 카테고리 prefix | 모호한 일반어 |

이름 prefix 로 카테고리 힌트를 담으면 더 좋아요 (<span class="mod">browser_&#42;</span>, <span class="mod">file_&#42;</span>, <span class="mod">git_&#42;</span>). 모델이 *"브라우저 관련 도구 있나?"* 같은 keyword 검색 시도할 때 정확히 잡힘.

</div>

## Take-home

이 글의 정리:

<blockquote>
<p><strong>settings.json 의 핵심 키 다섯</strong> (제가 본 빌드 기준). <span class="mod">language</span> · <span class="mod">outputStyle</span> · <span class="mod">autoMemoryEnabled</span> · <span class="mod">autoMemoryDirectory</span> · <span class="mod">claudeMdExcludes</span>. 4 개는 System Block 안에, 1 개 (claudeMdExcludes) 는 Messages Block의 <span class="k">#&nbsp;claudeMd</span> context 에 영향. 나머지 수십 개는 모두 harness 내부 plumbing.</p>
<p><strong>system block 본체를 갈아끼우는 두 입구</strong>. settings.json 으로는 손댈 수 없는 <span class="mod">customSystemPrompt</span> / <span class="mod">appendSystemPrompt</span> 가 <em>CLI flag</em> 와 <em>SDK</em> 양쪽으로 노출. 21 slot 본체를 <em>통째 갈아끼우거나 끝에 추가</em> 가능.</p>
<p><span class="key"><strong>outputStyle 이 가장 의외</strong></span>. 단순한 슬롯 추가가 아니라 <em>System Block의 네 자리</em> 를 동시에 수정 (정적 slot intro 다시 쓰기 + <span class="k">#&nbsp;Doing&nbsp;tasks</span> 제거 + 동적 slot 13 헤더 + style-specific sub-block). <em>정적 부분까지 깨는 유일한 settings 키</em>.</p>
<p><strong>MCP 는 .mcp.json 이 아니라 연결된 서버가 닿음</strong>. 세 채널 (tool 정의 / 서버 instructions / @-mentioned resource) 로 동시에. tool 정의 채널은 <em>defer 가 default</em> 라 schema 가 turn 0 에 안 들어가고, 모델이 ToolSearch 로 필요 시 받아 옴.</p>
</blockquote>
