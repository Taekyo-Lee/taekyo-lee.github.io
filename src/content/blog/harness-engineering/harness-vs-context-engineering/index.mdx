## Task 뒤엔 정교한 harness 가 있다

Claude Code 에 *"이 코드 분석해줘"* 같은 짧은 한 줄을 던졌다고 해 봅시다. 다섯 단어. 그런데 Langfuse trace 를 열어 보면 LLM 에게 도착한 prompt 는 12,000 토큰이 넘어요. 그 사이를 채운 토큰들은 어디서 왔을까요?

<Image src={harnessMachinery} alt="유저는 답변만 보지만 뒤에서는 정밀한 harness machinery 가 돌아간다" />

<p class="img-caption">유저가 보는 건 깜찍한 봇의 답변뿐이지만, 그 뒤에선 수많은 일꾼들이 코드를 읽고 · 도구를 돌리고 · 결과를 조립하는 정밀한 harness 가 돌아갑니다.</p>

답은 **harness** 입니다. user 의 입력과 LLM 사이에 끼어 있는 layer 예요.

## Harness 안의 Context Engineering

Harness 가 떠맡는 책임은 꽤 광범위해요. 도구 실행과 hook dispatch 같은 *agent 메커니즘*, 인증과 재시도와 error handling 같은 *runtime 신뢰성*, cost tracking · telemetry · audit log · guardrails 같은 *production 운영* 까지 두루 다룹니다. user 가 한 줄을 던지고 LLM 응답이 돌아올 때까지, 이 모든 일이 뒤에서 동시에 일어나거든요. 그 중에서 *"LLM 에게 어떤 정보를 어떻게 채워 넣을 것인가"* 만 따로 떼어 본 게 **context engineering** 이에요.

<span class="term">Context engineering</span> 이라는 용어는 2025 년에 Andrej Karpathy 가 popularize 했고, figure 안의 인용문이 그의 정의입니다 — *"context window 를 다음 step 을 위한 딱 맞는 정보로 채우는 섬세한 art and science"*. 단순히 *"좋은 prompt 를 쓰자"* 가 아니라, **이번 turn 에 LLM 이 봐야 할 정보를 정확히 추려서 정확한 자리에 넣는 일** 이에요.

다만 *harness engineering* 과 *context engineering* 사이의 **경계는 깔끔하지 않습니다.** 두 영역에 걸쳐 있는 대표 사례:

- **Hook** — harness 의 lifecycle event 메커니즘 *(harness engineering)* 이지만, 그 출력이 LLM context 에 주입되는 순간 *context engineering* 영역으로 넘어와요.
- **Tool 실행 결과** — tool 을 실제로 돌리는 건 harness 의 일 *(harness engineering)* 이지만, 그 결과가 *어디까지 · 어떤 모양으로 다음 LLM 호출에 들어가는지* 는 *context engineering* 의 결정이에요.

용어의 범위 자체도 author 마다 달라요 — 같은 단어가 정반대 의미로 쓰입니다.

<Image src={twoFramings} alt="Harness Engineering 과 Context Engineering — 두 framing 비교" />

<p class="img-caption">두 framing 모두 통용됩니다. 위 figure 는 왼쪽 (*context ⊂ harness*) 을 따랐어요.</p>

이 카테고리는 왼쪽의 **context ⊂ harness** framing 을 기본으로 갑니다. 큰 제목은 *harness engineering* 이지만 당분간 올라올 글은 그 안의 *context engineering* 이 대부분일 거예요. 어디서 출발해도 결국 같은 지형을 답사하게 되거든요.
