정적 블로그를 만들 때 가장 먼저 정해야 하는 건 **무엇으로 빌드할 것인가** 입니다. (여기서 *빌드* 는 사람이 쓴 소스, 즉 마크다운 글이나 설정 파일을, 브라우저가 그대로 읽을 수 있는 HTML/CSS/JS 묶음으로 변환하는 작업입니다.) 글은 보통 마크다운으로 쓰는 게 편합니다. 헤딩, 리스트, 링크 같은 기본 요소를 짧은 기호로 표현할 수 있으니까요. 다만 마크다운만으로는 시각 표현이 빈약합니다. 그래서 마지막 결과물은 표현이 자유로운 HTML 로 만듭니다. 레이아웃, 색, 인터랙션을 마음대로 줄 수 있는 게 HTML 이니까요.

Jekyll, Hugo, Sphinx, MkDocs 같은 기존 도구가 정확히 그 일을 합니다. 마크다운을 깔끔한 HTML 로 바꿔줍니다. 다만 본문 한가운데에 인터랙티브한 시각 요소를 넣고 싶을 때 한계가 옵니다. 이미지로 대체하거나, 외부 페이지로 빠져나가거나. 둘 다 글의 흐름을 끊습니다.

Astro 가 그 자리에서 한 발 더 나간 도구입니다. 단순 정리하면, <span class="key">마크다운으로 글을 쓰되, 필요한 자리에 component를 끼워 넣을 수 있는 도구.</span> 여기서 <span class="term">component</span> 는 HTML/CSS/JS 한 묶음을 하나의 재사용 단위로 쓸 수 있게 만든 작은 단위입니다. 예를 들어 카드 모양 도식 한 개, 또는 클릭하면 펼쳐지는 토글 박스 한 개를 component로 만들어두면, 본문 어느 자리에서든 한 줄로 불러서 띄울 수 있습니다.

## 먼저, Astro 는 정적 사이트 생성기입니다

Astro 가 어떤 카테고리에 속하는지부터 짚습니다. Sphinx 나 MkDocs 를 써본 적이 있다면, 그 도구들이 <span class="term">정적 사이트 생성기 (Static Site Generator, SSG)</span> 라는 분류에 속한다는 걸 알 것입니다. SSG 는 빌드 시점에 모든 페이지를 미리 HTML 로 만들어두는 프로그램입니다.

## Astro 가 Sphinx/MkDocs 와 다른 두 가지

여기까지면 그냥 "JavaScript 로 된 MkDocs" 정도입니다. Astro 가 그 자리에서 한 걸음 더 나가는 부분이 두 가지입니다.

**(a) component 기반.** Astro 는 React 나 Vue 처럼 UI 를 component로 쪼갤 수 있습니다. Header.astro, Footer.astro, ClosingHero.astro 처럼 재사용 단위로 분리할 수 있고, 더 중요한 건 <span class="key">.mdx 글 안에서 그 component들을 import 해서 본문 흐름에 직접 박아 넣을 수 있다</span> 는 것. 글 한가운데에 인터랙티브한 시각 요소를 넣을 때 필요한 자리가 정확히 이 자리입니다.

**(b) Islands 아키텍처 (기본은 HTML, JS 는 필요한 곳에만).** Astro 의 기본 출력은 **순수 HTML** 입니다. SPA 프레임워크들처럼 빈 페이지를 보내고 JS 로 채우는 방식이 아닙니다. 정말 인터랙티비티가 필요한 component만 자기 JS 번들을 갖습니다. 그 각각이 <span class="term">"island"</span> 한 개입니다. 블로그처럼 본문이 거의 정적인 사이트에서는, 페이지 로드가 굉장히 빠릅니다.

파이썬 엔지니어 관점에서 한마디로 줄이면. <span class="key">MkDocs 가 본문 안에 component를 import 할 수 있게 되고, 거기서 다큐먼트 사이트라는 한정된 용도를 벗어나 일반 웹 프레임워크로 확장된 도구.</span> 그게 Astro 입니다.

## .mdx 가 정확히 뭔가요

.md (Markdown) 는 익숙할 것입니다. .mdx 는 <span class="term">"Markdown + import 가능한 JSX component"</span> 입니다.

```mdx
---
title: '글 제목'
pubDate: 'Apr 27 2026'
category: about-astro
---

import ClosingHero from '../../components/ClosingHero.astro';

# 글 제목

여기는 평범한 마크다운 본문입니다.

<ClosingHero />  {/* 본문 한가운데 박힌 Astro component */}

마크다운이 다시 이어집니다…
```

위쪽 --- 사이에 들어간 YAML 블록이 <span class="term">frontmatter</span> 라고 부르는 메타데이터입니다. Astro 는 content.config.ts 에 적힌 Zod 스키마로 이걸 검증합니다. 예를 들어 category 필드가 정해진 enum 안에 있는지 빌드 시점에 체크할 수 있습니다. 오타를 내면 빌드가 실패하면서 어떤 파일의 어떤 값이 잘못됐는지 알려줍니다.

본문은 일반 마크다운인데, 그 사이사이에 &lt;ComponentName /&gt; 태그를 JSX 처럼 끼워 넣을 수 있습니다. 파이썬 비유를 들자면, **Jupyter 노트북이 마크다운 셀과 코드 셀을 섞듯, .mdx 는 마크다운과 UI component를 섞습니다.**

## `astro build` 가 실제로 하는 일

npm run build (안에서 astro build 가 호출됩니다) 를 돌리면 Astro 는 이런 단계를 거칩니다.

1. src/content/blog/ 아래의 모든 .md / .mdx 파일을 읽습니다.
2. 각 frontmatter 를 content.config.ts 의 스키마로 검증합니다 (잘못된 category 값은 여기서 걸립니다).
3. 마크다운을 HTML 로 렌더하면서, 본문 안의 component들도 함께 렌더합니다.
4. 공통 레이아웃 (BlogPost.astro 같은 것) 을 적용해서 nav / footer / meta 태그를 끼워 넣습니다.
5. CSS, 이미지, 폰트, 그 외 자산을 최적화합니다.
6. 모든 결과를 **dist/** 폴더에 씁니다.

dist/ 의 모양은 대략 이렇습니다.

```
dist/
├── index.html                         ← 홈
├── about/index.html                   ← About 페이지
├── blog/
│   ├── index.html                     ← 글 목록
│   └── some-post-slug/index.html      ← 개별 글
├── _astro/                            ← 번들된 CSS/JS
│   ├── main.abc123.css
│   └── ...
├── sitemap-index.xml
└── rss.xml
```

이 폴더 하나면 끝입니다. GitHub Pages, S3, 사내 정적 서버 — 어떤 정적 호스트든 그저 이 파일들을 서빙하기만 하면 됩니다. 방문자가 https://&lt;username&gt;.github.io/blog/some-post-slug/ 를 요청하면, 서버는 dist/blog/some-post-slug/index.html 을 그대로 돌려줍니다.

## `astro dev` 와 `astro build`, 어떻게 다른가

자주 헷갈리는 두 명령어를 정리합니다.

| 명령어 | 용도 | 출력 |
|---|---|---|
| npm run dev (= astro dev) | 로컬 미리보기. 파일을 저장하면 브라우저가 즉시 갱신 (핫리로드) | 없음. 메모리에서 서빙 |
| npm run build (= astro build) | 운영 빌드. 최적화/미니파이까지 모두 포함 | dist/ 폴더 |

파이썬 비유. uvicorn --reload 가 astro dev 자리고, mkdocs build / sphinx-build 가 astro build 자리입니다. **글을 작성하는 동안에는 dev, GitHub Actions 가 push 후에 돌리는 건 build.** 둘은 같은 도구의 다른 모드입니다.

## astro.config.mjs 가 설정하는 것들

대표적인 설정은 이렇게 생겼습니다.

```js
import { defineConfig } from 'astro/config';
import mdx from '@astrojs/mdx';
import sitemap from '@astrojs/sitemap';

export default defineConfig({
  site: 'https://<username>.github.io',
  integrations: [mdx(), sitemap()],
  // ... 폰트 설정 등
});
```

- site: 사이트가 배포될 공개 URL. sitemap.xml, canonical URL, RSS 피드를 만들 때 이 값을 씁니다. 잘못 넣으면 검색 엔진과 RSS 리더가 엉뚱한 주소를 따라가게 됩니다.
- integrations: Astro 에 기능을 더해주는 플러그인 (여기서는 .mdx 지원과 sitemap 자동 생성).

파이썬 비유. mkdocs.yml 의 site_url: 과 plugins: 블록 자리입니다.

## 정리

- Astro = JS 진영의 정적 사이트 생성기. Sphinx/MkDocs 의 자리에서 일반 웹 프레임워크로 확장됐습니다.
- .mdx = 마크다운에 component import 가 섞입니다.
- astro build → dist/ 에 완성된 HTML/CSS/JS. 그 폴더를 어떤 정적 호스트로든 보내면 끝.
- dev 는 글 쓸 때 보는 모드, build 는 배포할 때 굽는 모드.
