## 배경

Claude Code 에는 OpenTelemetry instrumentation 이 내장돼 있습니다. OTLP exporter 만 활성화해서 띄우면 한 prompt 의 lifecycle 이 외부 도구로 export 돼요. 받는 쪽이 [Langfuse](https://langfuse.com) 면 토큰 수, 시스템 프롬프트, tool 정의, 응답 본문, latency 까지 한 화면에 시각화됩니다.

<Image src={langfuseScreenshot} alt="Langfuse 에서 본 한 trace — 좌측 span 트리 (interaction → llm_request → tool → tool.blocked_on_user / tool.execution), 우측 Preview 패널에 user prompt 와 assistant 응답" />

<p class="img-caption">Langfuse 의 trace 화면. 왼쪽 <strong>span 트리</strong>에서 노드를 클릭하면, 오른쪽 패널에 그 단계의 자세한 메타 데이터 정보가 펼쳐집니다.</p>

OTLP 가 송신하는 데이터의 단위는 **trace** 와 **span** 두 개입니다.

이 reference 는 그 trace 안에 등장하는 모든 attribute 를 한 자리에 정리한 자료예요. Langfuse 에 도착한 한 span 에는 두 출처의 key 가 섞여 있는데, 하나는 **Claude Code 가 자체 instrumentation 으로 자동으로 넣는 key**, 다른 하나는 **Langfuse UI 가 잘 알아먹게 하려고 직접 추가한 key** (<span class="k">langfuse.observation.input/output</span>, <span class="k">gen_ai.&#42;</span>) 입니다. 거기에 표준 출처 구분 (<span class="k">OTel</span> / <span class="k">OTel-GenAI</span> / <span class="k">Langfuse</span>) 까지 더해져, *"이 attribute 는 누가 넣었고 누가 읽지?"* 가 trace 를 열어 볼 때마다 헷갈리거든요. 그 헷갈림을 한 번에 정리한 자료입니다.

정식 글로 풀기엔 너무 좁고 raw 한 내용이라 Lab Notes 로 둡니다.

## Attribute 카탈로그

Langfuse 가 tracing 하는 정보는 세 카테고리로 분류됩니다 (<span class="k">attributes.&#42;</span> / <span class="k">resourceAttributes.&#42;</span> / <span class="k">scope.&#42;</span>). 이 중 핵심 데이터는 첫 번째 <span class="k">attributes.&#42;</span> 에 모이고, 나머지 둘은 host / process 정보나 SDK 식별자 같은 process-scoped 메타데이터예요.

<Image src={metadataScreenshot} alt="Langfuse 의 한 span Metadata 패널 — attributes (37 items), resourceAttributes (10 items), scope (3 items) 세 카테고리가 빨간 박스로 강조" />

<p class="img-caption">한 span 의 <strong>Metadata 패널</strong>. <strong>attributes</strong> · <strong>resourceAttributes</strong> · <strong>scope</strong> 세 그룹이 나란히 놓이고, 각 그룹의 화살표를 클릭하면 그 안의 key-value 들이 펼쳐집니다.</p>

### Trace 구조: 세 그룹

각 그룹에 구체적으로 어떤 값이 들어가는지 한 줄씩 짚으면:

- <strong><span class="k">attributes.&#42;</span></strong>: 사용된 model, input/output, token, latency, span 종류 discriminator 등 그 trace 의 핵심 데이터. 섹션 1 ~ 5 에서 다룸.
- <strong><span class="k">resourceAttributes.&#42;</span></strong>: <span class="k">service.name</span>, <span class="k">host.name</span>, <span class="k">host.arch</span>, <span class="k">process.pid</span>, <span class="k">process.owner</span>, <span class="k">telemetry.sdk.&#42;</span> 같은 host / process / SDK 식별 정보. 섹션 6 에서 다룸.
- <strong><span class="k">scope.&#42;</span></strong>: <span class="k">&#123; name, version, attributes &#125;</span> 형태. trace 를 만든 instrumentation library 식별자. 섹션 7 에서 다룸.

### Provenance (출처)

각 표의 Provenance 컬럼에 붙는 태그는 그 key 가 어디서 왔고 누가 쓰는지를 표시합니다.

<div class="legend">
  <div class="legend-row"><span class="tag tag-otel">OTel</span> OpenTelemetry 표준 (관측 백엔드 중립)</div>
  <div class="legend-row"><span class="tag tag-genai">OTel-GenAI</span> OTel 의 LLM 특화 서브 namespace (역시 관측 백엔드 중립)</div>
  <div class="legend-row"><span class="tag tag-langfuse">Langfuse</span> Langfuse OTLP ingestion 이 UI 렌더링에 사용</div>
  <div class="legend-row"><span class="tag tag-upstream">Upstream</span> Claude Code 가 자체적으로 넣음</div>
  <div class="legend-row"><span class="tag tag-fork">Custom</span> 직접 추가한 key</div>
</div>

<span class="tag tag-otel">OTel</span> 과 <span class="tag tag-genai">OTel-GenAI</span> 가 처음이라면 짧게 — **OpenTelemetry (OTel)** 는 한 프로그램의 동작을 외부 관측 도구가 표준 형식으로 받을 수 있게 만든 **관측 백엔드 벤더 중립** 프레임워크예요. 여기서 *벤더* 는 Anthropic · OpenAI 같은 AI vendor 가 아니라 **trace 를 받아서 시각화해 주는 관측 백엔드 vendor** (Datadog, New Relic, Honeycomb, Jaeger, Splunk, Langfuse 등) 를 가리킵니다. 본문에서 OTel 태그가 붙은 key (예: <span class="k">service.name</span>, <span class="k">host.arch</span>, <span class="k">process.pid</span>) 는 그 어떤 백엔드든 같은 의미로 해석해요. **OTel-GenAI** 는 별개 표준이 아니라 같은 OTel 프로젝트 안에 들어 있는 **LLM / agent 시나리오용 서브 namespace** 입니다. 그래서 OTel 의 백엔드 중립성을 그대로 상속해요. 본문의 <span class="k">gen_ai.&#42;</span> 키 (예: <span class="k">gen_ai.request.model</span>, <span class="k">gen_ai.usage.input_tokens</span>, <span class="k">gen_ai.tool.definitions</span>) 가 거의 다 여기 해당하고, 관측 도구들이 token 사용량·tool 호출 같은 LLM 특화 UI 요소를 trace 에서 자동 추출할 때 보고 가는 key 들입니다.

요약: OTel 과 OTel-GenAI 는 **같은 표준의 두 범위** (일반 vs. LLM 특화) — 둘 다 *관측 백엔드* 중립이지만, 그 키를 받아서 *UI 로 어떻게 그려주느냐* 는 백엔드마다 다릅니다 (그래서 <span class="tag tag-langfuse">Langfuse</span> 태그가 별도로 필요).

---

<span class="k">attributes.&#42;</span> 안의 available key 들은 어떤 span 위에 붙느냐에 따라 달라지므로, **섹션 1 ~ 5 는 <span class="k">attributes.&#42;</span> 를 span 종류별로 묶은 것** 입니다. **섹션 6 은 <span class="k">resourceAttributes.&#42;</span>, 섹션 7 은 <span class="k">scope.&#42;</span>** 를 다룹니다 (본문 안의 섹션 3.1 / 섹션 3.4 같은 표기는 같은 카탈로그 안의 cross reference 입니다).

### 1. Identity & session (모든 span 에 공통)

<p class="section-lead">Claude Code 의 telemetry layer 가 자동으로 생성해 모든 span 에 merge 합니다.</p>

| Key | Provenance | Description |
| --- | --- | --- |
| attributes.user.id | <span class="tag tag-upstream">Upstream</span> | 머신 단위로 고정된 user 식별자. getOrCreateUserID() 가 홈 디렉토리의 식별자 파일에 저장한 random UUID. OAuth account id 와는 다름. |
| attributes.session.id | <span class="tag tag-upstream">Upstream</span> | 프로세스 단위 session UUID (getSessionId()). 프로세스를 새로 띄울 때마다 새 값. OTEL_METRICS_INCLUDE_SESSION_ID 로 gating (default on). interaction span 에서는 한 번 더 명시적으로 세팅됨. |
| attributes.terminal.type | <span class="tag tag-upstream">Upstream</span> | 감지된 터미널 emulator (vscode, iTerm.app, xterm 등). envDynamic.terminal 출처. |
| attributes.span.type | <span class="tag tag-upstream">Upstream</span> | span 종류 discriminator. 값: interaction, llm_request, tool, tool.blocked_on_user, tool.execution, hook. createSpanAttributes() 에서 무조건 세팅. |

<p class="section-lead">선택적 identity 필드 (해당 기능이 켜져 있을 때만 등장):</p>

- <span class="k">attributes.app.version</span>: 빌드 버전. <span class="k">OTEL_METRICS_INCLUDE_VERSION</span> gating (default off).
- <span class="k">attributes.organization.id</span>, <span class="k">attributes.user.email</span>, <span class="k">attributes.user.account_uuid</span>, <span class="k">attributes.user.account_id</span>: Anthropic OAuth 인증된 경우만. 보통 OAuth 인증을 안 쓰면 부재.

### 3. LLM-request span attributes (span.type = "llm_request")

<p class="section-lead">Claude Code 가 실제로 LLM API 를 호출할 때 만들어지는 span. 본 reference 의 대부분 key 가 여기 모입니다.</p>

#### 3.1 Request shape (span 시작 시점)

| Key | Provenance | Description |
| --- | --- | --- |
| attributes.model | <span class="tag tag-upstream">Upstream</span> | 호출에 넘어간 model id (예: claude-haiku-4-5-20251001, gpt-4o-mini). |
| attributes.llm_request.context | <span class="tag tag-upstream">Upstream</span> | interaction span 안에 nested 면 "interaction", 아니면 "standalone". |
| attributes.speed | <span class="tag tag-upstream">Upstream</span> | fastMode on 이면 "fast", 아니면 "normal". |
| attributes.query_source | <span class="tag tag-upstream">Upstream</span> | 호출을 쏜 시스템 부위. 값 enumeration 은 표 아래 §3.1.1 참고. |
| attributes.langfuse.span.name | <span class="tag tag-fork">Custom</span> <span class="tag tag-langfuse">Langfuse</span> | &lt;id&gt;:&lt;model&gt; 포맷의 span 이름. Langfuse 트리 뷰의 span 제목. |
| attributes.gen_ai.request.model | <span class="tag tag-fork">Custom</span> <span class="tag tag-genai">OTel-GenAI</span> | attributes.model 과 같은 값을 OTel 표준 key 로. |
| attributes.gen_ai.operation.name | <span class="tag tag-fork">Custom</span> <span class="tag tag-genai">OTel-GenAI</span> | "llm_call". |
| attributes.gen_ai.agent.name, attributes.gen_ai.agent.description | <span class="tag tag-fork">Custom</span> <span class="tag tag-genai">OTel-GenAI</span> | interaction span 과 같은 값. |

##### 3.1.1 <span class="k">query_source</span> 값 분류

##### 3.4.1 <span class="k">new_context</span> 의 세 가지 format

들어온 user 측 메시지의 종류에 따라 페이로드 모양이 달라지고, 어떤 모양인지는 **<span class="k">span.type</span> 으로 깔끔히 갈립니다**.

| Case | 발생 조건 | 페이로드 모양 | 어디 span 에 |
| --- | --- | --- | --- |
| **(a) 첫 turn** | user 가 prompt 를 처음 던짐 | 각 user 메시지에 <span class="k">[USER]</span> prefix, 사이는 <span class="k">\n\n---\n\n</span> 으로 join | <span class="k">span.type = llm_request</span> |
| **(b) Synthesis turn** | 여러 tool result 를 한 번에 받아 model 이 종합 | 각 tool result 가 <span class="k">[TOOL RESULT: &lt;toolName&gt;]\n&lt;JSON&gt;</span> 블록, 같은 separator 로 join | <span class="k">span.type = llm_request</span> |
| **(c) Tool span 위** | parent tool span 자체에 결과 attach | 단일 <span class="k">[TOOL RESULT: &lt;toolName&gt;]\n&lt;JSON&gt;</span> 블록 (그 한 tool 호출 결과) | <span class="k">span.type = tool</span> (별도 path: <span class="k">addBetaToolResultAttributes</span>) |

##### 3.4.2 <span class="k">system_reminders</span> — 부재 규칙과 hook injection

**부재 규칙**: 새 reminder 가 없는 turn 에서는 attribute key 자체가 **존재하지 않습니다** (present-but-empty 가 아니에요). 새 reminder 가 있을 때만 <span class="k">setAttribute</span> 가 호출되기 때문. telemetry 쿼리는 <span class="k">WHERE system_reminders IS NOT NULL</span> 로 필터하세요 — <span class="k">!= ''</span> 는 잘못된 패턴.

**Hook injection 두 path** (둘 다 empirical 검증됨):

| Path | 해당 event | 페이로드 모양 |
| --- | --- | --- |
| **(a) stdout path** | <span class="k">SessionStart</span>, <span class="k">UserPromptSubmit</span> 만 | 캡처된 stdout 이 <span class="k">&lt;hookName&gt; hook success: &lt;stdout&gt;</span> 블록으로 append |
| **(b) additionalContext path** | 대부분 event (<span class="k">PreToolUse</span>, <span class="k">PostToolUse</span> 등) | hook 의 JSON return 안 <span class="k">hookSpecificOutput.additionalContext</span> 가 <span class="k">&lt;hookName&gt; hook additional context: &lt;content&gt;</span> 블록으로 append |

**`<hookName>` 자동 생성 규칙**:
- tool 외 event → bare 이름 (예: <span class="k">UserPromptSubmit</span>)
- tool-related event → <span class="k">&lt;event&gt;:&lt;matcher&gt;</span> (예: <span class="k">PreToolUse:Bash</span>, <span class="k">PostToolUse:Edit|Write</span>)

두 LLM 호출 사이에 hook 이 여러 번 발동하면, 다음 호출의 <span class="k">system_reminders</span> 에 모두 join 됩니다.

#### 4.4 Agent tool wrapper · special case

<p class="section-lead"><span class="k">tool_name = "Agent"</span> 인 경우, parent tool span 의 <span class="k">tool_input</span> / <span class="k">new_context</span> 에는 다른 tool 들보다 훨씬 상세한 구조의 payload 가 들어갑니다.</p>

**tool_input 페이로드** (<span class="k">[TOOL INPUT: Agent]\n</span> 뒤):

```json
{
  "description": "<short summary the parent provides>",
  "prompt": "<the actual user-message content the subagent receives>",
  "subagent_type": "general-purpose"
}
```

**new_context 페이로드** (<span class="k">[TOOL RESULT: Agent]\n</span> 뒤) — 시스템에서 가장 정보량이 많은 tool result:

```json
{
  "status": "completed",
  "prompt": "<echo of the prompt sent>",
  "agentId": "<unique 17-char identity for this subagent run>",
  "agentType": "general-purpose",
  "content": [
    { "type": "text", "text": "<subagent's final reply>" }
  ],
  "totalDurationMs": 3337,
  "totalTokens": 6184,
  "totalToolUseCount": 1,
  "usage": {
    "input_tokens": 6112,
    "output_tokens": 72,
    "cache_creation_input_tokens": 0,
    "cache_read_input_tokens": 0
  }
}
```

Agent wrapper 의 특이점:

- <span class="k">agentId</span> 는 cross-trace linkage 역할 — parent LLM 이 받는 tool_result 문자열에도 embed 되어 *"(use SendMessage with to: '&lt;agentId&gt;' to continue this agent)"* 힌트로 multi-step subagent flow 가능.
- <span class="k">usage.iterations[]</span> 는 현재 build 에서 비어 있음. schema 상으로는 subagent 내부 LLM 호출 별 usage breakdown 을 가져야 하지만, population path 가 dormant.
- <span class="k">totalTokens</span> / <span class="k">totalDurationMs</span> / <span class="k">totalToolUseCount</span> 는 subagent 내부 LLM 호출 전체의 aggregate. 이 LLM 호출들은 별도 sibling span 으로 등장하며 <span class="k">query_source</span> 가 <span class="k">agent:builtin:&lt;name&gt;</span>.
- subagent 의 실제 LLM 호출 trace 는 <span class="k">session.id</span> 만으로 연결됨 (<span class="k">parent_trace_id</span> attribute 부재 — known gap).

### 5. Hook spans (span.type = "hook")

<p class="section-lead">Claude Code 의 hook 이 실행될 때 만들어지는 span. <span class="k">startHookSpan</span> 에서 생성, <span class="k">endHookSpan</span> 에서 종료. beta tracing 으로 gating 되어 있어, off 면 active span 을 그대로 반환 (no-op).</p>

#### 5.1 span 시작 시점

| Key | Provenance | Description |
| --- | --- | --- |
| attributes.span.type | <span class="tag tag-upstream">Upstream</span> | 리터럴 "hook". |
| attributes.hook_event | <span class="tag tag-upstream">Upstream</span> | hook event 식별자 (<span class="k">UserPromptSubmit</span>, <span class="k">PreToolUse</span>, <span class="k">PostToolUse</span>, <span class="k">SubagentStart</span>, <span class="k">Stop</span>, <span class="k">SessionStart</span> 등). |
| attributes.hook_name | <span class="tag tag-upstream">Upstream</span> | hook 의 설정된 이름. config 에 명시 안 했으면 event 종류에 따라 자동 생성 — 규칙은 §5.1.1 참고. |
| attributes.num_hooks | <span class="tag tag-upstream">Upstream</span> | 이 event 에 등록된 hook 수 (informational; 전체 matcher set 범위). OTLP attribute-value 인코딩 때문에 string 으로 (<span class="k">"1"</span>, <span class="k">"2"</span>). |
| attributes.hook_definitions | <span class="tag tag-upstream">Upstream</span> | matched hook definition 들의 JSON 배열 (각 entry: <span class="k">type</span> + <span class="k">command</span> + matcher metadata). 예: <span class="k">[&#123;"type":"command","command":"echo 'hook fired ...'"&#125;]</span>. |

##### 5.1.1 <span class="k">hook_name</span> 자동 생성 규칙

| Event 종류 | hook_name 형태 | 예 |
| --- | --- | --- |
| **tool 외** event (<span class="k">UserPromptSubmit</span>, <span class="k">SessionStart</span>, <span class="k">Stop</span> 등) | event 이름 그대로 fallback | <span class="k">"UserPromptSubmit"</span> |
| **tool-related** event (<span class="k">PreToolUse</span>, <span class="k">PostToolUse</span>) | <span class="k">&lt;event&gt;:&lt;matcher&gt;</span> | <span class="k">"PreToolUse:Bash"</span>, <span class="k">"PostToolUse:Bash"</span> |

이 자동 생성 이름은 §3.4 의 <span class="k">system_reminders</span> 안에 model 이 보는 그대로 들어가는 문자열이에요. 그래서 grep 할 때는 bare <span class="k">PreToolUse hook ...</span> 가 아니라 <span class="k">PreToolUse:Bash hook ...</span> 가 정답.

#### 5.2 span 종료 시점

<p class="section-lead">아래 4 카운터는 **항상 set** (값이 0 이어도 생략 안 됨). 0 일 때는 OTLP 정수 wrapper 로 들어와요 — <span class="k">&#123;intValue: 0&#125;</span>.</p>

| Key | Provenance | Description |
| --- | --- | --- |
| attributes.duration_ms | <span class="tag tag-upstream">Upstream</span> | hook 실행 wall time. 숫자 string (<span class="k">"28"</span>) 으로. |
| attributes.num_success | <span class="tag tag-upstream">Upstream</span> | 에러 없이 완료된 hook 수. |
| attributes.num_blocking | <span class="tag tag-upstream">Upstream</span> | action 을 막은 hook 수 (예: PreToolUse 가 <span class="k">decision: deny</span> 반환). |
| attributes.num_non_blocking_error | <span class="tag tag-upstream">Upstream</span> | 에러는 났지만 block 하지 않은 hook 수. |
| attributes.num_cancelled | <span class="tag tag-upstream">Upstream</span> | cancel 된 hook 수 (예: user interrupt). |

Hook span 에 관한 중요 포인트:

- tool span 처럼 <span class="k">langfuse.observation.input</span> / <span class="k">.output</span> 을 **가지지 않습니다**. Langfuse 가 attribute table 을 직접 렌더.
- 개별 hook 의 <span class="k">command</span> / <span class="k">exit_code</span> 가 span attribute 로 노출되지 않아요. <span class="k">hook_definitions</span> JSON 안과 <span class="k">num_*</span> 카운터 분포로 추적합니다.
- **Hook 의 stdout 이 LLM 에 도달하는 경로는 hook span attribute 가 아니라 다음 LLM-call span 의 <span class="k">system_reminders</span>** (섹션 3.4 참고). 즉 hook span 은 *"hook 이 이런 결과로 돌았다"* 를 알려주고, model 이 실제로 본 stdout 은 다음 LLM-call span 의 system_reminders 에서 확인.

## Content patterns (substring 패턴)

섹션 1~7 카탈로그가 *attribute 이름* 까지 였다면, 이 섹션은 *attribute 값 안에 등장하는 substring 패턴* 입니다. grep / inspect 할 때 anchor 로 쓰세요.

### System prompt 안 — <span class="k">gen_ai.system_instructions</span>

**맨 위 헤더 라인.** <span class="k">x-anthropic-billing-header: cc_version=&lt;ver&gt;; cc_entrypoint=cli;</span> 가 항상 첫 줄. 끝의 3-char <span class="k">cc_version</span> suffix (<span class="k">dd8</span>, <span class="k">f17</span> 등) 는 **call-type-dependent** — 같은 process 안에서도 parent trace 와 subagent trace 의 suffix 가 다릅니다 (system-prompt 본문의 content hash 로 추정). <span class="k">query_source</span> 외에 call type 을 fingerprint 하는 보조 단서.

**<span class="k">repl_main_thread</span> system prompt 조립 순서.** 위에서 아래로 다음 순서로 concatenate 됩니다:

<div class="prompt-flow">
  <div>정적 intro</div>
  <div><span class="k"># System</span></div>
  <div><span class="k"># Doing tasks</span> <span class="flow-note">— output style 이 <span class="k">keepCodingInstructions: false</span> 면 suppress</span></div>
  <div><span class="k"># Executing actions with care</span></div>
  <div><span class="k"># Using your tools</span></div>
  <div><span class="k"># Tone and style</span></div>
  <div><span class="k"># Output efficiency</span></div>
  <div class="flow-divider">— boundary —</div>
  <div><span class="k"># Session-specific guidance</span></div>
  <div><span class="k"># auto memory</span></div>
  <div><span class="k"># Environment</span></div>
  <div><span class="k"># Language</span> <span class="flow-note">— 있을 때만</span></div>
  <div><span class="k"># Output Style: &lt;name&gt;</span> <span class="flow-note">— 있을 때만</span></div>
  <div><span class="k">summarize_tool_results</span> 한 줄</div>
  <div><span class="k">gitStatus: ...</span> 블록 <span class="flow-note">— 항상 마지막</span></div>
</div>

**Subagent 호출** (<span class="k">query_source: agent:builtin:&#42;</span>): 훨씬 짧은 agent-specific template. <span class="k">gitStatus</span> 없음 · <span class="k"># Doing tasks</span> 없음 · parent 의 대부분 섹션 부재.

### Conversation 입력 안 — <span class="k">langfuse.observation.input</span>

conversation 메시지 array. 섹션 3.1 의 렌더링 규칙 외에 실제 trace 에서 자주 grep 되는 substring:

| 패턴 | 의미 |
| --- | --- |
| <span class="k">&lt;system-reminder&gt;...&lt;/system-reminder&gt;</span> | 2026-05-04 fix 이후 verbatim 보존. claudeMd · MEMORY.md · 첨부 · hook 컨텍스트 · todo 알림 등이 모두 여기 담김. |
| <span class="k"># claudeMd</span> | <span class="k">&lt;system-reminder&gt;</span> 안에서 <span class="k">prependUserContext</span> 가 prepend 한 bundle header. 뒤에 <span class="k">Contents of &lt;path&gt; (&lt;description&gt;)</span> 블록이 따라 붙음. |
| <span class="k"># currentDate\nToday's date is YYYY-MM-DD.</span> | <span class="k">CLAUDE_CODE_DISABLE_CLAUDE_MDS=1</span> 이 아닌 한 항상 <span class="k"># claudeMd</span> 옆에 등장. disable 시에는 claudeMd 블록 전체가 빠지고 <span class="k"># currentDate</span> 만 살아남음. |
| <span class="k">[Tool call: &lt;toolName&gt;] &#123;&lt;JSON&gt;&#125;</span> | <span class="k">tool_use</span> content 블록 (2026-05-06 fix). JSON 은 tool 의 인자. |
| <span class="k">[Tool result] &lt;content&gt;</span> | <span class="k">tool_result</span> content 블록. Agent tool result 면 <span class="k">agentId: &lt;id&gt;</span> 와 <span class="k">&lt;usage&gt;...&lt;/usage&gt;</span> 블록 포함 가능. |
| <span class="k">&lt;command-name&gt;/&lt;cmd&gt;&lt;/command-name&gt;</span> &nbsp;+ <span class="k">&lt;command-message&gt;</span> + <span class="k">&lt;command-args&gt;</span> | slash command invocation 의 XML header. 그 뒤에 <span class="k">$N</span> 인자 substitution 이 적용된 resolved command body 가 따라옴. |
| <span class="k">Called the &lt;Tool&gt; tool with the following input: &lt;JSON&gt;</span> + <span class="k">Result of calling the &lt;Tool&gt; tool: &lt;content&gt;</span> | file 첨부 (예: compact 후 CLAUDE.md re-attach) 를 렌더하는 **합성** tool-call/tool-result 문자열. 진짜 tool call 이 아닙니다. |
| <span class="k">This session is being continued from a previous conversation that ran out of context.</span> | compaction preamble. 뒤에 <span class="k">&lt;summary&gt;...&lt;/summary&gt;</span> 가 따라옴. *"이 turn 은 post-compaction 인가?"* 의 grep anchor. |
| <span class="k">&lt;hookName&gt; hook success: &lt;stdout&gt;</span> | <span class="k">SessionStart</span> / <span class="k">UserPromptSubmit</span> hook 의 stdout 캡처. |
| <span class="k">&lt;hookName&gt; hook additional context: &lt;content&gt;</span> | 임의 event 의 <span class="k">additionalContext</span> JSON-return injection. <span class="k">&lt;hookName&gt;</span> 은 tool 관련 event 의 경우 <span class="k">&lt;event&gt;:&lt;matcher&gt;</span> (예: <span class="k">PreToolUse:Bash</span>). |
| <span class="k">&lt;hookName&gt; hook blocking error from command: ...</span> | <span class="k">PreToolUse</span> hook 이 <span class="k">decision: "block"</span> 반환한 경우 (드물게 등장). |

### Compactor 출력 안 — <span class="k">langfuse.observation.output</span>

(<span class="k">query_source: compact</span> 일 때) compactor 가 produce 하는 **고정 XML shape**, 9개 섹션이 항상 같은 순서로 등장합니다:

```
<analysis>
[free-form analysis the model wrote while thinking]
</analysis>

<summary>
1. Primary Request and Intent: ...
2. Key Technical Concepts: ...
3. Files and Code Sections: ...
4. Errors and fixes: ...
5. Problem Solving: ...
6. All user messages: ...
7. Pending Tasks: ...
8. Current Work: ...
9. Optional Next Step: ...
</summary>
```

이 9-section template 은 **rigid** — non-coding 대화에도 동일하게 적용됩니다.
