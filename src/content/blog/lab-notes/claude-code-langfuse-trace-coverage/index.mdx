[Langfuse Trace Keys Reference](../claude-code-langfuse-trace-keys/) 의 부록 격 노트입니다. 그 reference 가 어디까지 *직접 관측된* 결과이고 어디부터 *추정·미관측* 인지 — 즉 어떤 row 를 신뢰해도 되고 어떤 row 는 더 검증이 필요한지 — 를 솔직히 적어 둡니다.

본 reference 는 직접 수집한 **40여 개의 Langfuse trace** 를 분석해 작성했습니다. 거기서 관측된 attribute 는 모두 [Reference 의 섹션 1~7](../claude-code-langfuse-trace-keys/) 에 row 가 있어요. 다만 그 40개 trace 로 Claude Code 의 가능한 코드 경로를 전부 돌려본 건 아니라서, 일부 카테고리는 정보가 부분적이거나 아예 비어 있습니다.

---

## 솔직히 다 본 것 (관측 + 소스 확인)

- 4가지 주요 <span class="k">span.type</span> — interaction · llm_request · tool (+ tool.execution, tool.blocked_on_user) · hook
- 모든 <span class="k">resourceAttributes.&#42;</span> (12개)
- 모든 <span class="k">scope.&#42;</span> (3개)
- Agent tool 의 structured envelope (가장 데이터가 풍성한 tool result)
- per-turn delta 의미 — <span class="k">new_context</span>, <span class="k">system_reminders</span> 의 omission rule 과 content 패턴
- Hook injection 두 path (stdout / additionalContext) — 실제 <span class="k">&lt;system-reminder&gt;</span> 포맷까지 검증

## 부분 커버 (Reference 섹션 4~5 안에서 명시)

- **per-tool 추가 attribute** — Bash 의 <span class="k">full_command</span> 만 관측. Read · Edit · Write · Glob · Skill · ToolSearch 도 <span class="k">file_path</span> 등 자기 진단 필드를 가질 가능성이 높지만 trace 가 없어 미확인.
- **tool result JSON schema** — Bash · Grep · Agent 만 정리. 나머지 5개는 source-readable 이지만 미열거.
- **multi-hook 구성** — 1 hook/event 만 테스트. N>1 일 때의 <span class="k">hook_definitions</span> array 직렬화 미검증.
- **hook event 종류** — <span class="k">UserPromptSubmit</span> · <span class="k">PreToolUse</span> · <span class="k">PostToolUse</span> 만 trace 로 exercise. 나머지 17개 (<span class="k">PreCompact</span>, <span class="k">PostCompact</span>, <span class="k">SessionEnd</span>, <span class="k">Notification</span>, <span class="k">PermissionRequest</span> 등) 는 schema 만 있고 span 동작 미확인.

## 아예 안 다룬 것 (정직한 gap)

- **OTel envelope 필드** — <span class="k">trace_id</span>, <span class="k">span_id</span>, <span class="k">parent_span_id</span> 는 OTLP span envelope 에 살고 <span class="k">attributes.&#42;</span> 가 아닙니다. raw OTLP export 와 Langfuse 의 span 트리에서는 보이지만, application 이 set 하지 않고 OTel SDK 가 채움.
- **Span events** — OTel 은 "events" (timestamped log entry, 자체 attribute 보유) 와 "attributes" (span 의 key-value) 를 구분합니다. Reference 본문에서 두 event 를 지나가며 언급 — <span class="k">system_prompt</span> (섹션 3.2) 와 <span class="k">tool</span> (섹션 3.3) — 하지만 event 자체의 schema (event 이름 + event attribute) 는 미열거. OTel logs 를 같이 inspect 하는 경우 거기서 더 찾을 수 있어요.
- **MCP-specific attribute** — MCP server connect / disconnect 시 <span class="k">mcp_instructions_delta</span> gating 과 tool 정의 refresh 가 일어나는데, 전용 span attribute 가 emit 되는지는 미확인 (test 환경에 MCP server 없음).
- **<span class="k">compact</span> query_source 변종** — bare <span class="k">compact</span> 와 <span class="k">session_memory</span> 만 관측. <span class="k">microCompact.ts</span> 등 source 에 더 있을 수 있지만 trace 미관측.
- **실패 모드 attribute payload 구조** — <span class="k">status_code</span> / <span class="k">error</span> 가 only-on-failure 라고 적었지만, provider-specific error payload (rate limit, malformed response, timeout 등) 의 *구조* 는 미열거.
- **Langfuse 가 인식하는 추가 extension key** — <span class="k">observation.level</span>, <span class="k">observation.metadata</span>, <span class="k">observation.status_message</span> 등 fork 가 set 하지 않는 valid key. 35개 trace dump 모두에서 부재 확인. 찾을 일 없도록 여기 남김.

## 독자에게 의미하는 것

- 관측된 trace 하나를 열면 **모든 visible attribute** 가 [Reference 의 섹션 1~7](../claude-code-langfuse-trace-keys/) 에 row 로 존재합니다.
- exercise 안 한 path 의 trace (MCP, custom subagent, multi-hook 구성, rare hook event) 를 보면 1~2 개 미수록 attribute 가 있을 수 있어요. bug 가 아니라 gap. 유용한 게 보이면 추가 제안 환영.

## Known absent attributes (audit gap)

아래 attribute 들은 *유용해 보이지만 현재 어떤 span 에도 등장하지 않습니다.* "왜 X 가 없지?" 의 답:

| 제안 attribute | 무엇을 담을까 | 부재가 왜 아쉬운가 |
| --- | --- | --- |
| <span class="k">attributes.attachment_sizes</span> | 첨부 이름 (<span class="k">claudeMd</span>, <span class="k">skill_listing</span>, <span class="k">nested_memory</span>, <span class="k">changed_files</span> 등) → token count 의 map | per-attachment token attribution. 무거운 CLAUDE.md 가 silent 하게 input_tokens 를 부풀려도 cost 를 attribute 할 방법이 없음. |
| <span class="k">attributes.cache_prefix_busted_section</span> | 정적 cache prefix 가 invalidate 됐을 때 어떤 섹션이 bust 했는지 (<span class="k">intro_rewrite</span>, <span class="k">doing_tasks_suppression</span>, <span class="k">tools_change</span> 등) | *"왜 turn N 에서 prefix 가 bust 했지?"* 진단. 현재는 miss 만 보일 뿐 cause 를 알 수 없음. |
| <span class="k">attributes.parent_trace_id</span> (또는 <span class="k">parent_span_id</span>) | subagent LLM-call span 에서 parent main-thread trace 식별 | 현재 cross-trace linkage 가 <span class="k">session.id</span> 만으로 됨. multi-agent flow 를 navigate 하기 어려움. |
| <span class="k">attributes.hook_context_injected</span> + <span class="k">attributes.hook_event_names</span> | "이 turn 에 hook injected content 가 포함됐다" 플래그 | 현재 <span class="k">system_reminders</span> 안의 substring (` hook success:`, ` hook additional context:`) 매칭으로만 detect 가능 — 전용 attribute 가 있으면 telemetry 쿼리가 깔끔. |
| <span class="k">attributes.subagent_start_additional_context</span> | subagent 첫 turn span 에 boolean 플래그 | <span class="k">SubagentStart</span> hook 의 <span class="k">additionalContext</span> 가 subagent 첫 user 메시지에 들어가지만 hook-derived 임이 표시 안 됨. |

찾고 있는데 못 찾는다면 — 그게 정상이에요. 계획됐지만 미배포 상태.

## OTLP integer encoding gotcha

OTLP 는 정수형 attribute 를 trace 안에서 일관되지 않게 인코딩합니다. Langfuse metadata 뷰에서 같은 logical key 가 다른 'shape' 으로 보이는 이유:

| 인코딩 | 등장 시점 | 예 |
| --- | --- | --- |
| **숫자 string** (<span class="k">"345944"</span>, <span class="k">"12621"</span>, <span class="k">"28"</span>) | 대부분의 non-zero 정수 attribute | <span class="k">process.pid: "345944"</span>, <span class="k">input_tokens: "12621"</span>, <span class="k">duration_ms: "28"</span>, <span class="k">num_hooks: "1"</span> |
| **Wrapped 정수** (<span class="k">&#123;intValue: 0&#125;</span>) | 0 값 정수 (특히 token-related counter) | <span class="k">cache_read_tokens: &#123;intValue: 0&#125;</span>, <span class="k">num_blocking: &#123;intValue: 0&#125;</span> |
| **Native number** | 일부 resourceAttributes (예: <span class="k">process.pid</span>) | <span class="k">process.pid: 345944</span> |
| **Native boolean** | 대부분의 boolean attribute | <span class="k">success: true</span>, <span class="k">response.has_tool_call: false</span> |

OTLP 의 표준 <span class="k">AnyValue</span> envelope 방식 때문이에요. 값이 정확히 0 (또는 다른 'default' 타입) 일 때는 wire format 이 type tag 를 visible 로 유지 (<span class="k">&#123;intValue: 0&#125;</span>); non-zero 면 type tag 가 collapse 되고 string 으로 렌더. **실용적 영향**: Langfuse metadata 를 programmatically 쿼리할 때 같은 logical key 에 대해 두 shape 모두 처리하세요.
