이 시리즈 [1편에서 Anthropic 의 공식 marketplace](/blog/real-world-plugin-anthropic-agent-skills/) 를 봤어요. *"17 skills 를 페르소나별 3 plugin 으로 묶은"* 패턴이었죠. 이번엔 정반대 모양의 marketplace 입니다. OpenAI 가 만든 [codex marketplace](https://github.com/openai/codex-plugin-cc). **Plugin 은 단 한 개인데, 그 한 개가 plugin component type 거의 전부 (skills · commands · agents · scripts · hooks) 를 안에 박아 넣은 '풍성한 single plugin'** 의 모범 사례예요.

이 repo 도 **marketplace 형태로 publish** 돼 있어서, `/plugin marketplace add openai/codex-plugin-cc` 한 줄로 marketplace 를 등록해 두면 codex plugin 을 Claude Code 안에서 쉽게 다운로드받을 수 있어요.

이 marketplace 의 [`marketplace.json`](https://github.com/openai/codex-plugin-cc/blob/main/.claude-plugin/marketplace.json) 은 이렇게 생겼어요. `plugins` 배열에 codex 한 개만 들어 있고, `source: "./plugins/codex"` 로 실제 plugin 디렉터리를 가리킵니다.

```json
{
  "name": "openai-codex",
  "owner": {
    "name": "OpenAI"
  },
  "metadata": {
    "description": "Codex plugins to use in Claude Code for delegation and code review.",
    "version": "1.0.3"
  },
  "plugins": [
    {
      "name": "codex",
      "description": "Use Codex from Claude Code to review code or delegate tasks.",
      "version": "1.0.3",
      "author": {
        "name": "OpenAI"
      },
      "source": "./plugins/codex"
    }
  ]
}
```

## 🎭 1 Plugin · Cross Agent Harness

<div class="plugin-card">

### <span class="term">codex</span> plugin <span class="persona-tag">FOR · Cross Agent Harness</span>

이름 그대로, OpenAI 의 자체 코딩 agent **Codex 를 Claude Code 안에서 호출할 수 있게 만든 다리** 예요. 두 가지 큰 시나리오:

- **Code review**: `/codex:review` 는 현재 작업을 OpenAI 의 reviewer 에게 보여 줍니다. `/codex:adversarial-review` 는 좀 더 공격적인 challenge 모드.
- **Task delegation**: `/codex:rescue` 는 Claude 가 막혔거나 다른 시각이 필요할 때 Codex 에 작업을 넘깁니다. `/codex:status` · `/codex:result` · `/codex:cancel` 로 백그라운드 job 을 관리.

전제 조건: ChatGPT 구독 또는 OpenAI API 키 + Node 18.18 이상. `@openai/codex` CLI 가 시스템에 깔려 있어야 하고요 (`/codex:setup` 이 환경 점검과 자동 설치까지 도와줍니다).

</div>

## 📜 <span class="term">Commands</span> in plugin

위 callout 에서 (user-defined) slash command 가 *"코드가 아니라 Claude 한테 줄 prompt"* 라고 했죠. 너무 추상적으로 들리니, 실제 파일 한 장을 같이 봅시다. `/codex:setup` 의 정체는 `commands/setup.md` 입니다.

````markdown
---
description: Check whether the local Codex CLI is ready and optionally toggle the stop-time review gate
argument-hint: '[--enable-review-gate|--disable-review-gate]'
allowed-tools: Bash(node:*), Bash(npm:*), AskUserQuestion
---

Run:

```bash
node "${CLAUDE_PLUGIN_ROOT}/scripts/codex-companion.mjs" setup --json $ARGUMENTS
```

If the result says Codex is unavailable and npm is available:
- Use `AskUserQuestion` exactly once to ask whether Claude should install Codex now.
- Put the install option first and suffix it with `(Recommended)`.
- Use these two options:
  - `Install Codex (Recommended)`
  - `Skip for now`
- If the user chooses install, run:

```bash
npm install -g @openai/codex
```

- Then rerun the setup command above.

Output rules:
- Present the final setup output to the user.
- If Codex is installed but not authenticated, preserve the guidance to run `!codex login`.
````

이게 전부예요. 이 한 장의 markdown 이 슬래시 명령의 모든 동작을 정의합니다.

SKILL.md 랑 비슷해 보이죠? 다만 결정적인 차이가 있어요. SKILL.md 는 Claude Code 가 정해 둔 schema (필수 frontmatter 필드, 폴더 구조 등) 를 따라야 하지만, `commands/*.md` 는 본질적으로 *그저 LLM 에게 줄 prompt 가 담긴 markdown 파일* 일 뿐이에요. 위에 보이는 frontmatter 형식도 사실 강제 사항이 아니라 권장 컨벤션입니다.

짚어볼 점:

- **Frontmatter** 는 메타데이터 선언입니다. `description` 은 `/help` 에 뜨는 한 줄, `argument-hint` 는 자동완성, `allowed-tools` 는 도구 화이트리스트. 여기서는 `Bash(node:*), Bash(npm:*), AskUserQuestion` 셋만 허용 — 임의 셸 명령 못 돌립니다.
- **본문은 Claude 가 따라 실행할 prompt** 입니다. *"node ...codex-companion.mjs setup --json \$ARGUMENTS 를 돌리고, 결과가 Codex unavailable 이면 AskUserQuestion 한 번 띄워서 install 여부 물어봐. install 선택 시 `npm install -g @openai/codex` 실행"* — 글로 적힌 절차일 뿐.
- **`AskUserQuestion` 이 유일한 interactive primitive** — codex plugin 은 자체 UI 안 만들고, *"두 옵션 중 하나"* 형식의 이 표준 도구 하나로 모든 사용자 선택을 처리합니다 (wait vs background, install vs skip 등 다른 command 도 같은 패턴).
- **`${CLAUDE_PLUGIN_ROOT}` 가 다시 등장** — helper script 를 부를 때 절대 경로 대신 이 환경변수로 잡으면 Claude Code 가 plugin 캐시 위치로 자동 resolve 해 줍니다.

자기 plugin 을 만들 때 *"한 장의 markdown 으로 시작"* 하는 자리는 이제 `commands/foo.md` 가 아니라 `SKILL.md` (단일 파일 skill) 입니다. codex 가 여전히 `commands/` 형식을 쓰는 건 앞서 본 cross-vendor 이유에서고요.

## 🤖 <span class="term">Agents</span> in plugin

`agents/codex-rescue.md` 의 frontmatter 만 봐도 이 subagent 의 성격이 한눈에 잡혀요.

```yaml
name: codex-rescue
description: Proactively use when Claude Code is stuck, wants a second
  implementation or diagnosis pass, needs a deeper root-cause investigation,
  or should hand a substantial coding task to Codex through the shared runtime
model: sonnet
tools: Bash
skills:
  - codex-cli-runtime
  - gpt-5-4-prompting
```

세 가지가 눈에 띕니다.

1. **Proactive 발동** — `description` 의 *"Proactively use when ..."* 표현 덕에, 사용자가 `/codex:rescue` 를 명시적으로 치지 않아도 Claude 가 *"여긴 내가 막혔으니 codex 한테 넘기는 게 낫겠다"* 라고 판단해서 알아서 invoke 합니다. 위 (user-defined) slash command 가 `disable-model-invocation: true` 로 자동 호출을 막은 것과 정반대 방향. 같은 plugin 안에서 component 마다 LLM 의 자동 발동 권한을 다르게 줘요.
2. **Sonnet + Bash 만** — 이 subagent 는 *판단·코딩을 직접 하지 않습니다.* 그래서 가벼운 Sonnet 모델, tool 도 `Bash` 하나뿐. 본문 첫 줄에 *"You are a thin forwarding wrapper. Your only job is to forward."* 라고 못박혀 있어요. 모든 지능은 Codex 쪽에 두고, subagent 는 Claude Code 의 hand-off 메커니즘 (subagent 호출) 을 빌리는 얇은 forwarder 역할만.
3. **2개 skill 첨부** — `codex-cli-runtime` (codex CLI 호출법 지식), `gpt-5-4-prompting` (사용자 요청을 forward 하기 전 더 좋은 Codex prompt 로 다듬는 용). 두 skill 모두 *"자체 reasoning · repo 검사 금지"* 가 명시돼서, subagent 가 자기 영역을 침범하지 않게 가둬 둡니다.

이 subagent 의 핵심도 결국 router 자세예요. 자기 reasoning 하지 않고 받은 걸 Codex 에 넘기기. 앞서 본 slash command 들 (verbatim 반환만 시키는 prompt) 과 같은 방향입니다. **Plugin 의 거의 모든 component 가 *"Claude 의 일을 줄여 Codex 에 넘긴다"* 는 한 축으로 정렬돼 있다**는 게 codex plugin 의 일관된 미학이에요.

## 🪝 <span class="term">Hooks</span> in plugin

이전 [plugins 글의 trust 섹션](/blog/claude-code-plugins/#%EF%B8%8F-plugin-%EC%84%A4%EC%B9%98%EB%8A%94-%EC%8B%A0%EB%A2%B0-%EA%B2%B0%EC%A0%95%EC%9E%85%EB%8B%88%EB%8B%A4--%ED%95%9C-%EB%B2%88-%EB%A9%88%EC%B6%94%EC%84%B8%EC%9A%94) 에서 *"plugin 안에는 hook 의 셸 명령, MCP 서버, 실행 파일 같은 실제로 실행되는 코드도 함께 담길 수 있다"* 라고 했죠. codex plugin 의 `hooks.json` 이 정확히 그 사례입니다.

```json
{
  "hooks": {
    "SessionStart": [{ "hooks": [{
      "type": "command",
      "command": "node \"${CLAUDE_PLUGIN_ROOT}/scripts/session-lifecycle-hook.mjs\" SessionStart",
      "timeout": 5
    }]}],
    "SessionEnd": [{ "hooks": [{ "type": "command", "command": "node \"${CLAUDE_PLUGIN_ROOT}/scripts/session-lifecycle-hook.mjs\" SessionEnd", "timeout": 5 }]}],
    "Stop": [{ "hooks": [{
      "type": "command",
      "command": "node \"${CLAUDE_PLUGIN_ROOT}/scripts/stop-review-gate-hook.mjs\"",
      "timeout": 900
    }]}]
  }
}
```

Claude Code 세션이 **시작될 때 / 끝날 때 / stop 신호를 받을 때** 마다 OpenAI 가 작성한 Node.js 스크립트가 **사용자 권한으로 실행됩니다** (각각 5 초, 5 초, 900 초 (15분) timeout). `${CLAUDE_PLUGIN_ROOT}` 가 plugin 의 캐시 경로로 자동 resolve 되는 것도 그대로 드러나죠.

이 plugin 을 깔면 OpenAI 가 작성한 `.mjs` 가 내 셸에서 실제로 돕니다. Apache 2.0 으로 소스가 공개돼 있어 마음만 먹으면 직접 검토할 수 있어요. 다만 현실에선 그 코드를 한 줄씩 읽어 보고 설치하는 사람은 거의 없죠. 결국 plugin 설치는 *'OpenAI 의 코드를 내 셸에서 돌릴 만큼 신뢰하는가'* 의 결정이라는 점은 그대로입니다.

<div class="deep-dive">

### 🔍 한 hook 만 들여다보면 · Stop hook

셋 중 Stop hook 이 가장 눈에 띕니다. 다른 둘은 timeout 이 5초인데 이것만 15분이거든요. 스크립트 이름도 `stop-review-gate-hook.mjs`. 이름이 곧 용도예요 — **Claude 가 답변을 마치려고 stop 신호를 보낼 때, 그 turn 을 잠깐 막아 세우고 OpenAI Codex 에게 검토를 받는 게이트** 입니다.

참고로 이 gate 는 <span class="key">기본적으로 꺼져 있고, 앞서 본 slash command `/codex:setup --enable-review-gate` 로 직접 켜둔 경우에만 동작합니다.</span> 켜둔 상태라면 흐름은 단순해요.

1. Claude 가 turn 을 마치려고 stop
2. hook 이 Codex 에게 *"방금 Claude 가 한 답변 검토해 줘"* 요청
3. Codex 가 `ALLOW: ...` 또는 `BLOCK: <이유>` 로 답
4. ALLOW 면 정상 종료. BLOCK 이면 hook 이 `{ decision: "block", reason: ... }` 를 emit 해서 **Claude 가 stop 못 하고 그 이유를 받아 계속 작업**

다시 말해 *"Claude 가 답을 쓰고, Codex 가 reviewer 역할로 통과시켜야 turn 이 끝나는"* loop 예요. <span class="key">cross-vendor 협업의 구체적인 한 그림</span> 이고, 15분이라는 timeout 도 이 review 가 충분히 길어질 수 있어서 잡아둔 값이고요. 자세한 구현은 184줄 짜리 스크립트라 여기서는 생략합니다.

</div>

## ⚡ 직접 설치해서 보기

```bash
# 1) marketplace 추가
/plugin marketplace add openai/codex-plugin-cc

# 2) plugin 설치
/plugin install codex@openai-codex

# 3) 재로드
/reload-plugins

# 4) Codex CLI 환경 점검 (없으면 `npm i -g @openai/codex` 로 자동 설치 도와줌)
/codex:setup

# 5) 첫 review 돌려보기
/codex:review --background
/codex:status
/codex:result
```

설치 후 `/agents` 에 `codex:codex-rescue` 가 떠 있으면 정상. Claude 가 막힐 때 자동으로 발동돼서 Codex 에 위임할 거예요.

<div class="callout-accent">
🧩 한 가지 더. 위 4번 단계의 `/codex:setup` 정체는 앞서 <span class="term">Commands in plugin</span> 섹션에서 본 <span class="term">commands/setup.md</span> 였죠. 그 파일 안을 보면 정확히 `npm i -g @openai/codex` 로 별도 바이너리가 설치되는 걸 알 수 있어요. **codex plugin 자체에는 codex AI 가 들어 있지 않습니다.** plugin 이 ship 하는 건 얇은 통합 레이어 (`commands/*.md` + `codex-companion.mjs` + hooks · agents) 뿐이고, 진짜 codex 실행은 사용자 PC 의 별도 `@openai/codex` 가 담당해요. 인증·설정도 `~/.codex/config.toml` 을 그대로 재사용하고요. <span class="key">Plugin 은 통합이지 구현이 아니다</span> — codex 가 보여주는 가장 강한 설계 결정이에요. 덕분에 plugin 은 OpenAI 의 LLM 을 직접 들고 다니지 않으면서도 *"OpenAI 와 Claude Code 사이의 다리"* 역할을 깔끔하게 해냅니다.
