<span class="term">Claude Code</span> 의 Plugin 은 **새 기능이 아니라**, 이미 있던 것들 (skill · hook · MCP server · agent 등) 을 한 폴더에 묶어 배포할 수 있게 한 **포장 방식**이에요. 정체는 **큰 폴더**입니다.

비유하자면 <span class="key">과자 선물세트</span> 와 비슷해요. 박스 자체는 새로운 종류의 과자가 아닙니다. 안에 들어가는 쿠키 · 초콜릿 · 캔디 같은 건 다 따로 살 수 있는, 이미 존재하는 제품들. 박스가 하는 일은 그저 그것들을 **한 묶음으로 만들어 선물하기 좋게** 해주는 것뿐이죠.

## 📂 Plugin 을 어떻게 인식하고, 어떻게 배포하나

### 1. 한 줄 인식 규칙

**plugin 은 그저 폴더 하나** 예요. Skill 의 본질이 SKILL.md 가 든 폴더인 것처럼, plugin 도 결국 폴더입니다. 바이너리도 컴파일된 산출물도 아니고, **그저 특정 구조를 따르는 디렉터리**. 그 "특정 구조" 가 한 줄이에요.

<pre class="plugin-rule-tree"><span>[any folder]/</span>
<span>└── .claude-plugin/</span>
<span>    ├── plugin.json        <span class="pr-comment">← 있으면 plugin 폴더</span></span>
<span>    └── marketplace.json   <span class="pr-comment">← 있으면 marketplace 폴더</span></span></pre>

.claude-plugin/ 안에 **plugin.json** 또는 **marketplace.json** (혹은 둘 다) 만 있으면 Claude Code 가 그 폴더를 인식합니다. 그게 전부의 구조적 조건.

### 2. 배포의 두 갈래

이 구조에서 **plugin 배포 방식은 자연스럽게 두 갈래**로 갈립니다.

<div className="callout-accent">
<strong>① Plugin 단일 배포</strong> — 🎁 과자 선물세트 한 개

폴더 하나가 곧 하나의 plugin 이에요. plugin.json 만 있으면 됩니다. 흐름은:

- **로컬 테스트**: claude --plugin-dir ./my-plugin
- **배포**: git · plugin registry · marketplace registry 등에 등록

> 로컬 테스트를 위해 직접 만드는 자세한 단계는 → [공식 docs · Develop more complex plugins](https://code.claude.com/docs/en/plugins#develop-more-complex-plugins)
</div>

<div className="callout-warm">
<strong>② Marketplace 배포</strong> — 🎁🎁🎁 과자 선물세트 여러 개를 묶은 진열대

여러 plugin 을 한 카탈로그로 묶어 배포해요. marketplace.json 한 장에 plugin 목록과 각각을 어디서 가져올지 (git repo, npm 등) 를 적어두면 됩니다. 흐름은:

- **로컬 테스트**: /plugin marketplace add ./my-marketplace
- **배포**: git · marketplace registry 등에 등록

apt repository · Homebrew tap 과 같은 자리예요.

> 로컬 테스트를 위해 직접 만드는 자세한 단계는 → [공식 docs · Walkthrough: create a local marketplace](https://code.claude.com/docs/en/plugin-marketplaces#walkthrough-create-a-local-marketplace)
</div>

<div class="note-box">
💡 <strong>Tip.</strong> <strong>실전에서는 대부분 marketplace 형태로 배포합니다</strong>. 설령 plugin 이 하나뿐이라도 — 과자 선물세트 하나만 놓여 있는 진열대처럼 — marketplace.json 한 장을 얹어 카탈로그로 만드는 게 표준 흐름이에요. 사용자 입장에서도 <span class="term">/plugin marketplace add</span> → <span class="term">/plugin install</span> 흐름이 익숙하기 때문.
</div>

이 marketplace 구조의 실제 사례 (Anthropic 공식 skills repo 가 marketplace.json 한 장으로 17개 skill 풀에서 plugin 3개를 cherry-pick 해 정의해둔 케이스) 는 → <a href={`${import.meta.env.BASE_URL}blog/real-world-plugin-anthropic-agent-skills/`}>[Real-world plugin · 1편] Anthropic agent-skills</a> 포스트에서 자세히 분해해두었습니다.

## 🔑 plugin 의 다섯 가지 핵심 개념

**Claude Code 의 구성 요소들을 모아놓은 박스**라는 점을 빼면, plugin 컨텍스트에서 추가로 알아둘 개념은 다음 다섯 가지뿐이에요.

이 중 안전 측면에서 가장 중요한 건 **Trust boundary**. 별도로 한 번 짚고 갑니다.

## ⚠️ Plugin 을 깔기 전에 한 번 멈추세요

<div class="callout-danger">
<span class="callout-danger-title">⚠️ Plugin 은 sandbox 되지 않아요. 당신과 같은 권한으로 돕니다</span>
Plugin 은 단순한 설정 파일 묶음이 아니에요. <strong>그 안에는 실제로 실행되는 코드</strong>가 들어 있습니다: skill 디렉터리 안의 스크립트 (Python · shell 등), hook 의 셸 명령, MCP 서버 프로세스, bin/ 의 실행 파일. 그리고 이 코드들은 sandbox 안에서 안전하게 도는 게 아니라, <span class="key">당신이 터미널에 직접 명령을 친 것과 똑같은 권한</span> 으로 실행됩니다. 결국 누군가 만든 plugin 을 한 번 깔면 그 사람의 코드가 당신 컴퓨터에서:

<ul>
  <li>셸에서 어떤 명령이든 실행할 수 있고,</li>
  <li>당신이 열 수 있는 어떤 파일이든 (SSH 키, 환경 변수, 회사 코드 등) 읽을 수 있고,</li>
  <li>인터넷으로 그것들을 어디로든 보낼 수 있고,</li>
  <li>당신의 Claude Code 설정을 바꿀 수 있습니다.</li>
</ul>

이게 새로운 종류의 위험인 건 아니에요. <strong>brew install</strong>, <strong>npm install -g</strong>, <strong>apt-get install</strong> 을 할 때 이미 우리가 안고 있는 그 신뢰, "이 패키지를 만든 사람을 어디까지 믿느냐" 와 같은 종류입니다.
</div>

차이라면 plugin 은 진입 장벽이 낮다는 점이에요. **눈에 쉽게 보이는 폴더 다운로드** 라서 신뢰 결정 자체를 **잊기 쉬워요**. 그래서 한 줄 원칙. <span class="key">marketplace 를 추가할 땐 apt repository 를 추가할 때처럼 한 번 멈추고 "이 출처를 믿을 만한가" 를 물을 것.</span> 트위터에서 "이거 좋대" 한 마디로 random marketplace 를 그냥 추가하지 마세요.

(참고로 Claude Code 자체에 native sandbox 가 있긴 한데 **Bash 도구에만 적용**되고, plugin 의 위험 부위 (skill 스크립트 · hook · MCP) 는 그 밖에서 돕니다. 더 강한 격리가 필요하면 Anthropic 이 공식 권장하는 [devcontainer](https://code.claude.com/docs/en/devcontainer) 안에서 돌리는 게 가장 깔끔해요.)

## 🪟 /plugins UI = 네 가지 책임의 분리

<span class="term">/plugins</span> 를 치면 다음과 같이 네 탭 (Discover · Installed · Marketplaces · Errors) 이 뜹니다.

<Image src={pluginsTabs} alt="Claude Code 의 /plugins 출력 — Discover, Installed, Marketplaces, Errors 네 탭이 노란 박스로 강조" />

<p class="img-caption">네 탭이 각자 <strong>별개의 책임</strong>을 담당해요.</p>

<div class="ui-tabs">
  <div class="ui-tab">
    <div class="ui-tab-num">01 · TAB</div>
    <div class="ui-tab-name">Marketplaces — <span class="ui-tab-role">신뢰</span> 의 자리</div>
    <div class="ui-tab-desc">카탈로그 구독. 추가만으로는 <strong>아무것도 설치되지 않고</strong>, 그저 카탈로그를 탐색 가능하게 만들 뿐.</div>
  </div>
  <div class="ui-tab">
    <div class="ui-tab-num">02 · TAB</div>
    <div class="ui-tab-name">Discover — <span class="ui-tab-role">선택</span> 의 자리</div>

이 네 가지를 따로 떼어 보는 습관만 들이면 plugin 관리가 한결 쉬워져요. 핵심은 세 줄.

<div class="callout-accent">

- **Marketplace 추가 ≠ plugin 설치** — 카탈로그를 구독해도 자동 설치되는 건 없음
- **Plugin 설치 ≠ 활성화** — 설치는 했는데 꺼둔 (disabled) 상태도 가능
- **설치 ≠ 로드 성공** — <span class="term">/skills</span> 에 예상한 게 안 뜨면 **Errors 탭부터** 확인

</div>

VS Code 확장은 "설치 = 즉시 활성화 = 같은 출처" 가 한 토글에 묶여 있지만, plugin 은 이 셋을 일부러 떼어둡니다. 덕분에 디버깅과 권한 관리가 훨씬 깔끔해져요.

## 💡 한 줄 요약

<div class="callout-warm">
Plugin 은 어려운 개념이 아니에요. 이미 아는 skill · hook · MCP · agent 를 한 폴더에 모아 manifest 한 장 얹으면 그게 plugin 입니다. 추가로 알아둘 건 다섯 가지 키워드 (**marketplace · manifest · trust · namespace · 설치 방식**) 정도.
</div>
