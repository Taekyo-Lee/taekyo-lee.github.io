예전엔 .claude/commands/ 폴더에 myCmd.md 같은 markdown 파일 한 장만 두면 `/myCmd` 라는 slash command 가 만들어졌어요 (파일 이름이 곧 명령 이름). **지금도 그대로 작동합니다.** 다만 <span class="term">Claude Code</span> 가 **더 이상 공식적으로 권장하지는 않아요**. 그 자리는 이제 **Skills** 의 영역입니다.

이건 제 임의의 분류가 아니라 Anthropic 의 공식 입장입니다. 공식 Skills 문서가 두 가지를 명확히 못박고 있어요.

먼저, **분류 자체** — 빌트인 commands 와 bundled skills 가 첫 줄부터 별도 항목으로 안내됩니다.

<blockquote>
For built-in commands like <code>/help</code> and <code>/compact</code>, and bundled skills like <code>/debug</code> and <code>/simplify</code>, see the commands reference.
<footer style="margin-top: 0.6em; font-size: 0.85rem; color: rgb(var(--gray));">— <a href="https://code.claude.com/docs/en/skills" target="_blank" rel="noopener">Claude Code Docs · Skills</a></footer>
</blockquote>

그리고 더 중요한 한 줄, **user-defined commands 는 Skills 로 통합됐다**.

<blockquote>
Custom commands have been merged into skills. A file at .claude/commands/deploy.md and a skill at .claude/skills/deploy/SKILL.md both create <code>/deploy</code> and work the same way. Your existing .claude/commands/ files keep working. Skills add optional features: a directory for supporting files, frontmatter to control whether you or Claude invokes them, and the ability for Claude to load them automatically when relevant.
<footer style="margin-top: 0.6em; font-size: 0.85rem; color: rgb(var(--gray));">— <a href="https://code.claude.com/docs/en/skills" target="_blank" rel="noopener">Claude Code Docs · Skills</a></footer>
</blockquote>

여기까지 짧은 두 인용에 벌써 **네 가지 용어**가 등장했어요.

- **Built-in commands**
- **user-defined commands**
- **Bundled skills**
- **Skills**

이 네 가지를 차근차근 풀어보면서, 왜 **user-defined commands 가 Skills 에 흡수**됐는지를 따라가 봅니다.

## 🆚 Built-in Commands vs Skills

<div class="note-box">
✋ <strong>잠깐</strong>, <strong>user-defined commands 는 잠시 제쳐두고</strong> Command = Built-in commands 라고 생각합시다. 그러면 이 섹션의 비교는 <strong>(Built-in) Commands vs (Bundled or Agent) Skills</strong> 두 카드로 단순해져요. Skills 쪽은 Bundled Skills 든 사용자가 만든 Skills (Agent Skills 라고도 불리는) 이든 상관없습니다. 한 묶음으로 봐요.
</div>

그러면 두 카드의 진짜 차이는 무엇이냐. 한 줄로 줄이긴 살짝 까다롭습니다. **"LLM 을 호출하느냐 아니냐"** 는 기준이 안 돼요 — Built-in Commands 안에도 LLM 을 안 쓰는 것 (`/help`) 과 잔뜩 쓰는 것 (`/compact`) 이 섞여 있거든요. 진짜 기준은 좀 더 미묘한데, 두 카드를 먼저 보고 다음 섹션에서 정확히 정리해 봅니다.

위 표에서 Built-in commands 와 Bundled skills 는 **소스 코드에 박혀 있어 사용자는 볼 수도, 추가할 수도 없다**고 했죠? 그런데 2026년 3월 말, Claude Code 의 소스 코드가 통째로 유출되는 사건이 있었답니다 (자세한 이야기는 <a href={`${import.meta.env.BASE_URL}blog/claude-code-leak-source-map/`}>Claude Code 소스 코드 유출</a> 포스트). 그래서 우리는 당시 (v2.1.88) 기준의 Claude Code 소스 코드를 직접 들여다볼 수 있게 됐어요. 두 카드의 **'위치' 행** 이 가리키는 디렉토리를 그 안에서 보면 이렇게 생겼습니다.

정리하면 한 줄. **답을 쓰는 주체가 harness 면 Built-in Commands, LLM 이면 Skills**. Bundled Skills 도 Anthropic 이 만든 built-in 이지만, 답을 쓰는 건 LLM 이라 Skills 쪽이에요.

## 🎬 예제로 차이 알아보기

앞 섹션에서 **"LLM 호출 여부는 기준이 못 된다"** 고 짧게 짚었어요. 이게 왜 그런지, Built-in Commands 두 개 (`/help` 와 `/compact`) 가 직접 보여줍니다.

<div class="callout-accent">
<strong>Built-in Commands 예시</strong>

- **`/help`** → harness 가 도움말 다이얼로그를 직접 그립니다. **LLM 호출 0회.**
- **`/compact`** → harness 가 Claude 에게 "지금까지 대화 요약해" 라고 부르고, **받은 요약을 가지고 직접 컨텍스트를 압축**해요. LLM 은 요약 일을 한 번 한 부품일 뿐, **그 요약본은 사용자에게 보여주지도 않습니다**. 사용자에게 "이만큼 토큰을 아꼈습니다" 같은 보고를 띄우는 것도 harness.

LLM 호출 횟수는 **0 vs 잔뜩** — 정반대예요. 그런데도 둘 다 Built-in Commands 로 묶이는 건, <span class="key">답의 형식과 내용을 결정하는 쪽이 양쪽 다 harness</span> 이기 때문입니다.
</div>

반면 Skills 는 정반대 패턴.

<div class="callout-warm">
<strong>Bundled Skills 예시</strong>

- **`/debug`** → harness 가 디버그 로그·세션 정보·사용자 인자를 모아 "여기 문제가 있어 보임. 진단해" 같은 프롬프트를 조립해서 Claude 에게 넘기고, **그 다음부터는 일반적인 대화 턴**. Claude 가 도구를 쓰며 진단·수정해요. 사용자가 보는 답은 LLM 이 만든 답 그대로.

User 가 직접 만드는 **Agent Skills** 도 동작 방식은 똑같아요.

</div>

<div class="callout">
정리하면 진짜 축은 <span class="key">"답을 쓰는 주체"</span>. **Built-in Commands** 는 harness 가 답의 형식·내용을 책임지고, **Skills** 는 LLM 이 자유롭게 응답해 그게 그대로 사용자에게 갑니다.
</div>

## 📜 User-defined commands (.claude/commands/) 는 더 이상 사용하지 마라?

위에서 Anthropic 이 공식적으로 **user-defined commands 대신 Skills 를 쓰라**고 권장한다고 짚었죠? 그렇다고 예전 방식이 당장 안 되는 건 아니에요. .claude/commands/ 에 foo.md 라는 markdown 파일 한 장만 두면 `/foo` 가 여전히 만들어지고 잘 작동합니다. 다만 **legacy** 위치라, 새로 만들 거라면 SKILL.md 형식을 쓰는 게 권장 사항이에요.

왜 이 방향으로 정리됐냐. 두 줄이면 설명돼요.

- **user-defined commands 는 사실 Built-in Commands 의 메커니즘이 아니라 Skills 의 메커니즘으로 동작해요.** .claude/commands/foo.md 의 markdown 본문은 결국 LLM 에게 보내는 prompt 역할이거든요. harness 가 답을 쓰는 게 아니라, LLM 이 그 prompt 를 받아서 답을 씁니다.
- **그렇다면 SKILL.md 파일 한 장으로 완전히 대체 가능해요.** SKILL.md 쪽이 더 풍부 (디렉토리 구조, frontmatter, 자동 발동 등) 하니까, 굳이 별도 카테고리로 둘 이유가 없죠.

그래서 통합되는 게 자연스러운 흐름이었습니다.

## 💡 한 줄 요약

<div class="callout-warm">

**Built-in Commands** = Anthropic 이 박아둔 빌트인 (harness 가 답을 씀).

**Skills** = Bundled Skills or (Agent) Skills (LLM 이 답을 씀).

**User-defined commands** = <span class="key">권장되지 않음. Skills 를 대신 사용</span> (Claude Code 에서 새 slash command 를 만들 거면 무조건 SKILL.md).

</div>
