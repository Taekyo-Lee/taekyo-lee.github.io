예전엔 `.claude/commands/` 폴더에 markdown 파일 한 장만 두면 `/myCmd` 라는 slash commands이 만들어졌어요. 지금도 그대로 작동합니다. 다만 Claude Code 가 **더 이상 공식적으로 권장하지는 않아요**. 그 자리는 이제 **Skill** 의 영역입니다.

이건 제 임의의 분류가 아니라 Anthropic 의 공식 입장입니다. 공식 Skills 문서의 첫 줄에 직접 적혀 있어요.

<blockquote>
For built-in commands like <code>/help</code> and <code>/compact</code>, and bundled skills like <code>/debug</code> and <code>/simplify</code>, see the commands reference.
<footer style="margin-top: 0.6em; font-size: 0.85rem; color: rgb(var(--gray));">— <a href="https://code.claude.com/docs/en/skills" target="_blank" rel="noopener">Claude Code Docs · Skills</a></footer>
</blockquote>

즉 **'Anthropic 이 만든 slash commands' 와 '내가 만드는 slash commands' 는 별개 카테고리** 로, 첫 줄부터 그렇게 분류되어 있는 셈이에요.

이 글은 그 분류를 한 멘탈 모델로 정리합니다 — **Anthropic 이 만든 slash commands 는 <span class="term">Command</span>, 내가 만드는 slash commands 는 <span class="key">Skill</span>**. 그리고 이 둘의 진짜 차이는 *"LLM 을 호출하느냐 아니냐"* 가 아니라 *"누가 답을 쓰는가"* 라는, 한 번 보면 깔끔하게 갈리는 축에 있습니다.

요약하면 **"Anthropic 이 박아둔 것 = Command, 그 외 모든 슬래시 = Skill"**. Bundled skill 도 결국 Anthropic 이 동봉한 거지만, 본질이 *"LLM 에게 줄 프롬프트"* 라서 Skill 분류에 속합니다.

## 🎬 진짜 차이: 누가 답을 쓰는가

여기까지 읽고 자연스럽게 떠오르는 의문 — *"그러면 LLM 을 호출하느냐 아니냐가 차이 아닌가?"* 그게 가장 흔한 오해예요.

<div class="callout-accent">
<span class="key">`/help` 는 LLM 을 한 번도 호출하지 않습니다. `/compact` 는 LLM 을 잔뜩 호출합니다. 그런데 둘 다 Command 예요.</span>
</div>

각각이 무슨 일을 하는지 보면 답이 나옵니다.

- **`/help`** → harness 가 도움말 다이얼로그를 그립니다. LLM 호출 0회.
- **`/compact`** → harness 가 Claude 에게 "지금까지 대화 요약해" 라고 부르고, **받은 요약을 가지고 직접 컨텍스트를 압축**합니다. LLM 은 요약 일을 한 번 한 부품일 뿐. 사용자에게 *"이만큼 토큰을 아꼈습니다"* 같은 보고를 띄우는 건 harness.

반면 Skill 은 다릅니다.

- **`/debug`** → harness 가 디버그 로그·세션 정보·사용자 인자를 모아 *"여기 문제가 있어 보임. 진단해"* 같은 프롬프트를 조립해서 Claude 에게 넘기고, **그 다음부터는 일반적인 대화 턴**. Claude 가 도구를 쓰며 진단·수정합니다. 사용자가 보는 답은 LLM 의 답.

<div class="callout-warm">
즉 차이의 축은 <span class="key">"답을 쓰는 주체"</span>. Command = harness 가 답의 형식과 내용을 책임짐. Skill = LLM 이 자유롭게 응답하고 그게 그대로 사용자에게 감.
</div>

## 📜 `.claude/commands/` 는 어떻게 됐나

예전 방식 (`.claude/commands/foo.md` 에 markdown 파일 한 장 두면 `/foo` 가 생기는 것) 은 여전히 작동합니다. 다만 **legacy** 위치예요. 새로 만들 거면 `SKILL.md` 형식을 쓰는 게 권장 사항입니다.

이 변화의 의미는 일관성 확보예요. 사용자가 자기 slash commands을 만들 때, 그건 본질적으로 *"Claude 에게 이런 프롬프트를 줘서 일을 시키고 싶다"* 이지 *"harness 가 이런 TypeScript 코드를 실행하도록 만들고 싶다"* 가 아니거든요. 사용자에겐 후자가 사실상 불가능하기도 하고요. 그렇다면 굳이 "사용자 정의 Command" 라는 분리된 개념을 둘 이유가 없습니다 — 그건 그냥 Skill 의 슬롯이에요.

기존 `.claude/commands/foo.md` 도 사실은 *"SKILL.md 만 있고 부속 파일이 없는 skill"* 과 거의 같은 동작을 합니다. 그래서 통합되는 게 자연스러운 흐름이었어요.

## 💡 한 줄 요약

<div class="callout-warm">
**Command = Anthropic 이 박아둔 빌트인 (harness 가 답을 씀). Skill = bundled + 내가 만든 것 (LLM 이 답을 씀).** 새 slash commands을 만들 거면 무조건 <span class="mod">SKILL.md</span>. <span class="key">"user-defined commands"</span> 라는 카테고리는 이제 잊고, "내가 만드는 슬래시는 Skill" 한 줄로 멘탈 모델을 정리하면 됩니다.
</div>
