이전 글에서 Claude Code 가 왜 터미널에 사는지를 이야기했어요. 그렇다면 이 친구가 켜지는 순간엔 정확히 무슨 일이 벌어질까요. 화면엔 그저 프롬프트 한 줄이 깜빡이지만, 그 200~500ms 사이에 꽤 많은 것이 동시다발로 굴러갑니다. 한 번 뚜껑을 열어 보고 싶었어요.

방법은 의외로 단순합니다. <span class="mod">claude --debug</span> 한 줄.

## 🔍 어떻게 들여다보나

가장 단순한 방법은 그냥 <span class="mod">--debug</span> 플래그를 붙여서 띄우는 거예요.

```bash
claude --debug
```

이러면 CLI 가 찍는 모든 디버그 로그가 자동으로 <span class="mod">&#126;/.claude/debug/&lt;sessionId&gt;.txt</span> 에 저장됩니다 (예: <span class="mod">&#126;/.claude/debug/f99a5fea-5fa1-46fb-bd57-77a5642da460.txt</span>). 위치·이름을 직접 지정하고 싶으면:

```bash
claude --debug-file ./startup.log
```

어느 쪽이든 100~200줄짜리 로그가 떨어져요. 매 줄이 코드의 어떤 지점을 통과했다는 흔적이고, 시간 정보까지 함께 박혀 있어서 부팅이 어떤 순서·어떤 페이스로 진행됐는지 그대로 보입니다.

이 4 계층이 <span class="key">settings.json, MCP 서버, Skills, Plugins</span> 네 영역에서 그대로 똑같이 반복됩니다. 한 번 익혀두면 어떤 항목을 어디에 두어야 하는지 헷갈릴 일이 없어요. "팀 공유면 저장소 안의 .claude/, 내 개인 환경이면 홈의 &#126;/.claude/" 한 줄 규칙으로 거의 모든 결정이 풀립니다.

## 🤔 거짓말처럼 보이지만 정상인 로그

처음 디버그 로그를 보면 깜짝 놀라는 줄들이 있어요. 그런데 자세히 들여다보면 모두 의도된 동작입니다. 자주 나오는 네 가지를 모아봤어요.

<div class="lie-card">
<span class="lie-line">[DEBUG] Broken symlink or missing file encountered for settings.json...</span>
<div class="lie-truth"><strong>실제로는 그냥 "파일이 없음".</strong> 설정 로더가 "broken symlink" 와 "파일 없음" 두 경우를 한 코드 경로로 처리하다 보니 메시지가 무서워 보일 뿐. 4 계층 중 비어 있는 항목마다 한 줄씩 찍힙니다.</div>
</div>

<div class="lie-card">
<span class="lie-line">[mcp-registry] Loaded 47 official MCP URLs</span>
<div class="lie-truth"><strong>활성 MCP 서버가 47개라는 뜻이 아닙니다.</strong> 바이너리에 내장된 <span class="key">설치 가능한 서버 카탈로그</span> (예: github, linear, playwright 등의 등록 정보). 실제 활성 서버 수와는 무관해요. 0개여도 47개여도 이 줄은 똑같이 찍힙니다.</div>
</div>

<div class="lie-card">
<span class="lie-line">[DEBUG] rg error ... /home/user/.claude/agents: No such file or directory</span>
<div class="lie-truth"><strong>최초 설치 시 예상되는 정상 동작.</strong> Skills 로더가 여러 선택적 디렉토리를 ripgrep 으로 스캔해 보는데, 아직 아무것도 설치되지 않은 상태라 디렉토리 자체가 없을 뿐. 각 디렉토리는 해당 기능을 처음 설치할 때 부수 효과로 생성됩니다.</div>
</div>

<div class="lie-card">
<span class="lie-line">[DEBUG] Fast mode unavailable: ...   ← 같은 줄이 20번 넘게 반복</span>
<div class="lie-truth"><strong>버그가 아니라 호출 지점이 많은 것.</strong> <span class="mod">isFastModeAvailable()</span> 함수가 모델 선택기, 상태 표시줄, <span class="mod">/model</span> 명령, 매 리렌더링 등 여러 UI 지점에서 호출됩니다. 매번 같은 결과를 다시 찍을 뿐 외형적 노이즈예요.</div>
</div>

이 네 가지를 알아두면 디버그 로그를 처음 봤을 때의 "어, 뭐 잘못된 거야?" 가 사라집니다.

## 🌐 단 하나의 동기 네트워크 호출

흥미로운 사실 하나. 부팅 과정 전체에서 **반드시 끝나야 다음으로 넘어갈 수 있는 네트워크 호출은 단 하나** 입니다.

<div class="callout-accent">
<span class="key">Bootstrap</span>: Claude Code 가 켜질 때 Anthropic 서버 (<span class="mod">api.anthropic.com</span>) 에 한 번 짧게 묻는 인사 같은 호출이에요. <span class="key">"이 사용자는 어떤 Claude 모델을 쓸 수 있는지", "지금 새로 켜진 기능 중 이 사용자에게 보여줄 게 있는지", "혹시 일부 사용자에게 시험 중인 새 화면이 있다면 이 사람도 그 그룹인지"</span> 같은 답을 받아옵니다. 모델 선택기·메뉴·UI 가 그려지기 전에 이 답이 손에 있어야 하니까 한 번은 꼭 끝내고 넘어가야 해요. 로그엔 <span class="mod">[Bootstrap] Fetching</span> 으로 시작해서 응답이 오면 <span class="mod">[Bootstrap] Fetch ok</span> 로 마무리됩니다.
</div>

다른 모든 네트워크 작업 (MCP 스키마, 모델 카탈로그, 사용자 설정 prefetch 등) 은 백그라운드로 돌고, 화면 렌더링·파일 인덱스 같은 로컬 작업과 병렬로 진행됩니다. 그래서 인터넷이 멀쩡할 때는 부팅 시간 = 로컬 작업 시간이지만, 인터넷이 느릴 때는 Bootstrap 한 호출이 시작 시간 전체를 잡아먹어요. "Claude Code 가 처음 뜨는 게 유난히 느린 날" 의 원인이 거의 항상 여기 있습니다.

## 🎬 REPL mounted — 프롬프트가 보이는 그 순간

로그를 따라가다 보면 정확히 이 한 줄이 등장합니다.

```
[DEBUG] [REPL:mount] REPL mounted, disabled=false
```

이 시점이 사용자 입장에서 "Claude Code 가 켜졌다" 라고 느끼는 순간이에요. 화면에 깜빡이는 커서가 처음 뜨고, 입력을 받기 시작합니다. 그 뒤에도 백그라운드 작업 (LSP 재초기화, MCP 카탈로그 로드, 설정 watcher 시작) 이 몇십~몇백 ms 더 굴러가지만, 이미 대화는 시작 가능한 상태입니다.
