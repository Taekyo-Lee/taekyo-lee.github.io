이 블로그는 제 local PC에서 글을 작성하고 git push 만 하면 몇 분 뒤에 자동으로 갱신됩니다.

Astro + GitHub Actions + GitHub Pages 조합으로 구현했습니다. 같은 스택으로 블로그를 운영하려는 분들을 위해 그 파이프라인을 한 장에 담아봅니다.

## 세 가지 도구가 만드는 파이프라인

<DeployPipeline />

이 셋이 각자 다른 일을 합니다.

| 도구 | 역할 |
| :--- | :--- |
| **Astro** | 정적 사이트 생성기 (SSG). .mdx / .astro / 이미지 같은 소스를 받아서, 브라우저가 바로 읽을 수 있는 HTML/CSS/JS 트리를 만들어줍니다. 파이썬으로 치면 Sphinx 나 MkDocs 의 자리. |
| **GitHub Actions** | repo에서 어떤 일 (예: main 으로 push) 이 일어났을 때 스크립트를 자동으로 돌려주는 <span class="term">CI/CD</span> 엔진. Jenkins / CircleCI / GitLab <span class="term">CI</span> 와 같은 카테고리. |
| **GitHub Pages** | 그 결과물을 무료로 호스팅해서 공개 URL 로 내보내는 정적 호스팅 서비스. dist/ 폴더의 파일을 그대로 돌려주는 서버 역할을 GitHub 가 대신해주는 셈이라, 직접 서버를 운영할 필요가 없습니다. |

핵심 한 줄. <span class="key">블로그의 글은 어디에도 데이터베이스로 저장되지 않습니다.</span> Astro 가 빌드 시점에 모든 페이지를 미리 HTML 로 찍어두고, GitHub Pages 는 요청이 올 때마다 그 HTML 을 그대로 돌려줍니다. 서버는 요청당 어떤 계산도 하지 않습니다. 그래서 DB 가 필요 없고, 그래서 프로세스가 안 돌고, 그래서 호스팅이 무료입니다.

## 자주 오해하는 지점: 로컬과 <span class="term">CI</span> 의 분업

위 다이어그램을 보면 "저자가 로컬 PC 에서 빌드해서 그 결과물을 push 하는 모양인가?" 하고 읽기 쉽지만, 그게 아닙니다. 이 차이가 이 시스템을 "CI/CD 파이프라인" 이라고 부르는 이유의 핵심입니다.

**저자가 로컬 PC 에서 하는 일**:

1. .mdx 파일을 작성합니다.
2. <span class="term">npm run dev</span> (= astro dev) 를 띄워둡니다. localhost:4321 에 라이브 미리보기 서버가 뜨고, 저장할 때마다 핫리로드가 일어납니다. 자동화 테스트 스위트가 따로 없는 블로그라면, 로컬 검증은 사실상 이 미리보기로 끝납니다. 레이아웃, 스타일, 컴포넌트, 링크가 깨지지 않았는지 눈으로 확인하는 게 검증의 전부입니다.
3. git add, git commit, git push.

**저자가 로컬 PC 에서 하지 않는 일**:

- astro build 를 직접 돌리지 않습니다. 운영용 빌드는 GitHub Actions runner 위에서 일어납니다.
- dist/ 를 commit 하지 않습니다. <span class="term">.gitignore</span> 에 들어가 있어서 실수로도 push 되지 않습니다.

dist/ 를 일부러 gitignore 하는 이유는 두 가지입니다. 첫째, **재현성**. 신뢰할 수 있는 빌드는 <span class="term">CI</span> 가 깨끗한 환경에서 버전을 고정해 만든 결과물뿐입니다. 로컬은 Node 버전이 다를 수도 있고, 의존성이 살짝 어긋나 있을 수도 있고, 정리 안 된 실험이 묻어 있을 수도 있습니다. 둘째, **노이즈**. 빌드를 한 번 돌리면 수백 개 파일이 바뀝니다. 매번 commit 하면 repo가 부풀어 오르고, 모든 diff 가 읽을 수 없게 됩니다.

일반 원칙. <span class="key">소스에서 다시 만들 수 있는 것은 모두 gitignore.</span> 보통 이 패턴에서는 dist/, .astro/, node_modules/ 가 모두 그렇게 처리됩니다.

## 그럼 왜 로컬 <span class="term">package.json</span> 도 astro 를 의존성으로 들고 있나요?

빌드는 <span class="term">CI</span> 가 한다면서요. 답은 <span class="key">package.json 은 양쪽이 같이 읽는 파일</span> 이라는 점입니다. 로컬과 <span class="term">CI</span> 둘 다 거기서 "이 프로젝트는 무엇이 필요한가" 를 읽고 각자 설치합니다.

<PackageJsonFork />

양쪽 다 자기만의 node_modules/ 를 갖게 되지만, 둘 다 commit 되지 않습니다. commit 되는 건 "어떤 의존성을 깔아야 하는지" 라는 **선언** 한 벌뿐입니다. 로컬의 Astro 는 미리보기와 도구용, <span class="term">CI</span> 의 Astro 는 운영 빌드용. 같은 버전, 같은 선언, 두 개의 설치 위치, 각자의 청중.

## 그럼 블로그가 일종의 <span class="term">npm</span> package 인가요?

Node 생태계의 "package" 라는 단어는 두 의미로 쓰입니다.

1. **publish 가능한 package** — <span class="term">npm publish</span> 로 공개 레지스트리에 올려서 다른 사람이 <span class="term">npm install</span> 해 쓰는 것. astro, react, lodash 같은 것들입니다.
2. **그냥 <span class="term">npm</span> 을 쓰는 모든 Node 프로젝트** — <span class="term">package.json</span> 으로 의존성을 선언하고 스크립트를 굴리는 어떤 폴더든.

블로그는 (2) 에 해당합니다. 블로그를 <span class="term">npm</span> 에 올릴 일은 없지만, 그래도 <span class="term">package.json</span> 이 필요합니다. 이유는:

- <span class="term">npm install</span> 이 무엇을 가져올지 읽는 manifest 입니다 (파이썬의 <span class="term">pyproject.toml</span> / requirements.txt 자리).
- <span class="term">npm run dev</span> / <span class="term">npm run build</span> 같은 스크립트가 거기에 정의됩니다.
- Node 버전이 거기서 핀 됩니다 (engines 필드).

실수로 publish 가 일어나는 일이 없도록, manifest 에 "private": true 를 박아두면 <span class="term">npm</span> 이 publish 자체를 거부합니다.

## 파이썬 엔지니어용 치트시트

파이썬 쪽 배경이 있다면, 이 매핑 한 장만 머리에 두면 나머지가 친숙해집니다.

| <span class="term">JavaScript</span> / GitHub | <span class="term">Python</span> / 익숙한 자리 |
| :--- | :--- |
| <span class="term">npm install</span> | <span class="term">pip</span> install -r requirements.txt |
| astro build 의 dist/ 출력 | mkdocs build 의 site/, sphinx-build 의 _build/html/ |
| GitHub Actions .yml workflow | <span class="term">tox.ini</span>, Jenkinsfile, GitLab .<span class="term">gitlab-ci.yml</span> |
| GitHub Actions runner (ubuntu-latest) | <span class="term">CI</span> 가 매번 새로 띄우는 일회성 VM |
| GitHub Pages 호스팅 | Read the Docs 에서 Sphinx 결과물을 서빙하는 모델 |
