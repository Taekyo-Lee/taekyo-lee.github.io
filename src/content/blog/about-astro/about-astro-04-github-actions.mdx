이전 두 글에서 도구들의 역할이 정해졌습니다. **Astro 가 빌드하고, GitHub Pages 가 그 결과를 서빙한다.** 그런데 아직 비어 있는 자리가 있습니다. **누가 빌드를 시키는가?** 글을 한 편 쓸 때마다 직접 astro build 를 돌리고 그 결과물을 어딘가에 올리는 일을 손으로 한다면 너무 번거롭습니다. 그 일을 대신 해주는 게 **GitHub Actions** 입니다.

이 글은 GitHub Actions 의 구조를 봅니다. 다음 글에서 실제 설정 파일을 한 줄씩 읽기 전에, 단어들과 층 구조를 먼저 정리하는 자리입니다.

## 이게 사실 <span class="term">CI/CD</span> 라는 개념입니다

GitHub Actions 의 본질은, <span class="key">repo에서 어떤 이벤트가 일어났을 때 스크립트를 가상 머신 (Virtual Machine, VM) 위에서 자동으로 돌려주는 서비스</span> 입니다. Jenkins, CircleCI, GitLab <span class="term">CI</span> 와 같은 카테고리입니다. GitHub 이 자기 안에 그 기능을 빌트인으로 넣은 형태로 보면 됩니다.

용어 두 개를 정리하고 갑니다.

- <span class="term">CI</span> (Continuous Integration): push 가 일어날 때마다 테스트, 린트 (lint, 코드 스타일/잠재 오류를 정적으로 검사하는 도구), 빌드를 자동으로 돌려서 깨진 것을 일찍 잡아내는 관행.
- CD (Continuous Deployment): 그렇게 검증된 빌드를 자동으로 운영에 배포하는 관행.

정적 블로그의 파이프라인은 사실상 <span class="key">CD 위주</span> 입니다. 자동 테스트를 따로 붙이지 않는 경우가 많아서 <span class="term">CI</span> 적인 검증은 거의 없고, 핵심은 <span class="key">build → deploy</span> 두 단계의 자동화입니다.

## workflow / job / step: 세 층

GitHub Actions 를 쓰려면, 프로젝트 repo 안에 **.github/workflows/** 라는 폴더를 만들고 그 안에 .yml 파일을 두면 됩니다. GitHub 이 push 가 들어올 때마다 그 폴더를 들여다보고, 안에 있는 .yml 파일들을 읽어서 정의된 동작을 실행합니다. 파일 한 개가 **workflow** 한 개고, 그 안은 세 층으로 중첩돼 있습니다.

```
.yml 파일 한 개 = workflow 한 개
├── trigger          ← "언제 돌릴 것인가?"
├── job A           ← 독립 실행 단위 (자기 VM 한 대)
│   ├── step 1
│   ├── step 2
│   └── ...
└── job B           ← 또 다른 독립 실행 단위 (또 다른 VM 한 대)
    ├── step 1
    └── ...
```

각각의 자리.

- workflow: .yml 파일 한 개. trigger, job, 설정이 다 들어 있습니다.
- job: workflow 안의 실행 단위. <span class="key">job 하나마다 새 runner (VM) 한 대가 떠서 돌고, 끝나면 폐기됩니다.</span> job끼리 파일을 공유하려면 명시적인 artifact 가 필요합니다.
- step: job 안에서 순차로 실행되는 한 줄짜리 작업 단위. runner 위에서 명령어를 직접 돌리거나 (run:), 미리 정의된 step 묶음을 끌어와서 (uses:) 실행합니다. 같은 job 안의 step들은 같은 runner 위에서 차례로 돌기 때문에 파일과 환경 변수가 그대로 이어집니다.

파이썬 비유. workflow가 <span class="term">tox.ini</span> 자리고, 각 job이 [testenv:...] 같은 격리된 환경, step이 그 안의 명령어 한 줄 한 줄에 가깝습니다. 차이가 있다면, GitHub Actions 는 job마다 **매번 새 VM 을 띄우고 끝나면 버린다** 는 점입니다.

## trigger: 언제 돌리나

on: 블록이 workflow가 실행되는 시점을 정합니다. 자주 쓰이는 옵션들.

| trigger | 실행 조건 |
|---|---|
| push | 지정 브랜치에 push 가 일어났을 때 |
| pull_request | PR 이 열리거나 갱신될 때 |
| schedule | cron 식 시간 간격으로 |
| workflow_dispatch | GitHub UI 의 "Run workflow" 버튼으로 수동 실행 |
| release | GitHub release 가 게시될 때 |

블로그 배포 workflow는 보통 push: branches: [main] 과 workflow_dispatch 를 같이 씁니다. **main 으로 push 가 들어오면 자동으로 돌고, 필요하면 수동으로도 돌릴 수 있는 workflow** 라는 뜻입니다.

## runner: 이건 어디서 도는가

runs-on: ubuntu-latest 한 줄이 "최신 Ubuntu VM 한 대를 띄워서, 이 job을 그 위에서 돌려라" 라는 지시입니다. <span class="key">job 하나마다 runner 한 대가 새로 만들어지고, job이 끝나면 사라집니다.</span> 이 단순한 사실이 굉장히 많은 결과를 만들어냅니다.

- 모든 job이 깨끗한 상태에서 시작합니다 (재현성에 좋습니다).
- 이전 빌드의 데이터를 가져오고 싶다면 **artifact** 나 **cache** 를 명시적으로 써야 합니다.
- 빌드 환경이 로컬 PC 와 다를 수 있습니다. 차이를 줄이려면 <span class="term">package.json</span> 의 engines 와 workflow의 <span class="term">node</span>-version: 으로 정확한 버전을 핀 박아야 합니다.

ubuntu-latest 외에 windows-latest, macos-latest 도 있긴 하지만, 정적 사이트 빌드는 거의 다 Ubuntu를 씁니다.

## 액션: 재사용 가능한 step

step 안의 uses: 는 다른 곳에 정의돼 있는 재사용 가능한 step 묶음을 끌어옵니다. 그 묶음을 액션 (action) 이라고 부릅니다. 보통은 GitHub 의 다른 repo에 공개돼 있는 것을 끌어 쓰지만, 같은 repo 안의 로컬 정의를 가리키게 만들 수도 있습니다.

```yaml
- uses: actions/checkout@v4
```

- actions/checkout@v4 = 공식 checkout 액션의 4번 버전.
- 이 한 줄이 "runner 위에 repo를 클론" 이라는 수십 줄의 자잘한 작업을 대체합니다.
- @v4 가 버전 핀입니다. 업스트림 자동 업그레이드 때문에 어느 날 갑자기 workflow가 깨지는 일을 막아줍니다.

파이썬 비유. <span class="term">PyPI</span> 에서 package를 가져오듯, GitHub Actions Marketplace 에 액션 생태계가 있습니다. 프레임워크 전용 액션도 풍부합니다. Astro 의 경우 withastro/action@v3 가 그런 예입니다.

## 권한: runner가 무엇을 할 수 있는가

```yaml
permissions:
  contents: read
  pages: write
  id-token: write
```

기본적으로 runner 안의 코드는 repo에 대한 **읽기 권한** 만 갖습니다. Pages 에 배포하려면 pages: write 를 명시적으로 줘야 합니다. id-token: write 는 OIDC 를 켜는 권한인데, 이게 있어야 runner가 "이 배포는 정말로 이 repo의 Actions 에서 나왔습니다" 라는 걸 Pages 에 증명할 수 있습니다.

원칙은 <span class="key">최소 권한</span>. 필요한 만큼만 줍니다.

## 동시성: 두 배포가 동시에 시작하면 어떻게 되는가

```yaml
concurrency:
  group: "pages"
  cancel-in-progress: false
```

빠르게 두 번 push 하면 workflow가 두 번 trigger됩니다. 그런데 Pages 입장에서는 "최신 버전이 라이브에 있으면 된다" 가 전부입니다. 두 배포가 병렬로 돌면 서로 꼬일 위험이 있습니다.

- group: "pages" — 같은 그룹 이름 아래에 줄을 세웁니다.
- cancel-in-progress: false — 진행 중인 배포는 그대로 끝까지 보내고, 새로 들어온 건 줄에서 기다립니다. (이 값을 true 로 두면 진행 중이던 배포를 취소하고 새 걸 시작하는데, 배포에는 위험한 선택입니다.)

## job 의존성과 artifact

```yaml
jobs:
  build:
    # ...
  deploy:
    needs: build      # build 가 성공한 뒤에만 실행
    # ...
```

- needs: build 가 순서를 강제합니다. build 가 실패하면 deploy 는 아예 돌지 않습니다.
- build 가 만들어낸 dist/ 가 deploy 까지 도달해야 하는데, <span class="key">두 job이 서로 다른 runner 위에서 돕니다. 파일시스템이 공유되지 않습니다.</span> 해결책은 build 에서 artifact 로 업로드, deploy 에서 다운로드.
- 다행히 withastro/action@v3 가 내부에서 dist/ 를 Pages 가 기대하는 정확한 포맷으로 업로드해주고, actions/deploy-pages@v4 는 그 artifact 를 자동으로 가져가서 배포합니다. artifact 이름을 직접 맞춰줄 필요가 없습니다.

## 로그는 어디서 보는가

push 가 끝나면, repo의 **Actions 탭** 에 workflow 실행 기록이 한 줄씩 쌓입니다.

- 실행을 클릭 → 각 job을 펼치기 → 각 step의 raw 로그까지.
- 실패한 경우 정확히 어느 step에서 깨졌는지 한눈에 보입니다.
- workflow_dispatch trigger가 있으면, UI 에 "Run workflow" 버튼이 떠서 수동으로 다시 돌릴 수 있습니다.

URL 패턴은 https://github.com/&lt;user&gt;/&lt;repo&gt;/actions (꺾쇠 안은 본인 계정/repo 이름으로 치환). 빌드가 한 번 깨졌다 싶으면 이 페이지부터 여는 게 정공법입니다.

## 정리

- GitHub Actions = repo 이벤트에 반응해서 스크립트를 돌려주는 CI/CD.
- 모양은 workflow (파일) → trigger + job들 → job 안의 step들.
- job마다 **새 Ubuntu runner** 한 대. job 사이에 파일을 넘기려면 artifact.
- uses: 가 미리 만들어둔 액션을 끌어옵니다. Astro + Pages 조합에서는 withastro/action@v3 + actions/deploy-pages@v4 두 개를 합치는 게 표준입니다.
- 권한은 최소 권한 원칙. 배포에 필요한 pages: write, id-token: write 만.
